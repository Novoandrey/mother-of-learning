# Мать Учения — Конституция проекта

## Миссия

Бесплатный open-source инструмент для ведения настольных ролевых кампаний.
Разрабатывается на примере сложной DnD 5e кампании с петлёй времени,
множеством игроков и обилием homebrew-контента — но ДОЛЖЕН быть адаптируем
под любой кампейн, любую редакцию и в перспективе любую систему.

Система ДОЛЖНА работать как связанный граф атомарных сущностей
с перекрёстными ссылками — по духу ближе к Obsidian, чем к красивому
приложению. UI — это линза поверх данных, а не самоцель.

## Философия: процесс важнее результата

Повторение — мать учения. Этот проект — не гонка за фичами,
а площадка для обучения программированию, архитектуре и дизайну.

- Качество понимания ВАЖНЕЕ скорости доставки.
- Каждая итерация — это урок. Переписать с нуля — нормально,
  если ты понял, почему первый раз было плохо.
- Код ДОЛЖЕН быть читаемым и понятным автору через месяц,
  а не только "работающим".
- Проект атомарный: каждый кирпичик самодостаточен, понятен
  и может быть объяснён другому человеку.

## Core Principles

### I. Атомарность данных

Каждая сущность (персонаж, НПС, заклинание, предмет, локация,
событие, механика, карта, день петли) — это отдельная "нода" с
типом, набором полей и уникальным идентификатором. Связи между
нодами — это отдельные "рёбра" с типом связи.

- Нода ДОЛЖНА быть самодостаточной: её можно открыть, прочитать
  и понять без контекста родительской коллекции.
- Поля ноды ДОЛЖНЫ быть типизированы, но допускать расширение
  через JSONB для homebrew-свойств.
- Одна и та же нода МОЖЕТ участвовать в разных контекстах
  (петлях, энкаунтерах, сессиях) через рёбра, а не дублирование.

**Обоснование:** Текущие таблицы дублируют НПС между листами,
теряют связи между петлями, и не позволяют ответить на вопрос
"где сейчас Дрипли и что у него в инвентаре?" одним запросом.

### II. Перекрёстные ссылки как первоклассные граждане

Любой текст (рекап, описание НПС, заметка ДМа) ДОЛЖЕН
поддерживать ссылки на другие ноды через единый синтаксис
(например, `[[Дрипли]]`, `[[Петля:2:День:15]]`).

- Ссылки ДОЛЖНЫ быть двунаправленными: если рекап ссылается
  на НПС, НПС показывает обратную ссылку на рекап.
- Битые ссылки ДОЛЖНЫ быть видимы, а не молча игнорироваться.
- Граф ссылок ДОЛЖЕН быть доступен как отдельное представление.

**Обоснование:** Главная боль — "потерялась инфа". Перекрёстные
ссылки превращают хаос разрозненных таблиц в навигируемую сеть.

### III. Петля времени как ось координат

Время в системе — это не линейная шкала, а циклическая структура:
Петля (номер) → День (1–31) → Время суток. Каждое событие, состояние
инвентаря, местоположение персонажа привязано к координатам петли.

- Система ДОЛЖНА поддерживать "вайп" — сброс состояний при начале
  новой петли с сохранением метаинформации (что помнят путешественники).
- Персистентное состояние между петлями: у игровых персонажей
  СОХРАНЯЮТСЯ уровни, способности, новые абилки и память.
  У мира — сбрасывается. Система ДОЛЖНА чётко различать
  "персистентные" и "вайпаемые" свойства на уровне модели данных.
- Сравнение "что изменилось между петлёй 2 и петлёй 3 в день 15"
  ДОЛЖНО быть одним запросом или одним экраном.
- Расписание академии (какие уроки в какой день) — это повторяющийся
  шаблон внутри петли, а не ручной ввод каждый раз.

**Обоснование:** Лист "Петля" с 4 колонками уже нечитаем. Система
петли — ядро вашей кампании, и она заслуживает первоклассной модели.

### IV. Работает без UI (данные-первичны)

Вся функциональность ДОЛЖНА быть доступна через структурированные
данные (API, markdown, таблицы), а UI — это опциональный слой
визуализации поверх них.

- Каждая фича сначала работает как запрос к данным, потом получает
  интерфейс.
- Данные ДОЛЖНЫ быть экспортируемы в markdown/JSON/CSV в любой
  момент без потерь.
- Импорт из Google Sheets ДОЛЖЕН быть поддержан как миграционный
  путь для существующих таблиц.

**Обоснование:** Защита от "скатывания в красивые интерфейсы".
Если фича не работает в виде текстового запроса — она не готова.

### V. Каждый релиз — играбельный инструмент

Ни одна итерация разработки НЕ ДОЛЖНА заканчиваться "подготовкой
к следующей итерации". Каждый шаг ДОЛЖЕН давать что-то, чем можно
пользоваться на ближайшей игровой сессии.

- MVP фичи — это "уже лучше, чем гугл-таблица" для конкретного
  юзкейса.
- Scope creep — главный враг. Если фичу нельзя закончить за
  1–2 недели, она слишком большая и ДОЛЖНА быть разбита.
- Каждый релиз ДОЛЖЕН содержать описание: "что именно стало
  удобнее для подготовки/проведения сессии".

**Обоснование:** СДВГ + бесконечный проект = нужны быстрые победы.
Если через две недели нечем похвастаться перед друзьями — что-то
пошло не так.

### VI. Мультиплеер и мульти-ДМ

Система ДОЛЖНА поддерживать одновременную работу нескольких ДМов
и игроков с разными уровнями доступа.

- Роли: владелец кампании, ДМ, игрок. Кампания МОЖЕТ иметь
  нескольких ДМов.
- Любой ДМ ДОЛЖЕН иметь возможность легко "подмить" свою историю
  в рамках общего мира — создавать свои сессии, НПС, сюжетные линии.
- ДМ видит всё в своей области. Игрок видит только то, что ему открыто.
- Изменения ДОЛЖНЫ синхронизироваться в реальном времени.
- Offline-режим НЕ ОБЯЗАТЕЛЕН на первом этапе, но архитектура
  НЕ ДОЛЖНА его исключать.

**Обоснование:** В текущем кампейне уже несколько ДМов. В будущем
любой пользователь должен мочь поднять свою кампанию и пригласить
друзей.

### VII. Внешние ссылки и медиа — ссылки, а не копии

Система НЕ ДОЛЖНА дублировать контент из внешних источников
(dnd.su, YouTube, плейлисты, AI-арт). Вместо этого — ссылки
с метаданными.

- Заклинание = homebrew-оверрайд + ссылка на оригинал на dnd.su.
- Карта = ссылка на изображение + метаданные (размер, тип, локация).
- Арт персонажа = ссылка на файл + тег "AI-generated".
- Система ДОЛЖНА хранить кеш-превью (заголовок, описание) для
  быстрого отображения без перехода.

**Обоснование:** У вас уже ~50 ссылок на dnd.su в бестиарии,
ссылки на YouTube для карт, арты персонажей. Копирование контента
— это технический и юридический тупик.

### VIII. Простота стека

Технологический стек ДОЛЖЕН быть минимальным и понятным одному
разработчику с СДВГ.

- Backend: Supabase (Postgres + Auth + Realtime + Storage).
- Frontend: Next.js + React + Tailwind.
- Деплой: Vercel (фронт) + Supabase Cloud (бек).
- Никаких микросервисов, очередей, кастомных серверов на первом
  этапе. Монолит — это нормально.
- Новый инструмент в стек — только если текущие доказано не
  справляются, а не потому что "интересно попробовать".

**Обоснование:** Каждый инструмент в стеке — это когнитивная
нагрузка. Supabase закрывает 80% бекенд-потребностей без написания
API вручную.

### IX. Микро-фичи как атомарные карточки

Каждая единица работы — это отдельная карточка-спецификация,
максимально маленькая и сфокусированная. Карточки организованы
в иерархию классов с возможностью структурного просмотра
(выпадающие списки, дерево).

- Фича ДОЛЖНА быть достаточно маленькой, чтобы уместиться в
  одну карточку с описанием, критериями приёмки и оценкой
  трудозатрат (идеал: 1–3 дня работы).
- Если фича не влезает — она ДОЛЖНА быть разбита на подфичи,
  каждая со своей карточкой.
- Карточки группируются по классам: `data-model`, `ui`,
  `api`, `sync`, `migration`, `dx` (developer experience).
- Структурный вид: дерево классов → список карточек внутри
  класса → детали карточки. Раскрываемые списки, не плоский
  бэклог.
- Зависимости между карточками ДОЛЖНЫ быть явными рёбрами,
  а не подразумеваемыми.

**Обоснование:** Защита от раздувания скоупа. Маленькие карточки
с чёткой классификацией — это и план работы, и документация,
и защита от "а давай ещё вот это добавим".

### X. Универсальность и open source

Система строится на примере конкретной тяжёлой кампании, но
ДОЛЖНА быть адаптируема под любой кампейн DnD, а в перспективе —
под любую настольную ролевую систему.

- Ничего специфичного для "Мать Учения" НЕ ДОЛЖНО быть
  захардкожено в ядре. Петля времени, академия, конкретные
  типы сущностей — это конфигурация кампании, а не код системы.
- Новый ДМ ДОЛЖЕН мочь поднять пустую кампанию и настроить
  её под свою игру без чтения исходников.
- Проект — бесплатный open source. Лицензия, документация
  и README ДОЛЖНЫ позволять любому склонировать, задеплоить
  и использовать.
- Сложная кампания — лучший тестовый стенд. Если система
  справляется с петлёй времени, 150 НПС и 4 параллельными
  таймлайнами — простой кампейн на 5 человек будет работать
  из коробки.

**Обоснование:** Начать со сложного — значит покрыть простое
автоматически. Open source — это и мотивация (твой проект
полезен другим), и дисциплина (код должен быть понятен
не только тебе).

## Ограничения и контекст

### Тестовая кампания (первый пользователь)
- DnD 5e + homebrew, основана на "Мать Учения" (Mother of Learning).
- ~15 активных игроков, 6–8 за сессию, несколько ДМов.
- 30-дневный цикл петли, 4+ пройденных петли.
- ~150 НПС, ~50 сыгранных сессий, обширный лор.
- Если система работает для этой кампании — она работает для любой.

### Разработчик
- Один человек, СДВГ, учится программировать.
- Процесс обучения важнее скорости. Переписать — нормально.
- Проект — и инструмент, и тренировочная площадка, и портфолио.

### Приоритеты первого релиза
1. Каталог сущностей с перекрёстными ссылками (замена таблиц НПС/PC).
2. Таймлайн петли (замена листа "Петля").
3. Трекер энкаунтера (замена "Трекер столкновения").

## Порядок разработки

Каждая фича проходит цикл:
`specify → clarify → plan → tasks → implement → играть`

"Играть" — обязательный шаг. Если фича не была использована
на реальной сессии — она не считается завершённой.

Фичи приоритизируются по принципу: "что чаще всего ищут/теряют
во время игры" (см. боли из интервью с ДМом).

## Governance

Конституция — это живой документ. Она пересматривается после
каждых 3–5 завершённых фич или при существенном изменении
требований кампании.

Изменения конституции требуют:
1. Описания причины изменения.
2. Обновления версии по semver.
3. Проверки, что существующие спеки не противоречат новым принципам.

## Протокол завершения чата

В конце каждого чата (перед тем как предложить создать новый) ОБЯЗАТЕЛЬНО:

1. Обновить `NEXT.md` в корне репозитория:
   - Что сделано в этом чате
   - Следующая задача (конкретно, с указанием файлов)
   - Любой важный контекст для следующего агента/чата
2. Обновить `backlog.md` — отметить выполненные пункты, добавить новые
3. Закоммитить оба файла: `docs: update NEXT.md and backlog for next session`
4. Запушить в main

`NEXT.md` — это передача эстафеты. Без него следующий чат начинает с нуля.
