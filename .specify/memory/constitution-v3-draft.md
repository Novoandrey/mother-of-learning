# Часть I: Продукт

## Миссия

Бесплатный open-source фреймворк для кооперативных настольных RPG
с фокусом на progression fantasy. Помогает группе друзей вместе
рассказывать историю, где персонажи растут через циклы.

Строится на DnD 5e / 5.5e. Адаптируем под любую систему.
По духу — движок для RPG-рогалика: петля прогрессии как ядро,
а не тактические карты.

## Для кого (три слоя)

1. **Наши друзья** — удобно прямо сейчас, на ближайшей сессии.
2. **Любая DnD-группа** — фреймворк без трения для progression fantasy,
   где есть петли, сложный лор, много НПС.
3. **RPG-движок** — фундамент для будущих систем и продуктов.

Правило принятия решений: сверху вниз. Сначала полезно друзьям,
потом обобщаемо для других, потом масштабируемо в движок.
Если фича нужна друзьям, но не масштабируется — делаем.
Если масштабируется, но друзьям не нужна — в бэклог.

## Два режима

Приложение обслуживает двух принципиально разных пользователей
с разными потребностями поверх одних и тех же данных.

### Игрок

Игроку нужна книга-помощник. Читалка, не дашборд.

- Мой персонаж: лист со статами, инвентарём, спеллами — живой,
  обновляемый каждую сессию.
- Моя история: что происходило с моим персонажем, петля за петлёй.
- Мои броски: кубики, особенно когда их много.
- Mobile-first. Скролл, текст, минимум кнопок. Информация
  потребляется, а не конфигурируется.

Игрок не должен понимать архитектуру приложения. Ему достаточно
открыть телефон и увидеть своего персонажа.

### ДМ (Мастер)

ДМу нужен рабочий стол. Граф, каталог, трекер, подготовка.

- Все сущности кампании: НПС, локации, предметы, монстры — граф
  с поиском, фильтрацией, связями.
- Трекер энкаунтера: инициатива, HP, условия, эффекты.
- Подготовка сессии: кто где, что запланировано, что произошло.
- Desktop-first, но работает и с телефона (ДМ на сессии
  больше ищет и читает, чем редактирует).

Один набор данных — два интерфейса. Принцип IV (данные-первичны)
делает это возможным: режим — это линза, не дублирование.

## Принципы продукта

### I. Петля прогрессии — ядро

Петля — не фича, а центральная метафора продукта. Персонаж
существует во времени: Петля → День → Момент. Каждый цикл —
рост, новые знания, новые способности.

- Система ДОЛЖНА различать персистентное (уровни, память, абилки
  игроков) и вайпаемое (состояние мира) на уровне модели данных.
- "Что изменилось между петлёй 2 и петлёй 3" — один экран.
- Повторяющиеся шаблоны (расписание, патрули, события) — шаблон
  внутри петли, не ручной ввод.

### II. Атомарность данных

Каждая сущность — нода с типом, полями и уникальным ID.
Связи между нодами — рёбра с типом связи.

- Нода самодостаточна: можно открыть и понять без контекста.
- Поля типизированы, но расширяемы через JSONB.
- Одна нода участвует в разных контекстах через рёбра,
  не через дублирование.

### III. Перекрёстные ссылки

Сущности связаны рёбрами. Рёбра кликабельны и двунаправлены.
Битые ссылки видимы. Граф ссылок доступен как представление.

Текстовые wiki-ссылки ([[Дрипли]]) — желательны, но не блокер.
Рёбра — основной механизм связей.

### III-b. Плоская навигация, группировка как линза

Данные — плоские. Вложенность в интерфейсе — иллюзия порядка,
которая быстро становится тюрьмой. Структура создаётся фильтрами
и группировкой в момент просмотра, а не иерархией в данных.

- Все сущности на одном уровне. "Академия → 3 курс → Группа 1" —
  три ноды с рёбрами, а не дерево с уровнями вложенности.
- Группировка — это конфигурация представления (по типу, по тегу,
  по связи), а не свойство данных. Одни данные — разные группировки.
- Ребро `contains` — валидная связь, но НЕ основа навигации.
  Навигация строится на фильтрах, тегах и типах.
- Один компонент (плоский список + конфигурируемая группировка)
  используется везде: каталог, сайдбар энкаунтера, лист персонажей.
- Если интерфейс требует больше одного уровня вложенности —
  это сигнал к пересмотру дизайна, а не к добавлению иерархии.

**Обоснование:** Плоская структура + фильтры = бесконечная гибкость
без когнитивной нагрузки на запоминание "где что лежит". Связи
между нодами дают ту же выразительность, что и вложенность, но
без ограничения "один родитель, одно место в иерархии".

### IV. Данные-первичны, UI — линза

Вся функциональность сначала работает как запрос к данным.
UI — опциональный слой визуализации.

- Данные экспортируемы в markdown / JSON / CSV без потерь.
- Импорт из Google Sheets — поддержан как миграционный путь.
- Два режима (игрок / ДМ) — это две линзы поверх одних данных.

### V. Event sourcing — история как данные

Каждое изменение в системе — событие. Текущее состояние =
результат всех событий до "сейчас". Это не просто лог —
это модель данных.

- Событие: кто, что, когда (петля/день/час), с кем, результат.
- Текущий стейт персонажа = replay событий до текущего момента.
- Temporal viewer, консоль, авто-бой, AI-контекст — всё читает
  один и тот же поток событий.
- "Сейчас" — фронтир. Дальше мотать нельзя (пока не сыграете).
- Снапшоты — кеш для производительности, не источник правды.

### VI. Читалка, не дашборд

Приложение — книга-помощник, не панель управления.

- Mobile: скролл вместо кликов, текст-центричный дизайн,
  дизайн ближе к читалке, чем к SaaS.
- Даже ДМ на сессии больше листает и ищет, чем редактирует.
- Минимум элементов управления. Каждая кнопка оправдана.

### VII. Каждый релиз — играбельный

Каждый шаг даёт что-то, чем можно пользоваться на ближайшей
сессии. "Уже лучше чем гугл-таблица" для конкретного юзкейса.

- Scope creep — главный враг. Фича > 2 недели = разбить.
- Если через две недели нечем похвастаться — что-то не так.
- **Правило переноса**: если есть работающая система (таблица,
  Excel, бумажка), сначала перенеси её как есть — чтобы было
  *хотя бы так же удобно*. Потом улучшай точечно. Не изобретай
  новый UX до того, как старый перенесён.

### VIII. Простота стека

Стек минимален и понятен одному разработчику.

- Backend: Supabase (Postgres + Auth + Realtime + Storage).
- Frontend: Next.js + React + Tailwind.
- Деплой: Vercel + Supabase Cloud.
- Новый инструмент — только если текущие доказано не справляются.

### IX. Универсальность и open source

Система строится на примере сложной кампании, но адаптируема
под любой кампейн и систему.

- Ничего специфичного для "Мать Учения" не захардкожено в ядре.
- Новый ДМ поднимает пустую кампанию без чтения исходников.
- Проект бесплатный, open source, документированный.

### X. Конституция кампании = правила мира

У каждой кампании есть своя "конституция" — человекочитаемый
документ с правилами мира, тоном, homebrew-механиками.

- ДМ-версия: полная (сеттинг, тайны, homebrew, механики).
- Игрок-версия: упрощённая (тон, сеттинг, базовые правила,
  без спойлеров).
- Формат: структурированный и машиночитаемый → может быть
  использован как контекст для LLM-генерации (НПС, энкаунтеры,
  лут, диалоги — всё в стиле этой кампании).
- Конституция кампании + граф сущностей + текущий момент =
  полный контекст для AI-помощника ДМа.

# Часть II: Процесс

## Философия: процесс важнее результата

Повторение — мать учения. Проект — площадка для обучения,
не гонка за фичами.

- Качество понимания ВАЖНЕЕ скорости доставки.
- Переписать с нуля — нормально, если понял почему.
- Код читаем автору через месяц, не только "работает".
- Каждый кирпичик самодостаточен и объясним.

## Цикл разработки

Каждая фича проходит цикл:

```
specify → clarify → plan → tasks → implement → играть
```

"Играть" — обязательный шаг. Фича не завершена, пока не
использована на реальной сессии.

## Декомпозиция

- Фича = карточка с описанием, критериями приёмки, оценкой
  (идеал: 1–3 дня).
- Не влезает → разбить на подфичи.
- Зависимости между карточками — явные.
- Классы: `data-model`, `ui`, `api`, `sync`, `migration`, `dx`.

## Приоритизация

Приоритеты пересматриваются постоянно. Критерий:
"что чаще всего ищут/теряют/делают руками во время игры?"

Текущие приоритеты (обновлять при пересмотре):
1. ~~Каталог сущностей~~ ✅
2. ~~Трекер энкаунтера~~ ✅
3. Лист персонажа для игрока (таблица "Персонажи" + обновление)
4. Консоль событий (лог изменений)
5. Петля как ось (вайп, сравнение, шаблоны)
6. Obsidian-layout (навигация через сайдбар)

## Единообразие UI

Каждый новый паттерн — долг. По умолчанию реюзай то что есть.

1. Компонент решает задачу → использовать.
2. Близок но нужна адаптация → расширить, не клонировать.
3. Ничего нет → создать в том же визуальном языке.

STYLE.md — источник правды для токенов и стилей.

## Роль AI-ассистента

AI-ассистент (Claude) — не просто исполнитель задач,
а участник процесса:

- **Следит за приоритетами**: если задача уводит от текущего
  приоритета — сигнализирует.
- **Замечает scope creep**: "это отдельная фича" — нормальная
  фраза от ассистента.
- **Предлагает пересмотр**: если конституция или приоритеты
  устарели — инициирует обсуждение.
- **Одна задача за раз**: СДВГ-friendly, не предлагает 5 вариантов.
- **Вайбкодинг**: пишет код сам, показывает результат, не процесс.
- **Эстафета**: в конце чата обновляет NEXT.md и backlog.md.

## Governance

Конституция пересматривается после каждых 3–5 фич
или при существенном изменении требований.

Изменения требуют:
1. Описания причины.
2. Обновления версии по semver.
3. Проверки совместимости со спеками.

## Контекст

### Тестовая кампания
- DnD 5e + homebrew, "Мать Учения" (Mother of Learning).
- ~15 игроков, 6–8 за сессию, несколько ДМов.
- 30-дневный цикл, 4+ петли, ~150 НПС, ~50 сессий.

### Разработчик
- Один человек, СДВГ, ПМ в геймдеве, учится программировать.
- Проект = инструмент + тренировка + портфолио.
