# Feature Specification: Граф сущностей — фундамент

**Feature Branch**: `001-entity-graph-foundation`
**Created**: 2026-04-13
**Status**: Done
**Input**: Карточки DX-001, DM-001, DX-002, UI-001 (wave 1). Создать проект,
модель данных нод/рёбер, засеять тестовыми данными из реальных таблиц кампании,
показать каталог сущностей с поиском и карточкой детали.

## User Scenarios & Testing

### User Story 1 — ДМ ищет НПС по имени (Priority: P1)

ДМ готовится к сессии и хочет быстро найти НПС по имени или части имени,
увидеть описание, статус и связи с другими сущностями. Сейчас для этого
нужно открывать Google Sheets, переключаться между листами НПС и НПС Академии,
искать Ctrl+F по тексту. Система должна заменить этот поток за одно действие.

**Why this priority**: Поиск НПС — самая частая операция при подготовке к игре.
150+ НПС в двух таблицах — главный источник "потерянной информации".

**Independent Test**: Открыть сайт, ввести "Тайвен" в поиск, увидеть карточку
с описанием и связями (команда, академия, знакомые персонажи).

**Acceptance Scenarios**:

1. **Given** ДМ на главной странице, **When** вводит "Тайв" в поле поиска,
   **Then** видит отфильтрованный список с "Тайвен" и может кликнуть на неё.
2. **Given** ДМ на карточке НПС "Тайвен", **When** смотрит раздел связей,
   **Then** видит кликабельные ссылки на связанных персонажей (Дрипли, Урик, Оран).
3. **Given** ДМ на карточке "Тайвен", **When** кликает на связь "Дрипли",
   **Then** переходит на карточку Дрипли.

---

### User Story 2 — ДМ просматривает всех персонажей игроков (Priority: P1)

ДМ хочет видеть список всех PC с именами игроков, кратким описанием
и текущим статусом (активен / неактивен / альт).

**Why this priority**: На каждой сессии ДМ работает с 6-8 персонажами
и их связями. Нужен быстрый обзор кто есть кто.

**Independent Test**: Открыть каталог, выбрать фильтр "PC", увидеть
список из ~15 персонажей с именами игроков.

**Acceptance Scenarios**:

1. **Given** ДМ в каталоге, **When** фильтрует по типу "Персонаж игрока",
   **Then** видит список всех PC с именем персонажа, именем игрока и кратким описанием.
2. **Given** ДМ на карточке "Каэл Дренвик", **When** смотрит поля,
   **Then** видит игрока (Женя Юрченков), описание, связи (Орвен Даллар, Лира Телван).

---

### User Story 3 — ДМ создаёт новую сущность (Priority: P2)

ДМ придумал нового НПС прямо перед сессией и хочет добавить его в систему
с минимальным описанием, чтобы не забыть.

**Why this priority**: Новые НПС появляются каждую сессию. Если добавление
сложнее чем "вписать строку в таблицу" — никто не будет пользоваться.

**Independent Test**: Нажать "Создать", выбрать тип "НПС", ввести имя
и описание, сохранить. Сущность появляется в каталоге и находится поиском.

**Acceptance Scenarios**:

1. **Given** ДМ нажимает "Создать сущность", **When** выбирает тип "НПС",
   вводит имя "Горнокрабль" и описание "Волшебник, поднял таверну на пальце",
   **Then** сущность сохраняется и появляется в каталоге.
2. **Given** ДМ только что создал "Горнокрабль", **When** ищет "Горно" в поиске,
   **Then** находит только что созданного НПС.

---

### User Story 4 — ДМ фильтрует каталог по типу (Priority: P2)

ДМ хочет видеть только заклинания, или только локации, или только НПС
определённого статуса (преподаватель, студент, городской).

**Why this priority**: 150 НПС + PC + предметы + локации в одном списке
нечитаемы. Фильтрация по типу — базовая навигация.

**Independent Test**: Выбрать фильтр "Локация", увидеть только локации
(Промежуточный слой, Нижние уровни, Гадкий Койот и т.д.)

**Acceptance Scenarios**:

1. **Given** ДМ в каталоге с 50+ сущностями, **When** выбирает фильтр "НПС",
   **Then** список показывает только НПС.
2. **Given** ДМ фильтрует по "НПС", **When** дополнительно вводит
   поиск "препод", **Then** видит только НПС со словом "преподаватель"
   в описании или статусе.

---

### Edge Cases

- Что происходит при поиске с пустым результатом? → Сообщение "Ничего не найдено"
  с предложением создать новую сущность.
- Как обрабатываются НПС без описания? → Показывается карточка с именем
  и пустым полем описания, доступным для редактирования.
- Что если два НПС с одинаковым именем? → Оба показываются в результатах,
  различаются по типу и/или описанию.
- Кириллица в поиске? → Поиск ДОЛЖЕН корректно работать с кириллицей,
  включая регистронезависимый поиск.

## Requirements

### Functional Requirements

- **FR-001**: Система ДОЛЖНА хранить сущности (ноды) с полями:
  уникальный идентификатор, тип, название, произвольные поля (ключ-значение).
- **FR-002**: Система ДОЛЖНА поддерживать типы сущностей: `character` (PC),
  `npc`, `location`, `group`, `organization`, `item`, `spell`, `event`,
  `mechanic`, `creature`. Список типов ДОЛЖЕН быть расширяемым без
  изменения кода.
- **FR-003**: Система ДОЛЖНА хранить связи (рёбра) между сущностями
  с полями: источник, цель, тип связи, метаданные.
- **FR-004**: Система ДОЛЖНА предоставлять текстовый поиск по названию
  и содержимому полей сущностей. Поиск ДОЛЖЕН быть регистронезависимым
  и поддерживать кириллицу.
- **FR-005**: Система ДОЛЖНА отображать каталог сущностей с возможностью
  фильтрации по типу.
- **FR-006**: Система ДОЛЖНА отображать карточку отдельной сущности
  со всеми полями и списком связей (входящих и исходящих).
- **FR-007**: Связи на карточке ДОЛЖНЫ быть кликабельными для навигации
  к связанной сущности.
- **FR-008**: Система ДОЛЖНА позволять создавать новую сущность
  с указанием типа, названия и произвольных полей.
- **FR-009**: Система ДОЛЖНА содержать начальные данные (seed),
  импортированные из реальных таблиц кампании: минимум 10 НПС,
  5 PC, 3 локации с установленными связями между ними.
- **FR-010**: Система ДОЛЖНА быть доступна онлайн по URL.
- **FR-011**: URL-структура ДОЛЖНА включать slug кампании:
  `/c/[campaign-slug]/catalog/[id]`. Это обеспечивает мультикампейн
  из коробки без будущего рефакторинга.
- **FR-012**: Система ДОЛЖНА позволять добавлять связь (ребро)
  между сущностями с карточки ноды: выбор типа связи из dropdown,
  поиск целевой ноды, сохранение.
- **FR-013**: Типы связей ДОЛЖНЫ храниться в таблице `edge_types`
  per campaign. Базовые типы (knows, teaches, located_in, owns,
  member_of, contains) помечены флагом `is_base=true` и доступны
  всем кампаниям. Кастомные типы создаются на уровне кампании.
- **FR-014**: Система ДОЛЖНА поддерживать теги как массив строк
  в поле `fields.tags`. Теги — это свойства для быстрой фильтрации
  ("преподаватель", "антагонист", "мёртв", "петля-2"), а не сущности.
- **FR-015**: Вложенность структур ДОЛЖНА моделироваться через
  ребро типа `contains` (parent → child): "Академия → contains →
  3 курс → contains → Группа 1". Произвольная глубина.

### Key Entities

- **Нода (Node)**: Атомарная сущность кампании. Имеет тип (character,
  npc, location, group...), название, набор произвольных полей (описание,
  статус, игрок, арт, ссылки) и опциональный массив тегов (fields.tags).
- **Ребро (Edge)**: Направленная связь между двумя нодами. Имеет тип
  (knows, teaches, member_of, contains, owns...) и опциональные
  метаданные.
- **Тип сущности (Node Type)**: Определяет категорию ноды и набор
  рекомендуемых (но не обязательных) полей. Расширяется пользователем.
- **Тип связи (Edge Type)**: Определяет категорию ребра. Базовые типы
  (is_base=true) общие для всех кампаний. `contains` используется
  для вложенности структур (parent → child).
- **Тег (Tag)**: Строковая метка в `fields.tags` для быстрой фильтрации.
  Теги — прилагательные ("антагонист", "мёртв"), не существительные.
  Правило: "Можно ли ткнуть пальцем в мире игры?" → да = нода, нет = тег.

## Clarifications

### Session 2026-04-13

- Q: URL-структура — с кампанией или без? → A: С кампанией: `/c/[slug]/catalog/[id]` (мультикампейн из коробки)
- Q: Создание связей — в этой фиче или отложить? → A: В этой фиче. Простая форма на карточке ноды: dropdown типа связи + поиск цели
- Q: Типы связей — свободный текст или таблица? → A: Таблица `edge_types` per campaign с флагом `is_base` для универсальных DnD-типов
- Q: Вложенность структур и теги — как моделировать? → A: Вложенность через ребро `contains` (parent→child). Теги — массив строк в fields.tags для фильтрации. Правило разграничения: "можно ткнуть пальцем?" → нода, "свойство для поиска?" → тег

## Success Criteria

### Measurable Outcomes

- **SC-001**: ДМ находит любого НПС по имени за менее чем 5 секунд
  (от открытия сайта до карточки НПС). Сейчас: ~30 секунд через
  Google Sheets.
- **SC-002**: Навигация от одного НПС к связанному — 1 клик.
  Сейчас: Ctrl+F по новому листу.
- **SC-003**: Создание нового НПС — менее 30 секунд.
  Сейчас: найти нужный лист, пролистать до конца, вписать строку.
- **SC-004**: Каталог содержит минимум 18 реальных сущностей
  (10 НПС + 5 PC + 3 локации) с корректными связями при первом деплое.
- **SC-005**: Сайт загружается и отвечает на поиск за менее чем
  2 секунды на обычном интернете.

## Assumptions

- Аутентификация и разграничение прав (ДМ / игрок) — вне скоупа
  этой фичи. На первом этапе доступ открытый.
- Редактирование существующих сущностей — вне скоупа. Только
  создание и просмотр.
- Импорт из Google Sheets — ручной (SQL seed-скрипт), не
  автоматический коннектор.
- Markdown-рендеринг и wiki-ссылки ([[...]]) — вне скоупа,
  будут в следующей фиче.
- Мобильная адаптация — желательна, но не блокирующее требование.
