# Research: Граф сущностей — фундамент

## R-001: Модель хранения графа в Postgres

**Decision**: Две таблицы — `nodes` и `edges` — с JSONB для произвольных полей.
Не графовая БД (Neo4j), не EAV-модель.

**Rationale**: Postgres + JSONB даёт гибкость схемы (каждый тип ноды
может иметь свои поля) без потери SQL-запросов, индексов и JOIN-ов.
Supabase работает поверх Postgres из коробки. Для масштаба ~500 нод
и ~2000 рёбер реляционная модель с JSONB достаточна.

**Alternatives considered**:
- Neo4j / graph DB: overkill для масштаба, дополнительная инфраструктура,
  не поддерживается Supabase.
- EAV (entity-attribute-value): сложные запросы, плохая производительность
  на JOIN-ах, трудно читать данные напрямую.
- Один JSON-документ на ноду в NoSQL: потеря реляционных запросов,
  нельзя делать JOIN по связям.

## R-002: Полнотекстовый поиск по кириллице

**Decision**: Postgres `tsvector` с конфигурацией `russian` + `unaccent`.
GIN-индекс на `search_vector` колонку в таблице `nodes`.

**Rationale**: Встроенный в Postgres, не требует внешних сервисов.
Supabase поддерживает full-text search через `.textSearch()` в клиенте.
Русская стемминг-конфигурация корректно обрабатывает кириллицу.

**Alternatives considered**:
- `ILIKE '%query%'`: работает, но не масштабируется и не поддерживает
  стемминг (Тайвен ≠ тайвен).
- Meilisearch / Typesense: отдельный сервис, противоречит принципу VIII
  (простота стека).
- pg_trgm (триграммы): хорош для fuzzy-поиска, но тяжелее для простых
  запросов. Можно добавить позже как улучшение.

## R-003: Расширяемые типы сущностей

**Decision**: Таблица `node_types` с колонками `slug`, `label`,
`default_fields` (JSONB-схема рекомендуемых полей). Типы создаются
на уровне кампании, не хардкодятся в коде.

**Rationale**: Принцип X (универсальность) требует, чтобы типы были
конфигурацией. Для кампании "Мать Учения" seed создаст типы character,
npc, location, creature и т.д. Другая кампания может иметь свой набор.

**Alternatives considered**:
- Enum в коде: нарушает принцип X, требует деплоя для нового типа.
- Свободный текст без таблицы типов: невозможно показать фильтры
  и рекомендуемые поля для создания.

## R-004: Структура Next.js проекта

**Decision**: Next.js 14+ App Router. Один проект, без монорепы.
`/app` для страниц, `/lib` для Supabase-клиента и утилит,
`/components` для UI. Tailwind CSS для стилей.

**Rationale**: App Router — текущий стандарт Next.js. Один проект —
принцип VIII. Supabase JS client используется напрямую, без
промежуточного API-слоя (Supabase сам является API).

**Alternatives considered**:
- Pages Router: устаревший паттерн, меньше документации для новичков.
- Отдельный Express/Fastify бекенд: лишний слой, Supabase уже
  предоставляет REST API и RLS.
- Monorepo (turborepo): overkill для одного разработчика.
