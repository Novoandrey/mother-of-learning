# Feature Specification: Obsidian-Layout — единая навигация

**Feature Branch**: `004-obsidian-layout`
**Created**: 2026-04-14
**Status**: Draft
**Input**: Редизайн UI: убрать дублирование навигации, сделать сайдбар единственным
местом навигации (как в Obsidian), а основную область — чистым просмотрщиком
контента. "Аппы" (энкаунтеры, петли, сессии) — изолированные режимы.

## Контекст проблемы

Сейчас навигация дублируется: сайдбар содержит дерево сущностей и поиск,
а страница каталога (`/catalog`) — свой поиск, карточки типов, грид результатов.
Карточка сущности показывает грид детей, хотя дерево в сайдбаре уже это делает.
Пользователь путается: где искать? где навигироваться?

**Obsidian-модель**: один файловый менеджер слева, один просмотрщик справа.
Никаких альтернативных путей навигации в основной области.

## User Scenarios & Testing

### User Story 1 — ДМ находит и открывает сущность (Priority: P1)

ДМ готовится к сессии. Открывает приложение. Видит сайдбар с деревом
сущностей, сгруппированным по типам. Вводит "Тайв" в поиск сайдбара —
дерево фильтруется. Кликает "Тайвен" — справа открывается карточка НПС.
Кликает в сайдбаре "Дрипли" — карточка справа меняется на Дрипли.
Основная область НЕ содержит своей навигации — только контент карточки.

**Why this priority**: Поиск и просмотр сущностей — 80% всего использования.
Устранение дублирования навигации — ядро редизайна.

**Independent Test**: Открыть сайт → сайдбар видим с деревом → поиск
фильтрует → клик открывает карточку справа → нет дублирующей навигации
в основной области.

**Acceptance Scenarios**:

1. **Given** ДМ открыл сайт, **When** ничего не выбрано в сайдбаре,
   **Then** основная область показывает приветственный экран (без навигации).
2. **Given** ДМ на сайте, **When** вводит "Тайв" в поиск сайдбара,
   **Then** дерево фильтруется, показывая только совпадения.
3. **Given** ДМ кликнул "Тайвен" в сайдбаре, **When** карточка открылась,
   **Then** основная область показывает ТОЛЬКО контент карточки (поля,
   теги, связи, летопись). Нет поиска, нет карточек типов, нет грида детей.
4. **Given** ДМ на карточке Тайвен, **When** кликает "Дрипли" в сайдбаре,
   **Then** карточка справа меняется на Дрипли.

---

### User Story 2 — ДМ навигирует по вложенности (Priority: P1)

ДМ хочет найти конкретную группу внутри академии. В сайдбаре разворачивает
"Академия Сиории" → видит "3 курс" → разворачивает → "Группа 1".
Клик на "Группа 1" → карточка справа.

Если нода — "папка" (имеет дочерние элементы через `contains`), она
работает в сайдбаре как разворачиваемый узел. Клик на неё разворачивает/
сворачивает. Если нода — "файл" (атомарная, без детей), клик открывает
карточку справа.

**Why this priority**: Вложенность через `contains` — ключевая структура
данных. Навигация по дереву = навигация по вложенности.

**Independent Test**: Развернуть "Академия" в сайдбаре → увидеть
подчинённые → клик на атомарную → карточка справа.

**Acceptance Scenarios**:

1. **Given** нода "Академия Сиории" с детьми (3 курс), **When** ДМ
   кликает по ней, **Then** узел разворачивается, показывая дочерние.
2. **Given** развёрнутая "Академия", **When** кликает повторно,
   **Then** узел сворачивается.
3. **Given** нода "Группа 1" без детей, **When** ДМ кликает,
   **Then** справа открывается карточка "Группа 1".
4. **Given** нода с детьми развёрнута, **When** ДМ хочет увидеть
   её карточку, **Then** может это сделать (например, двойной клик
   или клик по иконке/названию, а стрелка — для разворота).

---

### User Story 3 — ДМ создаёт новую сущность (Priority: P1)

ДМ хочет быстро добавить нового НПС. Жмёт "+" в сайдбаре (или кнопку
"Создать" в хедере). Открывается форма создания в основной области.
После сохранения: новая нода появляется в дереве сайдбара, карточка
открывается справа.

Если ДМ хочет создать новый тип сущности (например, "Нейроарт"),
он может это сделать на экране выбора типа.

**Why this priority**: Создание — вторая по частоте операция. Должно
быть одинаково легко из любого контекста.

**Independent Test**: "+" → выбор типа → заполнить → сохранить →
нода в сайдбаре и карточка справа.

**Acceptance Scenarios**:

1. **Given** ДМ на любой странице, **When** жмёт "Создать",
   **Then** форма создания открывается в основной области.
2. **Given** форма создания, **When** ДМ выбирает тип, вводит имя,
   сохраняет, **Then** нода создаётся, появляется в сайдбаре,
   карточка открывается справа.
3. **Given** нет подходящего типа, **When** ДМ жмёт "Создать свой тип",
   **Then** может ввести эмодзи + название → тип создаётся → продолжает
   создание ноды этого типа.

---

### User Story 4 — ДМ переключается на "апп" (Priority: P2)

ДМ хочет вести бой. Кликает "Энкаунтеры" в навигации сайдбара →
основная область переключается на список энкаунтеров (это "апп").
Сайдбар может показывать свой контекст для аппа (список энкаунтеров)
или оставаться с деревом каталога.

**Why this priority**: Аппы уже работают. Нужно только чистое
переключение контекста.

**Independent Test**: Клик "Энкаунтеры" в сайдбаре → основная область =
список энкаунтеров. Клик "Каталог" → возврат к режиму каталога.

**Acceptance Scenarios**:

1. **Given** ДМ в каталоге, **When** кликает "Энкаунтеры",
   **Then** основная область показывает список/детали энкаунтеров.
2. **Given** ДМ в энкаунтере, **When** кликает "Каталог",
   **Then** возвращается к дереву сущностей и карточке (если была).

---

### User Story 5 — ДМ работает со связями на карточке (Priority: P2)

На карточке сущности ДМ видит раздел "Связи" со списком связанных
нод. Каждая связь — кликабельная ссылка. Клик переходит к этой ноде
в сайдбаре и открывает карточку справа.

Связи типа `contains` НЕ показываются в списке связей (они видны
как дерево в сайдбаре). Только "смысловые" связи: knows, teaches,
member_of, etc.

**Why this priority**: Связи — навигация через смысл, а не через
иерархию. Дополняет дерево в сайдбаре.

**Independent Test**: Открыть "Тайвен" → увидеть связь "knows → Дрипли"
→ кликнуть → перейти на Дрипли.

**Acceptance Scenarios**:

1. **Given** карточка "Тайвен", **When** ДМ видит связи,
   **Then** `contains`-связи отсутствуют в списке (они в сайдбаре).
2. **Given** связь "knows → Дрипли", **When** клик,
   **Then** карточка Дрипли открывается, Дрипли подсвечен в сайдбаре.

---

### Edge Cases

- Что показывать когда ничего не выбрано? → Приветственный экран:
  название кампании, количество сущностей, быстрые действия.
- Что если нода одновременно "папка" (имеет детей) и содержит
  полезные данные? → Клик на стрелку = развернуть/свернуть. Клик
  на название = открыть карточку. Два отдельных хитбокса.
- Сайдбар на мобильном? → Скрывается, открывается по кнопке.
  Вне скоупа на первом этапе, но layout НЕ ДОЛЖЕН ломать мобильную
  вёрстку полностью.
- Что если сущность удалена, а карточка открыта? → Показать
  "Сущность не найдена", вернуться к приветственному экрану.

## Requirements

### Functional Requirements

- **FR-001**: Сайдбар ДОЛЖЕН быть единственным местом навигации по
  сущностям. Основная область НЕ ДОЛЖНА содержать свои элементы
  навигации (поиск, фильтры, гриды типов).
- **FR-002**: Сайдбар ДОЛЖЕН содержать: навигацию разделов (Каталог,
  Петли, Сессии, Энкаунтеры), поиск, дерево сущностей сгруппированное
  по типам с вложенностью через `contains`.
- **FR-003**: Поиск в сайдбаре ДОЛЖЕН фильтровать дерево в реальном
  времени (по мере ввода). Результат — плоский список совпадений
  внутри типов.
- **FR-004**: Ноды с дочерними (`contains`-связями) ДОЛЖНЫ работать как
  разворачиваемые узлы: стрелка для expand/collapse, клик по названию
  для открытия карточки.
- **FR-005**: Ноды без дочерних ДОЛЖНЫ открывать карточку в основной
  области при клике.
- **FR-006**: Основная область ДОЛЖНА показывать приветственный экран,
  когда ни одна сущность не выбрана.
- **FR-007**: Страница `/c/[slug]/catalog` ДОЛЖНА показывать
  приветственный экран. Страница `/c/[slug]/catalog/[id]` ДОЛЖНА
  показывать карточку сущности без собственной навигации.
- **FR-008**: Связи типа `contains` НЕ ДОЛЖНЫ отображаться в разделе
  "Связи" на карточке (они видны как дерево в сайдбаре).
- **FR-009**: Клик по связи на карточке ДОЛЖЕН открывать целевую
  сущность (обновить URL → карточка справа, подсветка в сайдбаре).
- **FR-010**: Кнопка "Создать" ДОЛЖНА открывать форму создания
  сущности в основной области.
- **FR-011**: Создание нового типа сущности ДОЛЖНО быть доступно
  на экране выбора типа при создании ноды.
- **FR-012**: "Аппы" (энкаунтеры, петли, сессии) ДОЛЖНЫ быть
  доступны через навигацию в сайдбаре и работать в основной области
  изолированно от каталога.

### Key Entities

Нет новых сущностей. Редизайн UI над существующей моделью данных.

## Success Criteria

### Measurable Outcomes

- **SC-001**: Навигация к любой сущности — исключительно через сайдбар.
  Ноль дублирующих элементов навигации в основной области.
- **SC-002**: ДМ находит сущность по имени за менее чем 5 секунд
  (поиск в сайдбаре → клик → карточка). Не хуже текущего.
- **SC-003**: Навигация по вложенности (expand → expand → click) —
  менее 3 кликов до любого уровня.
- **SC-004**: Переключение между каталогом и аппом — 1 клик в сайдбаре.
- **SC-005**: Создание сущности — менее 30 секунд.

## Assumptions

- Текущий сайдбар (`catalog-sidebar.tsx`) уже содержит 80% нужной
  функциональности: дерево, поиск, навигацию разделов. Редизайн —
  это в основном УДАЛЕНИЕ лишнего из основной области, а не создание
  нового.
- URL-структура `/c/[slug]/catalog/[id]` сохраняется.
- Данные, компоненты карточки (node-detail), формы создания,
  компоненты аппов — остаются. Меняется только layout и удаляются
  дублирующие навигационные элементы.
- Аутентификация — вне скоупа.
