# Feature Specification: Трекер энкаунтера v2 — Excel-first

**Feature Branch**: `005-encounter-tracker-v2`
**Created**: 2026-04-15
**Status**: Draft
**Input**: IDEA-010 (Excel-first redesign). Полная замена UI трекера
энкаунтера. Данные и схема БД сохраняются. Меняется только оболочка.

**Context**: Трекер v1 работает, но UX перегружен кастомными контролами.
Каждое поле имеет свой уникальный способ редактирования (кнопки +/−,
поповеры, дропдауны). ДМ на сессии хочет скорость и гибкость Excel-таблицы,
а не dashboard с виджетами.

**Principle**: "Сначала перенеси систему как есть, потом улучшай точечно."
Excel — это система, которую ДМ уже знает. Приближаемся к ней.

## User Scenarios & Testing

### User Story 1 — ДМ ведёт бой (Priority: P1) 🎯 MVP

ДМ сидит за столом, перед ним ноутбук. Идёт боевка. ДМ должен быстро:
вписать инициативу каждому участнику, отслеживать HP (урон/лечение/прямой
ввод), продвигать ходы. Всё должно работать так же быстро, как ввод чисел
в Excel: кликнул ячейку → вписал → Tab → следующая ячейка.

**Why this priority**: Без трекинга боя всё остальное не имеет смысла.
Это 80% использования трекера.

**Independent Test**: Открыть энкаунтер с 6 участниками → кликнуть на
ячейку инициативы первого → вписать 18 → Tab → вписать 14 → ... →
Tab до HP → вписать урон → вся таблица обновляется без перезагрузки.

**Acceptance Scenarios**:

1. **Given** энкаунтер с 6 участниками, **When** ДМ кликает на ячейку
   инициативы и вводит число, **Then** участник пересортировывается
   по убыванию инициативы.
2. **Given** энкаунтер с заполненной инициативой, **When** ДМ кликает
   на ячейку HP и вводит новое значение, **Then** HP обновляется,
   полоска HP отражает изменение.
3. **Given** энкаунтер с заполненной инициативой, **When** ДМ нажимает
   Tab в ячейке инициативы, **Then** фокус переходит на следующую
   редактируемую ячейку в строке (HP), а не на случайный элемент.
4. **Given** участник с HP = 0, **When** ДМ смотрит на таблицу,
   **Then** строка визуально отличается (приглушённая, зачёркнутое имя).
5. **Given** энкаунтер в ходе боя, **When** ДМ нажимает "Следующий
   ход" или горячую клавишу, **Then** выделение текущего хода
   перемещается на следующего активного участника.

---

### User Story 2 — ДМ наполняет энкаунтер участниками (Priority: P1)

ДМ создал новый энкаунтер и добавляет участников: PC из каталога
(авто-HP), монстров из каталога (авто-HP + клонирование ×N), или
кого-то произвольного вручную.

**Why this priority**: Без участников нечего трекать.

**Independent Test**: Создать энкаунтер → вписать "Тролль" в строку
добавления → увидеть подсказку из каталога → выбрать → указать ×3 →
три Тролля-1/2/3 с правильным HP в таблице.

**Acceptance Scenarios**:

1. **Given** пустой энкаунтер, **When** ДМ вводит имя в строку
   добавления, **Then** появляется автокомплит из каталога (НПС,
   монстры, PC).
2. **Given** ДМ выбрал монстра из каталога, **When** указывает
   количество 3, **Then** в таблице появляются Монстр 1, Монстр 2,
   Монстр 3 с HP из каталога.
3. **Given** ДМ вводит имя, которого нет в каталоге, **When**
   нажимает Enter, **Then** участник добавляется как "ручной"
   (без node_id).
4. **Given** участник в таблице, **When** ДМ нажимает кнопку
   клонирования, **Then** появляется копия с правильным порядковым
   номером и полным HP.

---

### User Story 3 — ДМ отмечает условия и эффекты (Priority: P2)

Во время боя ДМ ставит условия (ослеплён, отравлен) и эффекты
(Благословение, Щит Веры). Условия — из стандартного списка DnD 5e.
Эффекты — произвольные строки или из каталога кампании.

**Why this priority**: Условия влияют на механику боя, но без них
трекер всё ещё полезен.

**Independent Test**: Кликнуть на ячейку условий → начать печатать
"осл" → увидеть "Ослеплённый" в автокомплите → выбрать → бейджик
появляется в ячейке.

**Acceptance Scenarios**:

1. **Given** участник без условий, **When** ДМ кликает на ячейку
   условий и начинает печатать, **Then** появляется автокомплит
   с DnD условиями.
2. **Given** участник с условием "Ослеплённый", **When** ДМ кликает
   на бейджик, **Then** условие удаляется.
3. **Given** участник, **When** ДМ кликает на ячейку эффектов
   и вводит произвольный текст, **Then** эффект добавляется как
   свободный текст (не обязательно из каталога).

---

### User Story 4 — ДМ управляет энкаунтерами (Priority: P2)

ДМ видит список всех энкаунтеров кампании, может создать новый
(пустой или из шаблона), открыть существующий, завершить бой.

**Why this priority**: Навигация нужна, но для MVP достаточно
одного энкаунтера.

**Independent Test**: Открыть /encounters → увидеть список → кликнуть
"Создать" → ввести название → попасть на пустой энкаунтер.

**Acceptance Scenarios**:

1. **Given** ДМ на странице энкаунтеров, **When** видит список,
   **Then** энкаунтеры отсортированы по дате, активные наверху.
2. **Given** ДМ нажимает "Создать", **When** вводит название,
   **Then** создаётся пустой энкаунтер и ДМ перенаправляется на него.
3. **Given** завершённый энкаунтер, **When** ДМ открывает его,
   **Then** таблица показана в режиме "только чтение".

---

### Edge Cases

- Что если ДМ вводит отрицательный HP? → Clamp к 0.
- Что если HP > max HP? → Clamp к max HP. Temp HP — отдельное поле.
- Пустая инициатива? → Участник на "скамейке" (внизу таблицы, без номера хода).
- Два участника с одинаковой инициативой? → Сортировка по sort_order (порядку добавления). ДМ может дробить (18.1, 18.2).
- Что если ДМ удаляет участника, чей сейчас ход? → current_turn_id → SET NULL, ход переходит к следующему.
- Пустой энкаунтер? → Показать подсказку "Добавьте участников".
- Длинное имя участника? → Truncate с tooltip.

## Requirements

### Functional Requirements

- **FR-001**: Таблица энкаунтера ДОЛЖНА отображать участников в формате
  editable grid: каждая ячейка — редактируемая по клику.
- **FR-002**: Навигация между ячейками ДОЛЖНА работать через Tab
  (вправо) и Shift+Tab (влево) внутри строки.
- **FR-003**: Столбцы таблицы: роль (цвет), инициатива, имя, условия,
  эффекты, HP (current / max), temp HP.
- **FR-004**: Изменения ДОЛЖНЫ сохраняться автоматически при выходе
  из ячейки (blur/Enter/Tab). Без кнопки "Сохранить".
- **FR-005**: Сортировка по инициативе (убывание). Участники без
  инициативы — в конце списка ("скамейка").
- **FR-006**: HP ДОЛЖЕН редактироваться напрямую (ввести число)
  И через дельты (ввести "-14" или "+7" для быстрого урона/лечения).
- **FR-007**: Визуальные состояния строк: текущий ход (выделение),
  HP=0 (приглушённая строка), is_active=false (убран из боя).
- **FR-008**: Раунды: счётчик + кнопка "Следующий ход" (переход
  к следующему активному участнику, в конце списка → новый раунд).
- **FR-009**: Добавление участников: строка ввода внизу таблицы
  с автокомплитом из каталога. Поддержка ×N (клонирование).
- **FR-010**: Действия над участником: клонировать, убрать/вернуть
  в бой, удалить. Доступны через контекстное меню или иконки.
- **FR-011**: Условия: автокомплит из стандартных DnD условий.
  Отображаются как компактные бейджики в ячейке.
- **FR-012**: Эффекты: автокомплит из каталога + свободный ввод.
  Отображаются как бейджики.
- **FR-013**: Роли участников (PC, союзник, враг, нейтральный)
  ДОЛЖНЫ визуально различаться цветом строки.
- **FR-014**: Имя участника ДОЛЖНО быть ссылкой на карточку ноды
  в каталоге (если привязан к node_id).
- **FR-015**: Энкаунтер ДОЛЖЕН иметь статус: active / completed.
  Completed = read-only.
- **FR-016**: Список энкаунтеров: сортировка по дате, статус-бейдж,
  создание нового, создание из шаблона.
- **FR-017**: Сохранение как шаблон: кнопка на странице энкаунтера.
- **FR-018**: Существующие данные (энкаунтеры, участники, шаблоны)
  ДОЛЖНЫ отображаться без миграций — схема БД не меняется.
- **FR-019**: Статблок-ссылка: если у ноды есть statblock_url,
  показать иконку-ссылку рядом с именем.

### Key Entities

Все сущности уже существуют в БД. Новых таблиц и миграций нет.

- **Энкаунтер (encounter)**: Бой с названием, статусом, раундом,
  текущим ходом, деталями (jsonb).
- **Участник (encounter_participant)**: Строка в таблице боя.
  Имя, инициатива, HP, temp HP, роль, условия, эффекты, привязка
  к ноде каталога.
- **Шаблон (encounter_template)**: Сохранённый набор участников
  для клонирования в новый энкаунтер.

## Success Criteria

### Measurable Outcomes

- **SC-001**: ДМ заполняет инициативу для 6 участников за менее
  чем 20 секунд (клик → число → Tab → число → ... — без
  переключения между контролами).
- **SC-002**: Урон одному участнику — 1 действие (ввести "-14"
  в ячейку HP). Не 3 действия (ввести число → нажать кнопку "−").
- **SC-003**: Все существующие энкаунтеры и участники видны
  в новом интерфейсе без потери данных.
- **SC-004**: Добавление 3 одинаковых монстров из каталога — менее
  10 секунд.
- **SC-005**: Трекер полностью функционален на экране 1024px.
  На 375px — горизонтальный скролл с фиксированным столбцом имени.

## Assumptions

- Схема БД не меняется. Все миграции (002–007) уже применены.
- Server actions (lib/encounter-actions.ts) переиспользуются
  или рефакторятся, но API к Supabase остаётся тем же.
- Шаблоны (lib/template-actions.ts) переиспользуются.
- Маршруты: `/c/[slug]/encounters` (список), `/c/[slug]/encounters/[id]`
  (боевой экран) — не меняются.
- Навигация в layout.tsx (ссылка "Энкаунтеры") уже есть.
- Encounter details card (JSONB) может быть перенесён как есть
  или упрощён — не блокирующее требование.
