# Feature Specification: Авторизация, роли и RLS

**Feature Branch**: `006-auth-and-roles`
**Created**: 2026-04-19
**Status**: Draft v2 (после clarify про подход без email)

## Миссия

Превратить приложение из «одна публичная кампания с открытым доступом»
в «каждый пользователь видит свои кампании и играет в них с правильными
правами». ДМ выдаёт игроку логин и начальный пароль устно или в чате,
игрок при первом входе меняет пароль. Email и регистрация не нужны —
это камерная домашняя игра, где все друг друга знают.

Фундамент под:
- Мобилку игрока (spec-007 этап 5)
- IDEA-015 (игрок заявляет → ДМ подтверждает)
- IDEA-009 (realtime)
- Несколько кампаний у одного пользователя (в будущем)

## Философия auth'а

**Домашние игры, не SaaS.** ДМ знает всех своих игроков лично.
Не нужна проверка email, password reset через почту, капча, 2FA,
recaptcha, magic links, OAuth. Нужен логин, пароль и кнопка
«Сбросить» для ДМа на случай «забыл».

Логин = позывной в команде (например, «andrey», «zhenya», «katya»),
одинаковый для ДМа и игрока, одинаковый для всех кампаний, где
пользователь состоит.

Email вообще не участвует в UX. Внутри Supabase Auth мы используем
синтетический email вида `{login}@mol.local` — это чисто техническое
требование Supabase, пользователю оно невидимо.

## User Scenarios & Testing

### User Story 1 — ДМ логинится и видит свою кампанию (Priority: P1)

ДМ открывает сайт, вводит логин и пароль, попадает на свою кампанию.
Если это первый вход после создания аккаунта, сначала просят сменить
пароль.

**Why this priority**: Без этого не работает ничего. Минимальный
фундамент, после которого ДМ может начать игру как раньше.

**Independent Test**: Открыть сайт → /login → ввести логин+пароль →
попасть на /c/mat-ucheniya/catalog (или /onboarding, если первый вход).

**Acceptance Scenarios**:
1. **Given** ДМ не залогинен, **When** открывает `/c/mat-ucheniya/catalog`,
   **Then** редиректится на `/login`.
2. **Given** ДМ на `/login`, **When** вводит корректные логин+пароль
   и жмёт «Войти», **Then** залогинен и видит свою кампанию.
3. **Given** ДМ на `/login`, **When** вводит неверные данные,
   **Then** видит ошибку «Неверный логин или пароль» без подсказки,
   что именно не так.
4. **Given** ДМ залогинен и owner кампании, **When** открывает каталог,
   **Then** видит всех сущностей и может создавать/редактировать.

### User Story 2 — Первый вход: обязательная смена пароля (Priority: P1)

Пользователь (неважно какой роли) входит с начальным паролем, который
ему выдал ДМ. Система видит флаг `must_change_password=true` и
заставляет задать свой пароль, прежде чем пустить куда-либо ещё.

**Why this priority**: Без этого начальный пароль остаётся в истории
чатов Telegram навсегда. Security-минимум для домашней игры.

**Independent Test**: ДМ создаёт аккаунт игроку с паролем `temp123`.
Игрок логинится с этим паролем → попадает на /onboarding, не может
никуда уйти. Задаёт новый пароль → попадает в кампанию.

**Acceptance Scenarios**:
1. **Given** игрок только что получил аккаунт от ДМа, **When** впервые
   логинится с начальным паролем, **Then** редиректится на `/onboarding`.
2. **Given** игрок на `/onboarding`, **When** пытается перейти на любой
   `/c/*` URL, **Then** его редиректит обратно на `/onboarding`.
3. **Given** игрок на `/onboarding`, **When** вводит новый пароль
   (не равный старому, минимум 8 символов) и подтверждение,
   **Then** пароль меняется, флаг снимается, редирект на кампанию.
4. **Given** игрок задаёт пароль равный старому, **Then** ошибка
   «Новый пароль должен отличаться от начального».

---

### User Story 3 — Owner создаёт аккаунт второму ДМу (Priority: P1)

Owner хочет добавить второго ДМа. Заходит в «Участники», вводит логин
и начальный пароль, выбирает роль «ДМ». Получает готовые данные,
отправляет их коллеге в Telegram. Коллега входит, меняет пароль
при первом входе.

**Why this priority**: В текущей кампании несколько ДМов
(constitution.md). Без этого они не смогут работать.

**Independent Test**: Owner в `/members` → добавить → логин `vasya`,
пароль `dm-temp`, роль «ДМ». Потом вторая вкладка → /login →
`vasya`+`dm-temp` → /onboarding → новый пароль → кампания, полные
права ДМа.

**Acceptance Scenarios**:
1. **Given** Owner на `/members`, **When** вводит логин+пароль+роль dm
   и жмёт «Создать», **Then** создаётся auth-юзер + запись в
   `user_profiles` (must_change_password=true) + запись в
   `campaign_members` (role=dm). Owner видит нового ДМа в списке.
2. **Given** Второй ДМ логинится с начальными данными, **When**
   меняет пароль, **Then** попадает в кампанию с правами ДМа —
   может создавать/редактировать, но не может зайти в /members.
3. **Given** Owner в `/members` выбирает «Сбросить пароль» для
   второго ДМа, **When** вводит новый начальный пароль, **Then**
   пароль меняется, флаг `must_change_password` возвращается в true.
   При следующем входе второму ДМу снова придётся менять.

---

### User Story 4 — Owner создаёт аккаунт игроку и привязывает к PC (Priority: P1)

То же, что US3, но роль `player`, и сразу можно привязать к ноде-PC.

**Independent Test**: Owner добавляет логин `zhenya`, пароль `player-temp`,
роль player, привязывает к PC «Альд». Логинится как zhenya → меняет
пароль → видит кампанию как игрок.

**Acceptance Scenarios**:
1. **Given** Owner на `/members`, **When** создаёт аккаунт с ролью
   player и сразу выбирает PC-ноду из списка,
   **Then** создаётся аккаунт + `nodes.owner_user_id` на выбранной
   PC устанавливается в новый user_id.
2. **Given** Owner на карточке PC без владельца, **When** в UI ноды
   выбирает «Назначить игрока», **Then** видит список player'ов
   кампании и может привязать существующего.
3. **Given** PC уже имеет владельца, **When** Owner назначает другого,
   **Then** появляется подтверждение «заменить?». После подтверждения
   старый игрок теряет права на этот PC, новый получает.

---

### User Story 5 — Игрок видит кампанию с правами чтения (Priority: P1)

Игрок логинится, видит свою кампанию. Каталог читает весь. Может
редактировать только свой PC. Кнопки «Создать», «Редактировать»
(на чужих нодах) скрыты. Настройки скрыты. Участники скрыты.

**Why this priority**: Собственно ради этого всё это делается — игроки
видят свой срез мира.

**Independent Test**: Логин игроком → каталог открывается, список
видно, кнопки «+ Создать» нет. Открыть NPC — нет «Редактировать».
Открыть свой PC — есть.

**Acceptance Scenarios**:
1. **Given** игрок залогинен, **When** открывает каталог, **Then**
   видит всех сущностей, нет кнопок создания/редактирования.
2. **Given** игрок на своей PC-ноде, **When** смотрит, **Then** видит
   «Редактировать», может менять `fields` и `content` ноды.
3. **Given** игрок на чужой PC-ноде или NPC, **When** смотрит,
   **Then** кнопки «Редактировать» нет, и API-запрос на UPDATE
   отклоняется через RLS.
4. **Given** игрок пытается попасть на `/c/[slug]/settings` или
   `/c/[slug]/members`, **Then** 403 или редирект на каталог.

---

### User Story 6 — Смена своего пароля в настройках (Priority: P2)

Любой пользователь (включая игрока) может сменить свой пароль
в личных настройках. Это не настройки кампании — это настройки
пользователя.

**Independent Test**: Любой юзер → /account → «Сменить пароль» →
вводит старый+новый → работает.

**Acceptance Scenarios**:
1. **Given** юзер на `/account`, **When** вводит старый пароль+
   новый+подтверждение и жмёт «Сменить», **Then** пароль меняется.
2. **Given** юзер ввёл неверный старый пароль, **Then** ошибка,
   пароль не меняется.

---

### User Story 7 — Миграция существующих данных и создание owner'а (Priority: P1)

Сейчас все данные «ничьи». После включения auth + RLS ими кто-то
должен владеть, иначе они исчезнут. Нужен способ засидить первого
owner'а без веб-UI (логиниться ещё некем).

**Why this priority**: Блокирует всё.

**Independent Test**: Запустить `npm run seed-owner` локально с
указанными логином и паролем → логин через UI работает → owner
видит все ноды.

**Acceptance Scenarios**:
1. **Given** миграции применены и базу запустили первый раз,
   **When** запускается CLI-скрипт `scripts/seed-owner.ts` с
   параметрами `--login --password --campaign`, **Then** создаётся
   auth-юзер через service role + `user_profiles` (с
   must_change_password=false, т.к. owner сам вводил) + запись
   `campaign_members (role=owner)`.
2. **Given** owner залогинен, **When** открывает каталог, **Then**
   видит все 150+ существующих нод без потерь.
3. **Given** скрипт запущен повторно с тем же логином, **Then**
   идемпотентно завершается — не дублирует записи.

---

### User Story 8 — Выход из аккаунта (Priority: P2)

Пользователь выходит. Сессия удаляется. Следующий раз снова /login.

**Acceptance Scenarios**:
1. **Given** юзер залогинен, **When** жмёт «Выйти» в хедере,
   **Then** сессия уничтожается, редирект на `/login`.

---

### Edge Cases

- **Логин занят**: Owner пытается создать `zhenya`, а он уже
  существует. Ошибка «Логин уже используется».
- **Игрок забыл пароль**: ДМ в `/members` → «Сбросить» → генерит
  новый начальный пароль → передаёт игроку. Флаг
  must_change_password ставится в true.
- **Игрок не привязан к PC**: видит кампанию, но нет «свой
  персонаж». Показываем «у вас нет назначенного персонажа —
  обратитесь к ДМу».
- **Owner удаляет себя**: блокируем. Перед удалением нужно
  передать права owner'а другому ДМу.
- **Игрок переходит в новую кампанию**: пока не в скоупе (одна
  кампания). Архитектурно — новая запись в campaign_members с
  другим campaign_id для того же user_id.
- **Два пользователя хотят один логин**: первый выигрывает. Логин
  глобально уникальный (это технический email в auth.users).
- **Начальный пароль попал в чужие руки**: поскольку первая
  же сессия требует смены — risk маленький. Но ДМ может превентивно
  сбросить через /members, не дожидаясь сессии.

## Requirements

### Functional Requirements

- **FR-001**: Система ДОЛЖНА использовать Supabase Auth с password-
  based signin. Логин пользователя — произвольная строка
  (`^[a-z0-9_-]{3,32}$`). Внутри Supabase хранится как email
  вида `{login}@mol.local` (технический суффикс, не показывается
  в UI).
- **FR-002**: Email-верификация в Supabase ДОЛЖНА быть отключена.
  Signup производится админом (service role) с `email_confirm: true`.
- **FR-003**: Таблица `user_profiles` ДОЛЖНА содержать:
  - `user_id uuid PK FK auth.users CASCADE`
  - `login text UNIQUE NOT NULL CHECK (login ~ '^[a-z0-9_-]{3,32}$')`
  - `display_name text`
  - `must_change_password boolean NOT NULL DEFAULT true`
  - `created_at timestamptz DEFAULT now()`
- **FR-004**: Таблица `campaign_members` ДОЛЖНА содержать:
  - `id uuid PK`
  - `campaign_id uuid FK campaigns CASCADE`
  - `user_id uuid FK auth.users CASCADE`
  - `role text CHECK (role IN ('owner', 'dm', 'player'))`
  - `created_at timestamptz DEFAULT now()`
  - UNIQUE (campaign_id, user_id)
- **FR-005**: Кампания ДОЛЖНА иметь ровно одного owner.
  Может иметь N dm и N player.
- **FR-006**: Ноды типа `character` ДОЛЖНЫ иметь опциональное
  `owner_user_id uuid FK auth.users NULL`.
- **FR-007**: RLS на `nodes`: SELECT — все члены кампании.
  INSERT/UPDATE/DELETE — owner/dm. Исключение — узлы type character
  с `owner_user_id = auth.uid()`: их может UPDATE владелец.
- **FR-008**: RLS на `edges`: SELECT — все члены кампании.
  INSERT/UPDATE/DELETE — owner/dm.
- **FR-009**: RLS на `encounters`, `encounter_participants`:
  SELECT — все члены кампании. Модификация — owner/dm.
- **FR-010**: RLS на `campaign_members`: SELECT — все члены этой
  кампании. INSERT/UPDATE/DELETE — только owner.
- **FR-011**: RLS на `user_profiles`: SELECT — сам юзер +
  члены общих кампаний (для отображения display_name и login
  в /members). UPDATE (must_change_password, display_name) —
  сам юзер. Все остальные модификации — только через service role.
- **FR-012**: Страница `/login` принимает логин + пароль, вызывает
  `supabase.auth.signInWithPassword({ email: '{login}@mol.local',
  password })`.
- **FR-013**: Страница `/onboarding` доступна только когда
  `user_profiles.must_change_password = true`. Middleware
  редиректит с любых других страниц сюда, пока флаг не снят.
  При успешной смене — update auth-пароля + user_profiles.must_
  change_password=false.
- **FR-014**: Страница `/account` доступна любому залогиненному.
  Позволяет сменить пароль (со старым для подтверждения).
- **FR-015**: Страница `/c/[slug]/members` доступна только owner.
  Позволяет: создать аккаунт (логин+пароль+роль+опц. PC), удалить
  аккаунт, сменить роль, сбросить пароль, (пере)назначить PC
  игроку. Все операции создания/изменения юзеров идут через Server
  Actions с service role.
- **FR-016**: Страница `/c/[slug]/settings` доступна owner и dm.
- **FR-017**: Middleware ДОЛЖЕН:
  - редиректить неавторизованных с `/c/*` на `/login`
  - редиректить авторизованных с must_change_password=true
    с любых `/c/*` на `/onboarding`
  - редиректить авторизованных с `/login` на `/` (чтобы не
    возвращаться на логин-страницу при залогиненной сессии)
- **FR-018**: Главная `/`:
  - если не залогинен → `/login`
  - если залогинен и must_change_password → `/onboarding`
  - если залогинен и единственная кампания → `/c/[slug]/catalog`
  - если залогинен и несколько кампаний → список кампаний
- **FR-019**: Хедер показывает логин (display_name или login)
  текущего юзера и кнопку «Выйти».
- **FR-020**: UI ДОЛЖЕН скрывать кнопки действий, недоступных
  текущей роли (create/edit для players, settings/members для
  не-owner'ов). RLS — последняя линия защиты.
- **FR-021**: CLI-скрипт `scripts/seed-owner.ts` создаёт первого
  owner'а для указанной кампании. Аргументы: `--login`, `--password`,
  `--campaign`. Использует service role key из ENV. Идемпотентный.
- **FR-022**: `SUPABASE_SERVICE_ROLE_KEY` ДОЛЖЕН храниться только
  в Vercel ENV и использоваться только в Server Actions / API
  routes. Никогда не в клиентском коде.

### Key Entities

- **User (auth.users)**: стандартный Supabase. Email = `{login}@mol.local`.
  Не редактируем вручную.
- **UserProfile**: наш слой поверх User. Хранит человеко-читаемый
  логин, display_name, флаг смены пароля.
- **CampaignMember**: связка user + campaign + role.
- **Character ownership**: `nodes.owner_user_id` на PC-нодах.

- Q: Игрок может редактировать свой PC (fields + content)? →
  **A: да.** Нужно для автономного ведения листа персонажа
  (IDEA-017). HP и условия в энкаунтере — отдельная тема, это
  encounter_participants, не ноды. Пока в этом спеке игрок
  encounter_participants не меняет (это инкремент 4+).

### Session 2026-04-19 (пересмотр подхода: без email)

- Q: Email вообще нужен пользователю? → **A: нет.** Домашняя
  игра, все друг друга знают. ДМ выдаёт логин+пароль в чате.
  Внутри Supabase используем технический email `{login}@mol.local`,
  игрок его не видит никогда.

- Q: Password reset self-service или только через ДМа? →
  **A: только через ДМа.** Self-service reset требует email для
  ссылки. Если игрок забыл — пишет ДМу, тот сбрасывает в /members.

- Q: Force password change при первом входе? → **A: да.**
  Начальный пароль всегда висит в чате Telegram'а — критично,
  чтобы юзер его сменил при первой же сессии.

- Q: Минимальная длина пароля? → **A: 8 символов.** Просто,
  безопасно для домашней игры, не мешает.

- Q: Логин — per-campaign или глобальный? → **A: глобальный.**
  Логин = email в auth.users = уникален системно. Один человек
  = один аккаунт, даже если он в нескольких кампаниях с разными
  ролями.

  Нужны, чтобы запустить `scripts/seed-owner.ts`. Можно в приватном

## Success Criteria

- **SC-001**: Owner логинится с начальным паролем и видит все 150+
  нод кампании — 0 потерь.
- **SC-002**: Создание аккаунта игроку в /members занимает ≤ 30
  секунд.
- **SC-003**: Игрок логинится первый раз за ≤ 60 секунд (включая
  смену пароля).
- **SC-004**: Аноним на `/c/*` URL → редирект на /login за < 1 сек.
- **SC-005**: Игрок видит каталог, но не видит кнопок создания ни
  в UI, ни может создать через API (RLS блокирует).
- **SC-006**: Игрок, привязанный к PC «Альд», может редактировать
  fields+content Альда, не может редактировать других PC или NPC.

## Assumptions & Non-Goals

**Assumptions**:
- Supabase Cloud Free Tier (до ~50000 MAU) достаточно.
- Единственная кампания — `mat-ucheniya`.
- ДМ имеет канал связи (Telegram и т.п.) для передачи начальных
  паролей игрокам. Мы не решаем задачу secure password delivery.
- Сервер доверяет своему БД — service role key хранится в ENV
  Vercel и не утекает.

**Non-Goals** (не делаем в этом спеке):
- Self-signup (нет регистрации через UI).
- Self-service password reset (без ДМа не сбросить).
- Email и email-уведомления вообще.
- Social login, 2FA.
- Мобилка игрока (spec-007 этап 5).

## Incremental Roadmap

По принципу V конституции — каждый инкремент играбелен.

1. **Инкремент 1 (US1, US2, US7, US8)** — фундамент:
   - Миграция 024: `user_profiles`, `campaign_members`,
     `nodes.owner_user_id`, базовые RLS policies
     (все залогиненные члены кампании видят всё, модификация
     через service role или по роли).
   - `scripts/seed-owner.ts` — засеивает первого owner'а.
   - Страницы: `/login`, `/onboarding`, `/account`, `/` (редирект).
   - Middleware.
   - Хедер с кнопкой «Выйти».

   После этого инкремента: ДМ логинится, видит кампанию как раньше,
   анонимов нет.

2. **Инкремент 2 (US3)** — приглашение ДМов:
   - Страница `/c/[slug]/members` для owner'а.
   - Server Actions `createMember(login, password, role)` и
     `resetPassword(user_id, new_password)`.

   После: в кампании несколько ДМов.

3. **Инкремент 3 (US4)** — игроки + привязка к PC:
   - Расширение `/members` — создание с ролью player,
     выбор привязки к PC (dropdown с character-нодами без
     владельца).
   - UI на карточке PC: секция «Владелец» с dropdown player'ов.

   После: все игроки в системе, привязаны к PC, но пока видят
   всё как ДМ (RLS ещё не жмёт).

4. **Инкремент 4 (US5, US6)** — жёсткие RLS + условный UI:
   - Полные RLS-политики (character-ownership edge case).
   - UI прячет кнопки create/edit/delete от игроков.
   - Страница /account для смены собственного пароля.
   - Тесты: игрок не может создать/удалить через API.

   После: spec-006 закрыт, можно начинать spec-007 этап 5
   (мобилка игрока).
