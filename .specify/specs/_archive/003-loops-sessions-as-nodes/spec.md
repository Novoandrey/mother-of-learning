# Feature Specification: Петли и сессии как ноды графа

**Feature Branch**: `003-loops-sessions-as-nodes`
**Created**: 2026-04-13
**Status**: Draft
**Input**: Петли и сессии хранятся в отдельных таблицах (loops, sessions),
не участвуют в графе сущностей, не поддерживают теги, связи, поиск,
markdown-контент. Это нарушает принцип I конституции (атомарность данных)
и создаёт параллельную модель, которую придётся поддерживать отдельно.

## User Scenarios & Testing

### User Story 1 — ДМ ищет сессию в каталоге (Priority: P1)

ДМ готовится к игре и хочет найти сессию, где "Бенисек был арестован",
через общий поиск — тот же, которым ищет НПС. Сейчас сессии живут
в отдельном разделе и не участвуют в поиске каталога.

**Why this priority**: Поиск — главная точка входа. Если сессии
не находятся одним поиском с НПС, ДМ должен помнить, где что лежит.

**Independent Test**: Ввести "Бенисек" в поиск каталога → увидеть
и НПС "Бенисек", и сессию "Допрос Бенисека" в результатах.

**Acceptance Scenarios**:

1. **Given** ДМ на каталоге, **When** вводит "Допрос" в поиск,
   **Then** видит сессию "Допрос Бенисека" в результатах.
2. **Given** ДМ на карточке сессии, **When** смотрит поля,
   **Then** видит номер сессии, петлю, рекап, дату игры.
3. **Given** ДМ на карточке сессии, **When** добавляет тег "ключевая",
   **Then** тег сохраняется и работает так же, как теги на НПС.

---

### User Story 2 — ДМ связывает сессию с НПС (Priority: P1)

ДМ хочет отметить, что в сессии "Ночь пожара" участвовали Ильза,
Зориан и Бенисек. Сейчас это невозможно — сессии не в графе.

**Why this priority**: Перекрёстные ссылки — принцип II конституции.
"Где упоминается Ильза?" должен показывать сессии.

**Independent Test**: Открыть сессию → добавить связь "участвует" → Ильза.
Открыть Ильзу → увидеть обратную ссылку на сессию.

**Acceptance Scenarios**:

1. **Given** ДМ на карточке сессии, **When** добавляет связь
   типа "участвует" к НПС "Ильза", **Then** связь создаётся.
2. **Given** ДМ на карточке "Ильза", **When** смотрит входящие связи,
   **Then** видит обратную ссылку на сессию "Ночь пожара".

---

### User Story 3 — ДМ работает с петлями через привычный UI (Priority: P1)

ДМ открывает раздел "Петли" и видит тот же дашборд: список петель,
текущая петля, сессии в петле. Всё работает как раньше, но данные
теперь из графа.

**Why this priority**: Миграция не должна ломать рабочий флоу.
Специализированные страницы остаются — меняется только источник данных.

**Independent Test**: Открыть /loops → увидеть петли с сессиями.
Создать петлю → она появляется и в /loops, и в каталоге.

**Acceptance Scenarios**:

1. **Given** ДМ на странице петель, **When** выбирает петлю 3,
   **Then** видит заголовок, статус, заметки и список сессий этой петли.
2. **Given** ДМ создаёт новую петлю, **When** возвращается в каталог
   и фильтрует по типу "Петля", **Then** видит новую петлю в списке.
3. **Given** ДМ на странице сессии, **When** нажимает "Редактировать",
   **Then** может изменить рекап, заметки ДМа, петлю, дату.

---

### User Story 4 — ДМ пишет markdown на странице петли (Priority: P2)

ДМ хочет добавить развёрнутое описание петли в markdown — карту
событий, таймлайн, планы. Сейчас у петель есть только поле "notes".
Как ноды, петли получат полноценный markdown-блок.

**Why this priority**: Бонус миграции — функциональность "бесплатно"
из существующей инфраструктуры нод.

**Independent Test**: Открыть петлю → раздел "Контент" → написать
markdown → сохранить → отображается с форматированием.

**Acceptance Scenarios**:

1. **Given** ДМ на карточке петли, **When** нажимает "Написать"
   в блоке Контент, **Then** видит markdown-редактор.
2. **Given** ДМ сохранил markdown на петле, **When** ищет текст
   из контента в каталоге, **Then** петля находится.

---

### Edge Cases

- Что происходит с хрониками, которые ссылаются на loop_number?
  → Хроники продолжают работать через loop_number (поле в fields ноды).
  Миграция на edge-связь — отдельная задача.
- Как работает сортировка петель? → По полю `fields.number`,
  не по title. Специализированная страница знает про это поле.
- Уникальность номера сессии? → Проверяется на уровне приложения
  (не БД), так как JSONB не поддерживает UNIQUE constraints на поля.
- Удаление петли — что с сессиями? → Сессии связаны ребром `contains`.
  Удаление ноды каскадит рёбра, но не дочерние ноды. Сессии остаются
  как "осиротевшие" — это корректное поведение.

## Requirements

### Functional Requirements

- **FR-001**: Петли ДОЛЖНЫ храниться как ноды типа `loop` в таблице `nodes`.
  Поля: number (int), title (string), status (string), notes (string).
- **FR-002**: Сессии ДОЛЖНЫ храниться как ноды типа `session` в таблице `nodes`.
  Поля: session_number (int), loop_number (int), title (string), recap (string),
  dm_notes (string), played_at (string), game_date (string).
- **FR-003**: Связь сессии с петлёй ДОЛЖНА моделироваться ребром `contains`
  (петля → contains → сессия).
- **FR-004**: Петли и сессии ДОЛЖНЫ быть доступны через общий поиск каталога,
  фильтрацию по типу, теги, markdown-контент, связи.
- **FR-005**: Специализированные страницы (/loops, /sessions, /sessions/[id])
  ДОЛЖНЫ продолжать работать, запрашивая данные из таблицы `nodes`
  вместо отдельных таблиц.
- **FR-006**: Существующие данные из таблиц `loops` и `sessions` ДОЛЖНЫ быть
  мигрированы в `nodes` + `edges` без потерь.
- **FR-007**: Типы нод `loop` и `session` ДОЛЖНЫ быть добавлены
  в `node_types` с соответствующими `default_fields`.
- **FR-008**: Таблицы `loops` и `sessions` ДОЛЖНЫ быть удалены после
  успешной миграции данных.
- **FR-009**: Хроники (chronicles) ДОЛЖНЫ продолжать работать с loop_number,
  читая его из `fields.number` ноды петли.
- **FR-010**: Формы создания/редактирования петель и сессий ДОЛЖНЫ
  сохранять данные в `nodes.fields` JSONB.

### Key Entities

- **Петля (Loop)**: Нода типа `loop`. Обязательные поля: number (int),
  status ('past'|'current'|'future'). Опциональные: title, notes.
  Связь с сессиями через ребро `contains`.
- **Сессия (Session)**: Нода типа `session`. Обязательные поля:
  session_number (int). Опциональные: title, recap, dm_notes,
  played_at, game_date, loop_number (для обратной совместимости запросов).
- **Ребро contains**: Петля → contains → Сессия. Уже существует
  как базовый тип связи.

## Clarifications

### Session 2026-04-13

- Q: search_vector строится из title + fields.description. У сессий ключевой текст — recap, не description. Как индексировать?
  → A: Вариант C — индексировать ВСЕ текстовые значения из JSONB fields целиком. Это универсально: любой тип ноды с любыми полями автоматически попадает в поиск. Триггер собирает title + content + все string-значения из fields.

## Success Criteria

### Measurable Outcomes

- **SC-001**: Поиск "Допрос" в каталоге возвращает и НПС, и сессию —
  менее 2 секунд.
- **SC-002**: Карточка сессии показывает связи с НПС — навигация
  в 1 клик (как между любыми нодами).
- **SC-003**: Страница /loops отображает те же данные, что до миграции.
  Ноль потерянной информации.
- **SC-004**: Создание новой петли или сессии — менее 30 секунд,
  сущность сразу видна и в /loops, и в каталоге.
- **SC-005**: После миграции таблицы `loops` и `sessions` удалены.
  Одна модель данных, ноль дублирования.

## Assumptions

- Хроники (chronicles.loop_number) пока остаются как есть — миграция
  хроник на edge-связи выходит за скоуп этой фичи.
- Энкаунтеры не ссылаются на петли/сессии — миграция их не затрагивает.
- Количество петель и сессий невелико (~4 петли, ~60 сессий) —
  производительность JSONB-запросов не является проблемой.
- `search_vector` уже включает `fields->>'description'`. Для сессий
  нужно убедиться, что `recap` попадает в поиск (через триггер или
  расширение search_vector).
