# Implementation Plan — 025 Backups & Restore Drill (срез 3/5)

> HOW для `spec.md`. Решения Clarify уже вшиты в спеку. Пошаговый прогон
> оператором — в `infra/backup-restore-runbook.md` (пишется на Implement, не
> здесь). Исполнение на боксе из 023–024, **не** в песочнице Claude.

## Подход

Логический бэкап всего кластера (`pg_dumpall` — захватывает **роли** Supabase
вместе со схемами/данными) → сжатие gzip → заливка в **off-box R2** через
`rclone`. Ротация — отдельным проходом по двум префиксам. Restore-drill — на
пустом self-hosted стеке из 024.

**Почему `pg_dumpall`, а не `pg_dump`:** self-hosted Supabase живёт на ролях
`anon` / `authenticated` / `service_role` / `supabase_*`. `pg_dump` одной базы
их не заберёт — restore поднимет БД, но PostgREST/Auth/RLS развалятся.
`pg_dumpall` берёт глобальные объекты (роли, права) + все базы за один проход.

**Почему plain `.sql.gz`, а не restic/borg-репозиторий:** главная цель среза —
*надёжное и прозрачное восстановление*, в т.ч. как аварийный откат в 027 под
стрессом. Голый дамп восстанавливается тривиально (`gunzip | psql`) и не
зависит от ключа/пароля репозитория (потеря которого = потеря всех бэкапов).
Дедуп/инкременты/PITR (restic, `wal-g`, `pgBackRest`) — осознанно отложены
(см. Out of Scope спека).

## Хранилище: Cloudflare R2

- **R2 = выбранное эпиком object-storage** (зафиксировано в `infra/README.md`).
  Бэкапам хранилище нужно уже сейчас (до 026), поэтому первичная настройка
  R2-аккаунта подтягивается в **этот** срез — раньше, чем планировалось под
  портреты. Портреты позже переиспользуют тот же аккаунт со своим бакетом.
- **Bucket-per-purpose:** отдельный bucket `mat-ucheniya-backups`, **scoped
  API-токен только на него** (R2 → Manage API Tokens, Object Read & Write,
  ограничить конкретным бакетом). Токен — в env на боксе, не в git.
- **Egress бесплатный** у R2 → restore (скачивание дампа) ничего не стоит;
  для drill'а и для отката в 027 это приятный бонус.
- Шифрование бэкапов **не** настраиваем (решение Clarify): bucket приватный,
  токен scoped — базовой гигиены достаточно для bcrypt-хешей + email игроков.

## Инструменты

- **`rclone`** на боксе, S3-remote, тип `Cloudflare R2` (endpoint
  `https://<accountid>.r2.cloudflarestorage.com`). Конфиг в `~/.config/rclone/
  rclone.conf` (секреты там, не в git); шаблон без значений — `rclone.conf.example`.
- **`pg_dumpall`** запускается **внутри** db-контейнера через `docker exec`
  (бинарь и сокет уже там; снаружи 5432 закрыт — см. 024). Версия `pg_dumpall`
  совпадает с сервером (PG17), мисматча нет.
- **`cron`** на боксе — daily-расписание (ориентир 03:00 UTC, ночь по EU).

## Порядок реализации (высокоуровневый)

> Внутренняя нумерация plan; дробный Step-прогон — в runbook.

1. **R2:** создать bucket `mat-ucheniya-backups` + scoped-токен (Read&Write,
   только этот бакет). Записать креды в env на боксе.
2. **rclone:** установить, завести remote из шаблона, проверить доступ
   (`rclone lsd <remote>:` / тестовый `rclone copy` мелкого файла) — US1#2.
3. **backup.sh:** `docker exec db pg_dumpall` → gzip → `rclone` в `daily/`;
   лог в файл; ошибки → ненулевой exit (US1#1, US1#5, edge «бакет недоступен»).
4. **Ручной прогон + проверка дампа:** файл уехал в бакет (`rclone ls`),
   ненулевой; `gunzip -t` проходит; в дампе есть `CREATE ROLE` для
   `anon`/`authenticated`/`service_role`/`supabase_*` (US1#1–2, edge «роли»).
5. **cron:** daily-задание (`/etc/cron.d/...` или `crontab`), служба запущена,
   подтвердить `crontab -l` / лог следующего прогона (US1#3).
6. **Ротация:** воскресная копия в `weekly/`; чистка `daily/`>30д, `weekly/`>28д;
   проверить на подкрученных метках, что старое удаляется, свежее живёт (US1#4).
7. **Drill «снёс → поднял»** (US2 — ядро среза):
   - отложить текущее состояние (data-dir `mv … .old` / снапшот) — путь назад;
   - снести БД, восстановить из последнего дампа, поднять стек;
   - `pg_isready` + `SELECT 1` + сверка служебных схем Supabase; Auth/REST
     стартуют healthy;
   - **замерить время** stop→healthy (ориентир < 5 мин);
   - проверить откат назад (вернуть `.old` за < 1 мин).
   Ключевая HOW-деталь — ниже.
8. **Runbook + чек-лист:** зафиксировать copy-paste прогон, отметить пройденный
   dry-run, записать время и дату (US3).

## Открытые HOW-детали (закрыть в runbook / отладить на drill'е)

- **Главная:** конфликт инициализации при логическом restore. Свежий
  Supabase-контейнер при старте сам создаёт роли/служебные схемы; залив
  `pg_dumpall` поверх можно словить «role already exists». Кандидаты, выбрать
  на drill'е (в этом и навык): (а) restore в свежий **ванильный** Postgres до
  Supabase-инициализации; (б) `pg_dumpall` с осторожным `--clean`/
  `--if-exists`; (в) дамп только app-схем (`pg_dump public`) + роли отдельно
  (`pg_dumpall --roles-only`) поверх уже инициализированного стека. Для пустого
  стека вскрываем это сейчас, чтобы 026/027 прошли гладко.
- Точная форма сноса БД в docker-стеке 024 (stop+`mv` data-dir vs `dropdb`/
  `createdb` внутри) и команда возврата `.old`.
- Финальное расписание/таймзона cron и место лог-файла на боксе.
- Нужен ли отдельный `restore.sh`/`rotate.sh` или удобнее блоками в `backup.sh`
  + runbook (решаем на Implement по факту размера).
