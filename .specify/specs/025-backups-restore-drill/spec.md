# Feature Specification: Backups & Restore Drill (эпик «Своя инфра», срез 3/5)

**Feature Branch**: `025-backups-restore-drill`
**Created**: 2026-06-06
**Status**: Done — in prod (chat 85–86; drill ~20 c на реальных данных)
**Input**: Продолжение эпика «Переезд на свою инфру». 024 поднял self-hosted
Supabase и доказал паритет с продом. 025 ставит **операционный навык
восстановления**: автоматические off-box бэкапы + проверенный drill
«снёс → поднял» — всё на **пустом** стеке, до того как туда поедут реальные
данные (026).

Срез эпика: 023 фундамент → 024 self-hosted Supabase → **025 бэкапы + drill**
→ 026 миграция данных → 027 cutover.

## Context

### Зачем этот срез

024 дал работающий, параллельный проду self-hosted Supabase. Но прежде чем
доверить ему реальные данные (026) и тем более прод-трафик (027), нужно
**доказать, что мы умеем его восстанавливать**. 025 — это про механику и навык,
не про данные:

1. Настроить **автоматические бэказы** PostgreSQL, уезжающие **off-box**.
2. Провести полный **drill восстановления**: снести БД на пустом стеке, поднять
   из бэкапа, проверить целостность, замерить время.
3. Зафиксировать процедуру в **runbook**, чтобы в 027 откат при неудачном
   cutover занимал минуты, а не часы и не «вспоминаем по ходу».

### На каком стеке

На self-hosted Supabase из 024 — **пустом** (только служебные схемы Supabase,
без данных приложения). Это снимает весь риск: ломать и восстанавливать нечего
ценного, прод и приложение не затронуты. Тот же бокс Hetzner CPX32 (Helsinki,
Ubuntu 24.04), Postgres в контейнере (`supabase/postgres:17.6.x`, PG17),
data-dir на bind-mount.

### Решения (зафиксированы на Clarify)

- **Куда едет бэкап**: S3-совместимое **off-box** хранилище. На той же машине
  бэкап смысла не имеет (теряем бокс — теряем бэкап). Конкретный провайдер
  (Hetzner Storage Box / Cloudflare R2 / Backblaze B2) — выбор Plan-фазы; R2 уже
  планируется под картинки, аккаунт переиспользуем.
- **Метод бэкапа (baseline)**: логический дамп (`pg_dump` / `pg_dumpall` —
  включая роли Supabase) → сжатие → заливка в бакет. Физический бэкап и PITR
  через WAL-archiving (`wal-g` / `pgBackRest`) — возможное будущее улучшение,
  не в этом срезе.
- **Ротация**: хранить последние **30 дневных + 4 недельных**, старые удалять.
- **Шифрование**: **отложено** (фаза 2, опционально). Чувствительность данных
  низкая (хеши паролей bcrypt + email игроков); базовая защита — приватный
  бакет + credentials в env, не в git.

### Делёж труда

Как в 023–024 — исполнение на боксе оператора, не в песочнице Claude:

- **Claude поставляет**: эту спеку, `plan.md`/runbook (скрипт бэкапа, скрипт
  restore, настройка cron, конфиг доступа к бакету), чек-листы валидации.
- **Оператор (Andrey + Леша) исполняет**: настраивает доступ к бакету, ставит
  cron, проверяет первый бэкап, прогоняет drill, присылает логи и тайминги.
  Claude разбирает ошибки и правит артефакты.

## Operator Scenarios & Testing

### US1 — Автоматические off-box бэкапы работают (P1)

Оператор настраивает ежедневное снятие бэкапа Postgres с заливкой в off-box
бакет и убеждается, что бэкапы создаются, уезжают и ротируются без ошибок.

#### Acceptance Scenarios

1. **Бэкап-скрипт отрабатывает вручную** — запуск скрипта руками: дамп снят
   (`pg_dump`/`pg_dumpall`), сжат, exit code 0, в логе нет `ERROR`/`FATAL`.
2. **Дамп уезжает в бакет** — после запуска файл присутствует в off-box бакете
   (виден через `rclone ls` / консоль провайдера), размер ненулевой, имя
   содержит дату/время.
3. **Расписание активно** — cron-задание daily установлено и видно
   (`crontab -l` / `/etc/cron.d/...`), служба cron запущена.
4. **Ротация работает** — после симуляции нескольких дней (или ручной правки
   меток) старые бэкапы за пределами окна (30 дневных / 4 недельных)
   автоматически удаляются; свежие остаются.
5. **Логирование** — успехи и ошибки пишутся в лог-файл; последний прогон видно
   через `tail`.

#### Edge Cases

- **Бакет недоступен** (сеть/credentials) — скрипт сообщает об ошибке в логе и
  завершается ненулевым кодом, а не «молча успешно».
- **Локальный диск заполнен** при создании временного дампа — обнаруживается и
  логируется; временный файл чистится после заливки в любом случае.
- **Роли/расширения не попали в дамп** — для self-hosted Supabase критично
  захватить роли (`anon`, `authenticated`, `service_role`, `supabase_*`);
  проверяется, что они есть в дампе.

### US2 — Полный drill восстановления на пустом стеке (P1)

Оператор проигрывает реальный сценарий потери: сносит БД и поднимает её из
последнего бэкапа, проверяя целостность и время.

#### Acceptance Scenarios

1. **Бэкап выбран и доступен** — взят последний успешный бэкап из бакета;
   скачан/доступен на боксе, файл целый (распаковывается без ошибок).
2. **Откатываемая точка сохранена** — перед сносом текущий data-dir/состояние
   отложен в сторону (`mv … .old` или снапшот), чтобы был путь назад.
3. **БД снесена и восстановлена** — Postgres остановлен, состояние очищено,
   данные восстановлены из бэкапа, Postgres поднят. Exit code 0, в логах нет
   `FATAL`.
4. **Стек снова здоров** — `pg_isready` healthy, `SELECT 1` проходит; число
   объектов/схем соответствует тому, что было до сноса (для пустого стека —
   служебные схемы Supabase на месте, Auth/PostgREST стартуют healthy).
5. **Время восстановления замерено** — зафиксирован тайминг «stop → healthy».
   Для пустого стека ориентир < 5 мин. Это будущая метрика downtime в 027.
6. **Путь назад работает** — если восстановление неудачно, отложенное состояние
   возвращается за < 1 мин (проверено).

#### Edge Cases

- **Битый бэкап** — drill обязан обнаружить повреждение на этом этапе (на пустом
  стеке), а не в 027 на реальных данных.
- **Несовместимость мажора Postgres** — бэкап снят на PG17, восстанавливается на
  PG17 (паритет из 024); если когда-нибудь мажор разойдётся — учитывается отдельно.
- **Мало места под restore** — перед восстановлением проверяется свободное место.

### US3 — Runbook и чек-лист готовы для 027 (P1)

У оператора есть полный, уже проверенный на drill'е runbook и чек-лист быстрого
восстановления — страховка на случай неудачного cutover в 027.

#### Acceptance Scenarios

1. **Runbook задокументирован** — `.specify/specs/025-*/runbook.md` (или в
   `infra/`, см. ниже) содержит copy-paste шаги, без догадок:
   - как выбрать актуальный бэкап,
   - как остановить Postgres,
   - как восстановить (команды),
   - как проверить целостность,
   - как откатиться назад (план B).
2. **Чек-лист пройден на dry-run** — все пункты US1+US2 отмечены выполненными,
   зафиксированы время восстановления и дата прогона.
3. **Оператор готов к 027** — runbook прочитан; при сбое cutover откат
   выполняется по шагам за минуты.

## Out of Scope

- **Выбор конкретного S3-провайдера** — деталь Plan-фазы.
- **Шифрование бэкапов** — опциональная фаза 2 (одна строка в пайплайне).
- **PITR / WAL-archiving** (`wal-g`/`pgBackRest`) — будущее улучшение, не здесь.
- **Мониторинг/алертинг здоровья бэкапов** (уведомления о провале) — кандидат в
  backlog, не блокер среза.
- **Гео-репликация бэкапов на второй бокс** — позже.
- **Бэкап реальных данных приложения** — данных пока нет; они приедут в 026,
  после чего этот же механизм бэкапит уже их.
- **Формальные RTO/RPO SLA** — фиксируем ориентиры только после drill'а.

## Где живут артефакты

Runbook бэкапов — кросс-проектный и переносимый, поэтому кладётся в repo-root
`infra/` (как договорено в эпике: переносимые runbook'и отделены от
app-specific, чтобы вырезка в отдельный `infra`-репо была чистым copy-paste).
Спека/план/чек-листы самого среза — в `.specify/specs/025-*`.

## Success Criteria

- ✅ Бэкап-скрипт снимает дамп (с ролями Supabase) и заливает в off-box бакет.
- ✅ Cron daily активен; бэкапы создаются и ротируются (30 дневных + 4 недельных).
- ✅ Ошибки (бакет недоступен, нет места) логируются и дают ненулевой exit code.
- ✅ Drill «снёс → поднял» на пустом стеке успешен; стек снова healthy.
- ✅ Время восстановления замерено (ориентир < 5 мин для пустого стека).
- ✅ Путь назад (откат к отложенному состоянию) работает за < 1 мин.
- ✅ Runbook полный, copy-paste, проверен на drill'е.
- ✅ Прод и приложение на managed Supabase не затронуты.
- ✅ Оператор готов к 027 (быстрый откат при сбое cutover).

## Drill outcome (post-implementation, chat 85)

The drill on the empty stack did its job: it surfaced that the chosen **logical
`pg_dumpall` baseline is unsuitable for restoring self-hosted Supabase** —
before any real data existed (026), losing nothing.

Two problems found:
1. `postgres` is stripped of superuser in self-hosted Supabase
   (remove-superuser-access), so `pg_dumpall -U postgres` cannot read tables
   owned by `supabase_admin`/`supabase_auth_admin` (`auth.users` with password
   hashes, etc.) — protected data is silently excluded from the dump.
2. Reloading a logical dump into a freshly-initialised Supabase container
   conflicts with self-created roles/schemas/migrations (ownership errors,
   duplicate-key on `schema_migrations`) — a known Supabase issue (cli#3532),
   which points to `pg_basebackup` as the only reliable path.

**What 025 still delivered:** off-box backups + rotation + cron (US1 mechanics),
the stop→restore→healthy cycle in 18s, and the verified rollback (US2 mechanics).
The rclone/R2 pipeline, rotation, cron, and stop/restore/rollback mechanics are
all reusable.

**Decision deferred to 026:** switch the backup method to **physical** (cold
data-dir copy vs `pg_basebackup` — tradeoffs to weigh), then re-snapshot and
re-run the drill with real data. Only the dump/restore core changes; the rest of
025 carries over.
