# Feature Specification: Data & Auth Migration (эпик «Своя инфра», срез 4/5)

**Feature Branch**: `026-data-auth-migration`
**Created**: 2026-06-07
**Status**: Clarify resolved (awaiting Plan)
**Input**: Продолжение эпика «Переезд на свою инфру». 024 поднял self-hosted
Supabase и доказал паритет с продом; 025 настроил off-box бэкапы + drill, но
**вскрыл**, что выбранный логический метод бэкапа/restore для Supabase не
годится. 026 чинит метод бэкапа на надёжный, затем переносит **реальные данные
приложения + `auth.users` (хеши паролей)** с managed Supabase в self-hosted —
**параллельно проду**, без переключения трафика (cutover — это 027).

Срез эпика: 023 фундамент → 024 self-hosted Supabase → 025 бэкапы + drill →
**026 миграция данных & auth** → 027 cutover & decommission.

## Context

### Зачем этот срез

К началу 026 self-hosted Supabase живёт на боксе, доказал паритет с продом
(PG 17.6 = 17.6, расширения self-hosted ⊇ прод, все прод-схемы присутствуют) и
**пуст** — только служебные схемы Supabase, без данных приложения. Приложение и
прод-трафик по-прежнему на managed Supabase. 026 решает две связанные задачи:

1. **Починить метод бэкапа/restore self-hosted** (находка 025) — чтобы бэкап
   реально захватывал всё, включая защищённые `supabase_admin`-таблицы
   (`auth.users` с хешами), и восстанавливался без конфликтов. **Это первый шаг
   среза** — до него бэкапам реальных данных доверять нельзя.
2. **Перенести реальные данные с managed в self-hosted** — схема + данные
   `public` + `auth.users`/identities + роли + последовательности, так чтобы
   self-hosted стал верной копией прода, а существующий игрок мог
   **залогиниться против self-hosted своим текущим паролем**.

Всё — **параллельно проду**: managed не трогаем, приложение остаётся на managed,
никакого переключения env. 026 готовит цель к cutover; сам cutover — 027.

### Находка 025, которую 026 обязан закрыть

Drill на пустом стеке доказал: **логический `pg_dumpall` baseline непригоден для
self-hosted Supabase**.

1. В self-hosted Supabase у роли `postgres` снят superuser
   (remove-superuser-access) → `pg_dumpall -U postgres` **не читает таблицы,
   принадлежащие `supabase_admin`/`supabase_auth_admin`** (`auth.users` с хешами
   паролей и т.п.) — защищённые данные молча выпадают из дампа.
2. Накат логического дампа в **самоинициализирующийся** Supabase-контейнер
   конфликтует с уже созданными ролями/схемами/миграциями (ownership errors,
   duplicate-key на `schema_migrations`) — известный баг Supabase (cli#3532),
   который указывает на физический путь.

Из 025 переиспользуется без изменений: R2/rclone-пайплайн, ротация (30 дневных /
28 недельных дней), ночной cron, механика «стоп → restore → healthy → откат»
(`infra/restore.sh` доказал 18 сек на пустом стеке). **Меняется только ядро
dump/restore.**

### Два разных «переноса» — не путать

Это ключевое различие среза. «Физический бэкап» и «вытащить данные из managed» —
**разные** механизмы с разными ограничениями:

- **A. Бэкап самого self-hosted (наш бокс).** Здесь физический метод возможен и
  желателен: побайтовая копия bind-mount data-dir (`volumes/db/data`) с краткой
  остановкой `db` **или** `pg_basebackup`. Физическая копия захватывает `auth.users`
  и все роли как есть и обходит обе проблемы находки 025. Это и есть «первый шаг
  026 — сменить метод бэкапа на физический». Конкретный вариант (cold-copy с
  кратким downtime vs `pg_basebackup` без downtime) — решение Plan.
- **B. Вытащить данные из managed Supabase.** Managed — **не наш** бокс: нет
  доступа к файловой системе и (как правило) к репликации, поэтому **физическая
  копия из managed невозможна**. Перенос managed → self-hosted неизбежно
  **логический** (через инструментарий, который корректно захватывает auth и
  роли и корректно накатывается). Метод — решение Plan; в managed роль `postgres`
  сконфигурирована иначе, чем в self-hosted, поэтому читаемость `auth.users` со
  стороны managed нужно **проверить на Plan** (research), не считать данностью.

Спека требует только **исходов** (что должно оказаться в self-hosted и в каком
состоянии). Каким инструментом (Supabase CLI `db dump`, `pg_dump` с разбивкой
roles/schema/data, restore в vanilla-PG до init и т.п.) — фиксируется в Plan.

### Что переносим (scope данных)

- **`public`** — все данные приложения (ноды, рёбра, транзакции, категории,
  item_attributes, encounter-инфра, и т.д.). На сегодня объём небольшой
  (~1200 нод), так что размер/downtime самого переноса не проблема.
- **`auth`** — `auth.users` (включая bcrypt-хеши паролей), `auth.identities` и
  связанные служебные таблицы, необходимые, чтобы существующие логины работали.
- **Роли** — `anon`, `authenticated`, `service_role`, `supabase_*` (нужны для
  PostgREST/Auth/RLS). На self-hosted они уже созданы init-скриптом; перенос
  ролей решает вопрос грантов/паролей сервис-ролей — деталь Plan.
- **Последовательности** — после импорта пере-синхронизировать (`setval` по
  `max(id)`), иначе duplicate-key на следующих вставках (совет Леши).

### Что НЕ переносим

- **`storage` / `realtime` сервис-таблицы** — приложение эти продукты не
  использует (аудит 024). Схемы в self-hosted уже есть; их данные исключаем из
  переноса (подтверждено в Clarify: исключаем).
- **Расширения** — паритет 024 показал self-hosted ⊇ прод, доустанавливать
  нечего.

## Operator Scenarios & Testing

### US1 — Надёжный (физический) бэкап self-hosted; drill пере-доказан на реальных данных (P1)

Оператор заменяет негодный логический метод бэкапа на физический, и после того
как реальные данные оказались в self-hosted, заново прогоняет drill «снёс →
поднял» — теперь подтверждая, что восстанавливаются **настоящие** данные (а не
только служебные схемы пустого стека).

#### Acceptance Scenarios

1. **Физический бэкап снят** — обновлённый `backup.sh` снимает физический бэкап
   self-hosted (cold-copy data-dir с краткой остановкой `db` **или**
   `pg_basebackup`), упаковывает, заливает в off-box R2, exit code 0, в логе нет
   `ERROR`/`FATAL`.
2. **Бэкап содержит защищённые данные** — восстановленная из бэкапа БД содержит
   `auth.users` с непустыми хешами паролей (то, что логический дамп под
   `postgres` терял). Проверяется на restore, не на глаз.
3. **Drill на реальных данных проходит** — обновлённый `restore.sh` поднимает
   self-hosted из физического бэкапа; после restore `pg_isready` healthy,
   `SELECT 1` проходит, **число строк в ключевых таблицах `public` совпадает** с
   тем, что было до сноса, `auth.users` на месте, Auth/PostgREST healthy.
4. **Тайминг замерен** — зафиксировано «stop → healthy» на реальных данных
   (будущая метрика downtime для 027).
5. **Путь назад работает** — отложенное состояние возвращается за < 1 мин
   (проверено), как в 025.
6. **Ротация/cron не сломаны** — ежедневный cron активен, ротация (30/28)
   прежняя; изменилось только ядро бэкапа.

#### Edge Cases

- **Краткая остановка `db` для cold-copy** — если выбран cold-copy, остановка
  затрагивает только self-hosted (он параллельный, прод-трафика на нём нет);
  убедиться, что Auth/REST переподнимаются healthy после старта.
- **`pg_basebackup` и снятый superuser** — если выбран basebackup, проверить, что
  у используемой роли есть `REPLICATION`-право на self-hosted.
- **Битый бэкап** — restore обязан обнаружить повреждение (gunzip/checksum-guard)
  до того, как на него понадеются в 027.

### US2 — Данные `public` перенесены с managed в self-hosted (P1)

Оператор переносит схему + данные `public` с managed в self-hosted (параллельно
проду) так, что self-hosted становится верной копией прод-данных приложения.

#### Acceptance Scenarios

1. **Перенос отработал** — миграционный прогон managed → self-hosted завершается
   без `FATAL`; ownership/`schema_migrations`-конфликты находки 025 **не
   возникают** (метод Plan их обходит).
2. **Счётчики строк совпадают** — для ключевых таблиц `public` (`nodes`, `edges`,
   `transactions`, `categories`, `item_attributes`, и т.д.) число строк в
   self-hosted **равно** managed. Подсчёт делается методом, **не упирающимся в
   серверный клэмп `db_max_rows=1000`** (прямой SQL `count(*)`, не клиентский
   `.range`).
3. **Целостность ссылок** — выборочная проверка FK/связей проходит: транзакции
   ссылаются на существующие ноды, рёбра — на существующие endpoints, item_node_id
   разрешается.
4. **Последовательности синхронизированы** — после импорта `setval` по `max(id)`
   выполнен; **тестовая вставка** новой строки не падает с duplicate-key и
   получает корректный следующий id.
5. **Прод не затронут** — managed Supabase в ходе переноса доступен и неизменён
   (read-only со стороны миграции); приложение продолжает работать на managed.

#### Edge Cases

- **Накат в инициализированный стек** — если данные грузятся в уже поднятый
  Supabase, метод не должен спотыкаться о предсозданные объекты (кандидаты:
  restore в vanilla-PG до Supabase-init; либо упорядоченный roles→schema→data;
  либо загрузка только `public`/`auth` данных) — выбор Plan.
- **Повторный прогон** — миграцию можно повторить (idempotent или
  «снёс self-hosted data → залил заново»), не накапливая дубли.
- **Расхождение счётчиков** — если counts не сходятся, прогон считается
  неуспешным; диагностика до перехода к US3.

### US3 — `auth.users` с хешами перенесены; игрок логинится против self-hosted (P1)

Оператор переносит `auth.users`/identities/роли так, что **существующий игрок
входит против self-hosted своим текущим паролем** — то, что логический дамп под
`postgres` молча терял (находка 025).

#### Acceptance Scenarios

1. **`auth.users` присутствует с хешами** — в self-hosted `auth.users`
   непустая, число пользователей совпадает с managed, `encrypted_password`
   непустой у тех, у кого он непустой в managed.
2. **Identities/служебные таблицы перенесены** — связанные `auth.*` таблицы,
   нужные для входа, на месте; ссылочная целостность auth ↔ public (где есть)
   сохранена.
3. **Логин работает против self-hosted** — оператор проверяет вход существующего
   игрока против self-hosted Auth (GoTrue) **текущим паролем** (через временную
   точку доступа к self-hosted API — туннель/тестовый клиент, без публикации API
   наружу) — успех. Пароли заново не выдаются.
4. **RLS/роли работают** — `anon`/`authenticated`/`service_role` присутствуют;
   запрос под `authenticated` уважает RLS-политики так же, как на проде
   (выборочно).

#### Edge Cases

- **Читаемость `auth.users` со стороны managed** — подтвердить на Plan (research),
  что выбранный метод реально вытаскивает `auth.users` из managed (managed
  конфигурирует `postgres` иначе, чем self-hosted; не считать данностью).
- **JWT-секрет** — для валидности логинов self-hosted Auth должен использовать
  тот же `JWT_SECRET`, что ожидают перенесённые сессии/гранты (учесть в Plan;
  сами активные сессии переносить не обязательно — пере-логин допустим).
- **Дубли email** — конфликтов быть не должно (источник один); если всплывут —
  прогон неуспешен.

### US4 — Runbook миграции + чек-лист верификации готовы; managed untouched (P1)

У оператора есть проверенный, copy-paste runbook миграции и чек-лист
верификации (counts, auth-login, sequences, целостность), а прод и приложение
всё ещё на managed.

#### Acceptance Scenarios

1. **Runbook задокументирован** — шаги без догадок: смена метода бэкапа →
   снять бэкап → drill на реальных данных → перенос managed → self-hosted →
   verification → откат. Переносимая часть (бэкап/restore) — в `infra/`;
   app-specific миграция — в `.specify/specs/026-*`.
2. **Чек-лист верификации пройден** — counts совпали, sequences синхронизированы,
   логин игрока против self-hosted прошёл, целостность выборочно подтверждена;
   зафиксированы дата прогона и тайминг restore.
3. **Managed/прод не затронуты** — приложение и прод-трафик остаются на managed;
   ни одного изменения env/DNS/cutover в этом срезе.
4. **Готовность к 027** — self-hosted = верная копия прод-данных, бэкапится
   надёжно, restore доказан на реальных данных → cutover (027) можно планировать.

## Out of Scope

- **Cutover (переключение env/трафика на self-hosted)** — это 027.
- **Decommission managed Supabase** — 027 (+ грейс-период ~1–2 недели после
  cutover как revert/эталон, совет Леши).
- **Версионный бамп (0.9.0)** — инфра-only срез, бамп отложен на cutover (027).
- **Публикация self-hosted API наружу** — доступ к self-hosted для проверок
  только через туннель/тестовый клиент; публичная публикация — часть cutover.
- **PITR / WAL-archiving** (`wal-g`/`pgBackRest`) — будущее улучшение, не здесь.
- **Шифрование бэкапов** — как и в 025, опциональная фаза 2.
- **Перенос `storage`/`realtime` сервис-данных** — приложение их не использует.
- **Мониторинг/алертинг бэкапов** — кандидат в backlog.
- **Двусторонняя синхронизация managed ↔ self-hosted** — это однократный перенос
  (one-shot), не репликация; до cutover источник истины — managed.

## Где живут артефакты

Переносимая инфраструктура (физический `backup.sh`/`restore.sh`, runbook
бэкапа/restore) — в repo-root `infra/` (как в эпике: переносимые runbook'и
отделены от app-specific). App-specific миграция (скрипт managed → self-hosted,
verification-чек-лист, SQL-проверки) и спека/план — в `.specify/specs/026-*`.

## Success Criteria

- ✅ Метод бэкапа self-hosted сменён на физический; бэкап захватывает `auth.users`
  с хешами (доказано на restore).
- ✅ Drill «снёс → поднял» пройден на **реальных** данных; стек снова healthy,
  counts ключевых таблиц совпали.
- ✅ Тайминг restore на реальных данных замерен; путь назад работает за < 1 мин.
- ✅ Данные `public` перенесены с managed в self-hosted; counts совпадают
  (подсчёт мимо клэмпа `db_max_rows=1000`).
- ✅ Последовательности синхронизированы; тестовая вставка не падает duplicate-key.
- ✅ `auth.users` с хешами перенесены; существующий игрок логинится против
  self-hosted текущим паролем.
- ✅ Конфликтов находки 025 (ownership / duplicate `schema_migrations`) при накате
  нет — метод Plan их обходит.
- ✅ Прод и приложение на managed Supabase не затронуты; никакого cutover.
- ✅ Runbook миграции + чек-лист верификации полны, copy-paste, пройдены на прогоне.
- ✅ Готовность к 027 (self-hosted = верная копия прод-данных, надёжно бэкапится).

## Зависимости и предположения

- **Зависит от 024** (self-hosted поднят, паритет доказан) и **025** (R2-пайплайн,
  ротация, cron, механика restore/rollback) — переиспользуются.
- **Предполагается**, что прод-данные на managed малы (~1200 нод) → размер и
  downtime самого переноса несущественны.
- **Предполагается** PG-паритет 17.6 = 17.6 (из 024) на всём протяжении среза.

## Clarifications

### Session 2026-06-07 (chat 86)

- **Q: Объём переноса `auth.users` из managed — все пользователи или только
  активные члены кампании?**
  **A: Все пользователи.** Источник один, фильтрация рискованна; «counts
  совпадают» в US3 означает полное равенство `auth.users` managed = self-hosted.
- **Q: Сервис-данные `storage`/`realtime` — переносить или исключить?**
  **A: Исключить из переноса.** В 024 self-hosted-стек выпилил сервисы
  storage-api и realtime → принимающих сервис-таблиц нет, данным некуда лечь.
  Перенос = ноль пользы + риск конфликтов на неуправляемых схемах. Scope
  переноса = `public` + `auth` + роли + последовательности. Если Storage/Realtime
  понадобятся позже — отдельный аддитивный шаг.
- **Q: Чем доказываем перенос паролей (US3#3) — один логин или прогон по N?**
  **A: Один реальный логин.** Вход одного существующего игрока против
  self-hosted Auth текущим паролем — достаточное доказательство механики;
  массовая проверка избыточна.

### Deferred to Plan (механика, не WHAT)

- **Повторяемость миграции** — idempotent-скрипт vs «снёс self-hosted data →
  залил заново». Влияет на дизайн скрипта, не на исход → решается в Plan вместе с
  выбором инструмента переноса (Supabase CLI `db dump` vs `pg_dump` с разбивкой
  roles/schema/data) и физического метода бэкапа (cold-copy vs `pg_basebackup`).
