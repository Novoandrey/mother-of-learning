# Feature Specification: Server & PaaS Foundation (своя инфра, срез 1/5)

**Feature Branch**: `023-server-paas-foundation`
**Created**: 2026-06-02
**Status**: Draft (Specify) — awaiting Clarify
**Input**: User decision — съехать с managed Supabase на собственную
инфраструктуру ради DevOps-навыка; «хочется сразу взять сервер, на
котором будут крутиться все мои серверы». Друг-devops (Степан) помогает.
Это **срез 1/5** эпика «Переезд на свою инфру» (023 фундамент → 024
self-hosted Supabase → 025 бэкапы+drill → 026 миграция данных+auth →
027 cutover).

## Context

### Зачем весь эпик

Аудит показал, что приложение реально опирается на три продукта
Supabase — **Postgres + Auth (GoTrue) + PostgREST** — плюс RLS (76
политик на 35 таблицах, `auth.uid()` в 22 местах) и RPC/триггеры. Не
используются вовсе: Storage, Realtime, Edge Functions. Несущая и
*Supabase-специфичная* связка — Auth + PostgREST + RLS на `auth.uid()` —
поэтому «переехать на голый Postgres» = переписать аутентификацию и весь
слой запросов. Дроп-ин путь без переписывания кода — **поднять
self-hosted Supabase Docker-стек у себя**.

Но до Supabase нужен фундамент: сервер, на который умеешь катить, с
авто-SSL и понятным деплоем. Это и есть срез 1/5. Цель эпика — не только
переезд, но и **навык**: бэкапы, падения, восстановление. Поэтому слои
идут так, чтобы реальные данные пришли только после того, как отработана
дисциплина бэкапов (025) на пустом/тестовом стеке.

### Роль этого среза (1/5)

023 — чистый DevOps-onboarding, **минимальный риск**: ни реальных данных,
ни Supabase, ни прод-трафика. Только база, на которой будет жить всё
остальное:

1. Защищённый VPS (firewall, SSH-ключи, fail2ban, не-root).
2. Self-hosted PaaS-панель (**Dokploy**, выбран в chat 83) поверх Docker —
   единый «пульт» для всех будущих сервисов на одном боксе (это и есть
   «один сервер для всех моих серверов»).
3. Доказанный пайплайн: тривиальное приложение деплоится из git и
   открывается по HTTPS с валидным сертификатом, с возможностью отката.

### Делёж труда (важно)

В отличие от спек на код приложения, **исполнение происходит на твоём
сервере, не в песочнице Claude**. Поэтому:

- **Claude поставляет**: эту спеку, `plan.md`/runbook (конкретные команды,
  `compose`-файлы, конфиги reverse-proxy), скрипты, чек-листы проверки.
- **Оператор (ты + Степан) исполняет**: провижн VPS, установку PaaS,
  деплой. Claude разбирает логи/ошибки, которые ты вставляешь в чат, и
  правит артефакты.

Это часть смысла: руки на DevOps набиваются именно так.

## Operator Scenarios & Testing

### US1 — Защищённый сервер (P1)

Оператор поднимает VPS и приводит его в безопасное базовое состояние,
прежде чем что-либо на нём запускать.

#### Acceptance Scenarios

1. **Доступ только по ключу** — оператор заходит на сервер по SSH-ключу;
   парольный вход и root-логин по SSH отключены.
2. **Firewall закрыт по умолчанию** — наружу открыты только необходимые
   порты (SSH + HTTP/HTTPS); всё прочее заблокировано. Прямой доступ к
   портам сервисов (Postgres, дашборд PaaS «как есть») извне закрыт.
3. **Базовая защита от брутфорса** — fail2ban (или эквивалент) активен
   на SSH.
4. **Не-root рабочий пользователь** — повседневные операции идут под
   обычным пользователем с sudo, не из-под root.

#### Edge Cases

- **Оператор потерял ключ** — заранее настроен запасной способ доступа
  (второй ключ / консоль провайдера), чтобы не залочить себя.
- **Перезагрузка сервера** — firewall и сервисы поднимаются сами
  (enabled на старте), не требуют ручного включения.

### US2 — PaaS-панель за HTTPS (P1)

Оператор получает веб-пульт для деплоя приложений и провижна сервисов на
своём боксе — но не торчащий голым в интернет.

#### Acceptance Scenarios

1. **Дашборд доступен по HTTPS** на собственном (под)домене с валидным
   сертификатом, а не по голому IP:порт по HTTP.
2. **Доступ к дашборду закрыт** — за логином и/или ограничен по сети
   (не публичный анонимный доступ). Админка self-hosted PaaS наружу
   анонимно не торчит в принципе (хорошая практика для любой панели).
3. **PaaS управляет Docker** под капотом — оператор понимает, что каждое
   приложение/сервис = контейнер(ы) с томами; «escape hatch» в сырой
   Docker Compose доступен.
4. **Бэкапы конфигурации PaaS** — состояние самой панели (проекты,
   env-переменные) можно выгрузить/восстановить.

#### Edge Cases

- **PaaS съедает RAM** — учитываем оверхед панели (≈200–500 МБ) при
  выборе размера VPS, чтобы осталась память под Postgres-буферы (срез 024).
- **Обновление PaaS** — путь обновления известен и безопасен (учитывая
  историю CVE — держать пропатченным).

### US3 — Деплой из git по HTTPS + откат (P1)

Оператор доказывает, что пайплайн работает, на безопасной «болванке»,
прежде чем доверить ему что-то ценное.

#### Acceptance Scenarios

1. **Git-деплой** — тривиальное приложение (hello-world или staging-сборка
   `mat-ucheniya`) разворачивается из репозитория через PaaS.
2. **Авто-SSL** — приложение открывается на (под)домене по HTTPS с
   автоматически выпущенным и продлеваемым сертификатом (Let's Encrypt).
3. **Откат** — оператор может откатиться на предыдущий деплой / передеплоить
   при сломанной сборке.
4. **Логи** — оператор видит логи деплоя и рантайма из панели.

#### Edge Cases

- **Сломанная сборка** — неудачный деплой не роняет уже работающее
  приложение (atomic / blue-green по возможности).
- **DNS ещё не распространился** — понятно, как проверить до полной
  пропагации (или временный поддомен).

## Requirements

> WHAT/acceptance. Конкретные команды, compose-файлы, выбор reverse-proxy —
> в `plan.md`/runbook (фаза Plan). Названные инструменты ниже — кандидаты
> для решения в Clarify/Plan, не зафиксированный стек.

### Сервер

- **FR-001**: Доступен VPS под управлением оператора (кандидат — Hetzner
  CX33: 4 vCPU / 8 ГБ / 80 ГБ NVMe / 20 ТБ трафика; размер обоснован тем,
  что срез 024 поднимет несколько контейнеров Supabase + оверхед PaaS).
- **FR-002**: SSH — только по ключу; парольный и root-вход по SSH
  отключены; заведён не-root sudo-пользователь.
- **FR-003**: Хостовый firewall закрыт по умолчанию; открыты только
  SSH + HTTP/HTTPS; порты сервисов (БД, внутренние) наружу недоступны.
- **FR-004**: Базовая защита SSH от брутфорса (fail2ban или аналог),
  активна после перезагрузки.
- **FR-005**: Настроен запасной способ доступа на случай потери ключа
  (второй ключ / консоль провайдера).

### PaaS-платформа

- **FR-006**: На сервере установлен self-hosted PaaS (**Dokploy** —
  выбран в chat 83), дающий веб-пульт для деплоя приложений и провижна
  сервисов (Postgres/Redis/…), с реверс-прокси и авто-SSL «из коробки»,
  на одном боксе для нескольких проектов.
- **FR-007**: Дашборд PaaS доступен только по HTTPS на (под)домене и
  закрыт авторизацией/ограничением по сети — не анонимно-публичный.
- **FR-008**: Состояние PaaS (проекты, env, секреты) можно выгрузить и
  восстановить (foundation для дисциплины бэкапов, развиваемой в 025).

### Деплой-пайплайн (доказательство)

- **FR-009**: Тривиальное приложение деплоится из git-репозитория через
  PaaS и открывается по HTTPS на (под)домене с валидным авто-сертификатом.
- **FR-010**: Поддержан откат на предыдущий деплой и просмотр логов
  деплоя/рантайма из панели.
- **FR-011**: Сломанная сборка не роняет уже работающий деплой.

### Границы

- **FR-012**: Этот срез **не** поднимает Supabase, **не** переносит
  реальные данные, **не** трогает прод и существующий managed-инстанс.
  Только фундамент.

## Success Criteria

1. **SC-001**: Вход на сервер возможен только по SSH-ключу; попытка
   парольного/root-входа отклоняется.
2. **SC-002**: Сканирование портов снаружи показывает открытыми только
   SSH + 80/443; порт Postgres и порт дашборда «как есть» закрыты.
3. **SC-003**: Дашборд PaaS открывается по HTTPS на (под)домене с валидным
   сертификатом и требует авторизацию.
4. **SC-004**: Тестовое приложение задеплоено из git и отдаётся по HTTPS
   с авто-сертификатом.
5. **SC-005**: Оператор успешно откатывает тестовый деплой на предыдущую
   версию.
6. **SC-006**: После `reboot` сервера firewall, fail2ban и PaaS
   поднимаются сами; тестовое приложение снова доступно.
7. **SC-007**: Состояние PaaS выгружено в файл и проверено, что его можно
   восстановить.

## Out of Scope

- **Self-hosted Supabase** — срез 024.
- **Бэкапы реальных данных и restore-drill** — срез 025 (здесь — только
  выгрузка конфигурации PaaS, FR-008).
- **Перенос схемы/данных/`auth.users`** — срез 026.
- **Переключение прода (cutover) и гашение managed Supabase** — срез 027.
- **Cloudflare R2 / хранилище картинок** — вне всего эпика (в коде
  картинок нет; поедет с будущей фичей портретов).
- **Мобильный PWA** — spec-022 (после эпика переезда).
- **CI/CD-изыски** (preview-окружения, сложные пайплайны) — по запросу
  позже; здесь достаточно git-деплоя + отката.
