# Feature Specification: Авторизация, роли и RLS

**Feature Branch**: `006-auth-and-roles`
**Created**: 2026-04-19
**Status**: Draft
**Input**: Логины для ДМов и игроков. Роли owner/dm/player. RLS.
Подготовка фундамента под мобилку игрока (spec-007 этап 5) и IDEA-015
(игрок заявляет → ДМ подтверждает).

## Миссия

Превратить приложение из «одна публичная кампания с открытым доступом» в
«каждый пользователь видит свои кампании и играет в них с правильными
правами». ДМ контролирует, кто в его кампании и в каких ролях. Игрок
видит только то, что ему доступно. Анонимный доступ отключается.

Это необходимый фундамент для:
- **Мобилки игрока** — игроку надо знать, что он игрок, и кто его PC.
- **IDEA-015** (игрок заявляет → ДМ подтверждает) — нужны роли.
- **IDEA-009** (realtime) — нужна идентификация пользователей.
- **Спека v.Будущее** с несколькими кампаниями у одного юзера.

## User Scenarios & Testing

### User Story 1 — ДМ логинится и видит свою кампанию (Priority: P1)

ДМ открывает сайт, логинится через email. Попадает на список своих
кампаний (в MVP — одна кампания «Мать Учения»). Кликает — видит
каталог, энкаунтеры, настройки — ровно как сейчас. Всё, что он
мог делать раньше, он может делать и теперь.

**Why this priority**: Без этого не работает ничего. Это минимальный
фундамент, после которого ДМ может начать игру как раньше.

**Independent Test**: Открыть сайт → /login → ввести email → получить
magic link → кликнуть → попасть на свой каталог.

**Acceptance Scenarios**:
1. **Given** ДМ не залогинен, **When** открывает `/c/mat-ucheniya/catalog`,
   **Then** редиректится на `/login`.
2. **Given** ДМ на `/login`, **When** вводит email и жмёт «Отправить»,
   **Then** получает письмо со ссылкой.
3. **Given** ДМ кликнул по magic link, **When** переходит на сайт,
   **Then** залогинен и видит список своих кампаний.
4. **Given** ДМ залогинен и owner кампании, **When** открывает каталог,
   **Then** видит всех сущностей и может создавать/редактировать их.

---

### User Story 2 — Owner приглашает второго ДМа (Priority: P1)

Владелец кампании хочет дать права ДМа другому человеку. Заходит в
«Участники» кампании, вводит email, выбирает роль «ДМ». Приглашённый
при следующем логине автоматически видит кампанию с правами ДМа.

**Why this priority**: В текущей кампании уже несколько ДМов
(упомянуто в constitution.md). Без этой фичи они не смогут работать.

**Independent Test**: Owner заходит в `/c/[slug]/members`, добавляет
email второго ДМа. Второй ДМ логинится — видит кампанию в списке.
Открывает каталог — может редактировать.

**Acceptance Scenarios**:
1. **Given** Owner в `/members`, **When** добавляет email с ролью dm,
   **Then** создаётся запись в `campaign_members`.
2. **Given** второй ДМ логинится своим email, **When** открывает список
   кампаний, **Then** видит «Мать Учения».
3. **Given** второй ДМ внутри кампании, **When** создаёт НПС,
   **Then** получает ту же функциональность, что owner.

---

### User Story 3 — Owner добавляет игрока и привязывает его к PC (Priority: P1)

Owner хочет, чтобы игрок видел свою копию приложения. Добавляет email
игрока, роль «player». Отдельно — привязывает игрока к его ноде-PC
в каталоге (например, email «zhenya@...» → PC «Альд»).

**Why this priority**: Без привязки игрок → PC невозможно построить
мобилку игрока. Это ключ ко всему дальнейшему флоу.

**Independent Test**: Owner добавляет игрока, назначает ему PC. Игрок
логинится — видит только то, что должен видеть (см. US4).

**Acceptance Scenarios**:
1. **Given** Owner в `/members`, **When** добавляет email с ролью player,
   **Then** создаётся запись в `campaign_members`.
2. **Given** Owner на карточке PC «Альд», **When** выбирает владельца
   из списка игроков, **Then** у ноды проставляется `owner_user_id`.
3. **Given** Owner на списке игроков, **When** смотрит запись игрока,
   **Then** видит привязанный PC.

---

### User Story 4 — Игрок видит кампанию с правами чтения (Priority: P1)

Игрок логинится, видит свою кампанию. Открывает каталог — видит всех
PC, NPC, локации. Может читать описания и связи. Кнопки «Создать»,
«Редактировать» скрыты. Исключение — своя PC-нода, которую игрок
может редактировать. Настройки кампании скрыты.

**Why this priority**: Даёт игрокам доступ «посмотреть лор/НПС между
сессиями» — полезно даже без мобилки. Это новое поведение, которого
сейчас нет вообще.

**Independent Test**: Логин игроком → открыть каталог → увидеть список,
но не увидеть кнопку «+ Создать». Открыть NPC — нет «редактировать».
Открыть свой PC — есть.

**Acceptance Scenarios**:
1. **Given** игрок залогинен, **When** открывает каталог,
   **Then** видит всех сущностей без кнопок создания/редактирования.
2. **Given** игрок на своей PC-ноде, **When** смотрит,
   **Then** видит кнопку «Редактировать».
3. **Given** игрок на чужой PC-ноде или NPC, **When** смотрит,
   **Then** кнопки «Редактировать» нет.
4. **Given** игрок пытается попасть на `/c/[slug]/settings`,
   **Then** получает 403/редирект на каталог.
5. **Given** игрок пытается через API создать ноду,
   **Then** запрос отклоняется на уровне RLS.

---

### User Story 5 — Миграция существующих данных (Priority: P1)

Сейчас все данные «ничьи». После включения auth ими кто-то должен
владеть, иначе они исчезнут из-под RLS.

**Why this priority**: Блокирует всё. Если неправильно — пользователь
потеряет 150+ нод кампании.

**Independent Test**: После применения миграции + первого логина
owner'а все существующие ноды видны ему.

**Acceptance Scenarios**:
1. **Given** первый логин owner'а, **When** завершается,
   **Then** в `campaign_members` создаётся запись `owner` для
   кампании `mat-ucheniya` (привязка по email из `CAMPAIGN_OWNER_EMAIL`).
2. **Given** owner залогинен, **When** открывает каталог,
   **Then** видит все 150+ существующих нод без потерь.

---

### User Story 6 — Я вышел и забыл (Priority: P2)

Пользователь выходит из аккаунта. Его сессия прерывается. При
следующем открытии сайта снова видит `/login`.

**Independent Test**: Клик «Выйти» в хедере → редирект на `/login`.
Обновление страницы — остаётся на `/login`.

---

### Edge Cases

- **Приглашён по email, но никогда не логинился**: запись в
  `campaign_members` висит с `user_id = null`, привязывается при
  первом логине по email.
- **Игрок не привязан к PC**: видит кампанию как обычный член,
  но нигде нет «свой персонаж». Корректно показывать «у вас нет
  назначенного персонажа, обратитесь к ДМу».
- **Owner удаляет сам себя**: блокируем. В кампании всегда должен
  быть как минимум один owner. Owner МОЖЕТ передать права другому
  и потом удалить себя.
- **Игрок редактирует свой PC и удаляет себя как owner**: блокируем.
  PC либо принадлежит кому-то, либо ни у кого нет прав на редакт.
- **Magic link истёк**: стандартный Supabase-флоу — новое письмо.
- **Один юзер в нескольких кампаниях**: список кампаний на главной.
  MVP — только одна кампания, но архитектура должна это поддерживать.

## Requirements

### Functional Requirements

- **FR-001**: Система ДОЛЖНА использовать Supabase Auth для идентификации
  пользователей. Метод входа — magic link (email без пароля).
- **FR-002**: Анонимный доступ к данным ДОЛЖЕН быть отключён. Любая
  страница под `/c/[slug]/` требует авторизации.
- **FR-003**: Таблица `campaign_members` ДОЛЖНА хранить связку
  `(campaign_id, user_id или email, role)`, где role ∈ {owner, dm, player}.
- **FR-004**: Кампания ДОЛЖНА иметь ровно одного owner. Может иметь
  много dm и много player.
- **FR-005**: Ноды типа `character` ДОЛЖНЫ иметь опциональное поле
  `owner_user_id` (FK на `auth.users`), обозначающее игрока-владельца.
- **FR-006**: RLS на таблице `nodes`: SELECT — все члены кампании.
  INSERT/UPDATE/DELETE — owner и dm. Исключение — узлы типа `character`
  с установленным `owner_user_id`: их может редактировать также
  пользователь-владелец.
- **FR-007**: RLS на таблице `edges`: SELECT — все члены кампании.
  INSERT/UPDATE/DELETE — owner и dm.
- **FR-008**: RLS на таблице `encounters` и `encounter_participants`:
  SELECT — все члены кампании. Модификация — owner и dm.
- **FR-009**: RLS на таблице `campaign_members`: SELECT — все члены
  этой кампании. Модификация — только owner этой кампании.
- **FR-010**: Страница `/c/[slug]/members` ДОЛЖНА быть доступна
  только owner. Она позволяет добавить, удалить членов и менять роли.
- **FR-011**: Страница `/c/[slug]/settings` ДОЛЖНА быть доступна
  только owner и dm.
- **FR-012**: Страница-заглушка `/login` отправляет magic link.
  Страница `/auth/callback` обрабатывает возврат из письма.
- **FR-013**: Главная страница (`/`) ДОЛЖНА:
  - если не залогинен → редирект на `/login`
  - если залогинен и единственная кампания → редирект на `/c/[slug]/catalog`
  - если залогинен и несколько кампаний → показать список
- **FR-014**: Хедер приложения ДОЛЖЕН показывать email текущего
  пользователя и кнопку «Выйти».
- **FR-015**: UI ДОЛЖЕН скрывать кнопки действий, недоступных роли
  пользователя (создание ноды, редактирование, настройки, и т.д.).
  RLS остаётся последней линией защиты — UI просто не показывает
  невозможное, чтобы не вводить пользователя в заблуждение.
- **FR-016**: Миграция должна идемпотентно создавать запись `owner`
  для существующей кампании `mat-ucheniya` с email из переменной
  окружения `CAMPAIGN_OWNER_EMAIL`, которая читается при деплое.

### Key Entities

- **User (auth.users)**: Стандартный Supabase объект — id, email.
  Мы его не создаём, только ссылаемся.
- **CampaignMember**: Связка `(campaign_id, user_id, email, role)`.
  Роль определяет права. `email` нужен для инвайтов до первого логина
  приглашённого — после логина заполняется `user_id`.
- **Character ownership**: Поле `owner_user_id` на `nodes` (только
  для type = character). Указывает, какой пользователь играет этим
  персонажем. Влияет на права редактирования.

## Clarifications

### Session 2026-04-19 (начальная)

- Q: Метод входа — magic link или email+password? → **A: Magic link.**
  Причины: (1) Supabase Cloud шлёт письма из коробки без SMTP-настройки,
  (2) игроки не заморачиваются с паролями, (3) меньше безопасностных
  проблем (нет brute force, нет «забыл пароль»). При необходимости
  добавим password-флоу вторым этапом.

- Q: Owner существующей кампании — hardcode email в seed или первый
  логин? → **A: Hardcode через ENV.** Переменная `CAMPAIGN_OWNER_EMAIL`
  в Vercel, миграция создаёт запись при первом применении. Это явно
  и воспроизводимо. «Первый логин» — магично и пугает.

- Q: Видимость для игрока — читает всё или только открытое? →
  **A (дефолт): игрок-член кампании читает ВСЁ.** Это соответствует
  обычной практике настолки (партия знает НПС, места, лор после
  сессий). Приватные заметки ДМа пока не моделируются — если
  понадобятся, добавим поле `dm_notes` или тег `private`, и RLS
  скроет такие ноды от игроков. **Альтернатива (не берём)**: тонкая
  видимость через рёбра `visible_to_player` — слишком сложно для MVP,
  ДМ задолбается выдавать пермишены на каждую ноду.

- Q: Что делать со страницами после логина? → **A: редирект с `/`
  на `/c/[slug]/catalog`**, если у юзера единственная кампания.
  Список кампаний имеет смысл позже, когда будет несколько.

### Session 2026-04-19 (нужно подтвердить пользователя)

- Q: Какой email — owner существующей кампании `mat-ucheniya`? →
  **[NEEDS CLARIFICATION]** — нужен твой email, чтобы засидить.
  Его можно положить в Vercel ENV как `CAMPAIGN_OWNER_EMAIL` без

- Q: Игрок → свой PC: может редактировать HP, условия, инвентарь и

  fields, content). Это важно для автономного ведения листа персонажа

  ноды, это редактирование `encounter_participants`. **[NEEDS
  CONFIRMATION]** — согласен ли с этой моделью?

## Success Criteria

- **SC-001**: ДМ логинится через magic link за ≤ 60 секунд (от ввода
  email до /catalog).
- **SC-002**: Существующие 150+ нод кампании видны owner'у после
  миграции — 0 потерь.
- **SC-003**: Аноним, открывающий любой `/c/*` URL, редиректится на
  `/login`.
- **SC-004**: Игрок видит каталог, но не видит кнопок создания ни
  в UI, ни через API (RLS блокирует).
- **SC-005**: Второй ДМ, приглашённый owner'ом, имеет те же права,
  что и owner, кроме управления членством.
- **SC-006**: Игрок, привязанный к PC «Альд», может редактировать
  свойства и описание Альда и не может редактировать других PC.

## Assumptions & Non-Goals

**Assumptions**:
- Supabase Cloud Free Tier достаточно для MVP (ограничение ~50000 MAU).
- Кампания одна (`mat-ucheniya`). Архитектура готова к нескольким,
  но UI для создания новой кампании — out of scope этого спека.

**Non-Goals** (не делаем в этом спеке):
- Регистрация новых пользователей со своими кампаниями (self-signup).
  MVP — только для текущей кампании, новые юзеры только по инвайту.
- Password authentication как опция.
- Social login (Google, Discord, и т.п.).
- Мобилка игрока (отдельный spec-007 этап 5).
- Realtime-синхронизация между ролями (IDEA-009, отдельный spec).
- Player → DM approval flow (IDEA-015, отдельный spec).
- Приватные заметки ДМа (поле `dm_notes` или тег `private`). Добавим
  когда понадобится.
- Offline-режим. Конституция VIII — «не исключать», но и не делать.

## Incremental Roadmap

Чтобы не закопаться в большом рефакторинге и следовать принципу V
конституции («каждый релиз — играбельный инструмент»), фича бьётся
на инкременты:

1. **Инкремент 1 (US1, US5, US6)**: Supabase Auth + /login + middleware +
   миграция с owner'ом для существующей кампании. ДМ логинится,
   видит всё как раньше. Анонимного доступа больше нет. (Можно
   играть как раньше.)

2. **Инкремент 2 (US2)**: Таблица `campaign_members`, страница
   `/members`, инвайт второго ДМа. (Несколько ДМов работают.)

3. **Инкремент 3 (US3)**: Поле `owner_user_id` на нодах, привязка
   игрок→PC, добавление игроков. Игроки пока видят всё как ДМ —
   RLS ещё не ограничивает. (Добавили всех игроков, но они пока
   «не знают», что они игроки.)

4. **Инкремент 4 (US4)**: RLS-политики + условный UI (скрыты кнопки
   создания/редактирования для игроков, скрыты settings). Игрок
   видит ограниченный срез. **Это и есть конец spec-006.**

После этого spec-007 этап 5 (мобилка) строится на готовой системе
ролей.
