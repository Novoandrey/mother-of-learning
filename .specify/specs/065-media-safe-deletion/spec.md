# Feature Specification: Медиатека — безопасное удаление неиспользуемого ассета

**Feature Branch**: `codex/media-safe-deletion`
**Created**: 2026-07-20
**Status**: Implement — production verification pending
**Input**: Продолжить эпик медиатеки: MEDIA-05 — ДМ видит граф использований
ассета и удаляет только ассет, который нигде не используется.
**Epic**: `.specify/epics/media-library/constitution.md` (`MEDIA-05`)

## Context

MEDIA-01…04 сделали `media_assets` общей библиотекой и ввели первый настоящий
usage: `character_portraits.media_asset_id`. Ассет теперь может быть повторно
назначен, поэтому удаление записи из библиотеки без проверки ссылок может
сломать портреты и оставить объекты в R2 без управляемого жизненного цикла.

MEDIA-05 добавляет безопасный путь очистки: участник кампании сначала видит,
где ассет используется, и может удалить только неиспользуемый ассет. Польза
конкретна: ошибочно загруженный или больше не нужный файл перестаёт засорять
библиотеку, но повторно назначенный портрет нельзя сломать одной кнопкой.

Это не новая «система usages». Универсальное правило уже выражается обычной
ссылкой usage → `media_assets` с `ON DELETE RESTRICT`: каждый будущий
потребитель (map background, token prototype, placed token) обязан применять
то же правило. Для понятного UI сервер собирает существующие ссылки в одном
списке, но не сохраняет их второй раз в отдельной таблице.

## User Scenarios & Testing

### User Story 1 — Участник проверяет использования перед удалением (Priority: P1)

Участник кампании открывает ассет в медиатеке и видит понятный список его
активных использований: сейчас — портретные карусели с названиями нод и
ссылками на них. Если ассет используется, действие удаления недоступно и
показывает причину.

**Why this priority**: Общий ассет нельзя безопасно удалить, пока пользователь
не понимает, что именно на него ссылается.

**Independent Test**: Назначить один asset портретом, открыть его из
медиатеки и увидеть ноду-потребителя; попытка удаления не меняет ни запись,
ни варианты, ни отображение портрета после reload.

**Acceptance Scenarios**:

1. **Given** ассет назначен портретом одной или нескольких нод кампании,
   **When** участник открывает его usage-информацию, **Then** он видит каждый
   portrait usage с нодой-потребителем и не может начать удаление.
2. **Given** участник не состоит в кампании ассета, **When** он запрашивает
   usage-информацию или удаление вручную, **Then** сервер не раскрывает граф и
   не меняет данные или файлы.
3. **Given** у ассета нет реальных usages, но есть legacy
   `media_asset_node_links`, **When** участник открывает его usage-информацию,
   **Then** связанные импортом ноды показаны как контекст, но не выдаются за
   блокирующее использование.

---

### User Story 2 — Участник удаляет неиспользуемый ассет (Priority: P1)

Участник кампании открывает готовый, нигде не используемый ассет, явно
подтверждает удаление и после reload больше не видит его в библиотеке. Его
original и все derived variants больше не доступны как библиотечные объекты.

**Why this priority**: Библиотека остаётся управляемой без риска удалить
изображение, которое уже видно игрокам.

**Independent Test**: Загрузить отдельный test asset, дождаться variants,
удалить его с подтверждением, перезагрузить библиотеку и убедиться, что asset,
variant rows и соответствующие R2 objects больше не выдаются.

**Acceptance Scenarios**:

1. **Given** ассет не имеет блокирующих usages, **When** участник подтверждает
   удаление, **Then** asset исчезает из медиатеки и его original, variants и
   variant jobs очищаются без влияния на другие ассеты кампании.
2. **Given** usage создан между открытием диалога и подтверждением,
   **When** запрос удаления достигает сервера, **Then** сервер отказывает,
   сохраняет asset и сообщает, что нужно обновить usage-граф.
3. **Given** очистка R2 временно не завершилась, **When** пользователь
   перезагружает страницу, **Then** ассет не становится ложно «удалённым»;
   состояние и безопасный повтор/операторское восстановление определены
   планом без удаления работающего usage.

## Edge Cases

- Один asset назначен нескольким портретам или нескольким нодам: каждая
  блокирующая ссылка показана, но дубли одной ноды не создаются.
- Legacy portrait без `media_asset_id` не считается ссылкой на library asset:
  MEDIA-04 уже запрещает ему читать original в `/tg`.
- `media_asset_node_links` — provenance/discovery relation, а не usage; при
  удалении неиспользуемого asset его каскадная очистка не должна скрыть
  существующий portrait usage.
- Pending, processing и failed variants не позволяют обойти usage-проверку.
- Повторное подтверждение и параллельные запросы не должны удалить другой
  asset, оставить частично удалённую библиотечную запись или удалить asset,
  получивший usage в гонке.
- Результат удаления не должен раскрывать storage keys, original URL или
  чужие ноды.

## Requirements

### Functional Requirements

- **FR-001**: Система MUST предоставить участнику кампании asset-level usage
  graph, который различает блокирующие usages и неблокирующие связи/provenance.
- **FR-002**: Для текущей модели `character_portraits.media_asset_id` MUST
  быть блокирующим usage; UI MUST показать ноду-потребителя и дать перейти к
  ней пользователю с уже разрешённым доступом.
- **FR-003**: `media_asset_node_links` MUST быть показан отдельно как
  import/discovery context и MUST NOT сам по себе запрещать удаление: migration
  141 определяет его именно не как future media use.
- **FR-004**: Delete action MUST повторно проверить membership кампании и
  блокирующие usages на сервере; клиентский граф/disabled button не является
  авторизацией или защитой от гонки. Database-level FK `ON DELETE RESTRICT`
  MUST остаться последней защитой: при гонке создание usage и delete оставляют
  либо asset с usage, либо удалённый asset и контролируемый отказ создания
  usage.
- **FR-005**: Система MUST удалить только выбранный неиспользуемый asset своей
  кампании вместе с его variant metadata, jobs и R2 original/derived objects;
  она MUST NOT затронуть чужую кампанию или другой asset.
- **FR-006**: План MUST удалять library metadata только после того, как
  database-level usage guard разрешил delete, и затем адресно очищать R2
  original/derived objects. Сбой R2-cleanup MUST быть записан существующим
  activity/error log для операторской очистки, но MUST NOT оставлять рабочий
  portrait без asset или требовать новую очередь/таблицу состояний.
- **FR-007**: Каждый будущий usage MUST ссылаться на `media_assets` обычным FK
  с `ON DELETE RESTRICT` и добавлять свой read-only resolver в единый
  server-side список использования. Новая generic `asset_usages` table, event
  stream или дублирование ссылок MUST NOT создаваться. MEDIA-05 не создаёт
  будущие usage-типы.
- **FR-008**: Успешное удаление и отказ из-за usage MUST переживать reload;
  `/tg` и map reads MUST продолжать использовать только safe rendition
  contract и не делать fallback на original.
- **FR-009**: MEDIA-05 MUST NOT добавлять массовое удаление, retention/quota,
  metadata/tags/search, visibility policy, cropper или редактор токена.

### Key Entities

- **Blocking media usage**: существующая доменная ссылка на asset, из-за
  которой удаление запрещено; в MEDIA-05 это portrait usage.
- **Usage graph entry**: безопасное для участника кампании представление
  ссылки с типом, нодой-потребителем и переходом, без storage key.
- **Provenance link**: `media_asset_node_links`, импортная/поисковая связь с
  нодой, которая не владеет файлом и не является блокирующим usage.
- **Deletion cleanup**: одноразовое удаление library metadata и R2 original
  плюс variants только после отсутствия blocking usages; orphaned R2 object
  при сетевом сбое фиксируется в существующем error log, а не новой моделью.

## Success Criteria

- **SC-001**: Участник за один переход видит все portrait-потребители
  использованного ассета и не может удалить его.
- **SC-002**: Отдельно загруженный неиспользуемый asset исчезает из библиотеки
  после подтверждения и reload; ни один другой asset не изменяется.
- **SC-003**: Серверная гонка «создать usage одновременно с delete» оставляет
  asset и новый usage целыми либо завершает delete до создания usage так, что
  создание получает контролируемый отказ; частичной ссылки не остаётся.
- **SC-004**: Участник другой кампании не может прочитать usage-граф, удалить
  asset или получить original storage key/URL через этот путь.

## Assumptions

- До отдельного Visibility-эпика все участники своей кампании могут
  просматривать граф и очищать неиспользуемые assets, как и загружать их.
- MEDIA-04 production migration 142 с `ON DELETE RESTRICT` остаётся базовым
  database-level предохранителем для portrait usage.
- R2 API и существующий MEDIA-01 worker позволяют адресно удалить original и
  derived objects; точная recovery-механика будет выбрана в Plan.
- Нынешние 31 несопоставленных legacy portrait rows не принадлежат library
  asset и не меняются этой спекой.

## Clarifications

- **OQ-001 — proposed default**: `media_asset_node_links` отображается как
  неблокирующий импортный контекст. Это соответствует комментарию migration
  141 и не превращает discovery link в скрытое владение файлом.
- **OQ-002 — proposed default**: удалить неиспользуемый asset может любой
  участник кампании; отдельная роль ДМа не вводится, согласно M4/R3 эпика.
- **OQ-003 — RESOLVED (2026-07-20)**: не добавлять durable cleanup state/job.
  Сначала server-side delete проходит существующий FK guard в БД, затем
  best-effort удаляются перечисленные R2 objects. Неуспех R2 фиксируется в
  существующем activity/error log как orphan cleanup для оператора; это не
  ломает usage и не превращается в отдельную подсистему.
