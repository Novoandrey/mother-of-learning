**Input v2 (chat 91)**: «пора делать мобильную версию на основе того, как

## Context

Конституция v3 («Два режима») обещает игроку mobile-first «книгу-помощник».
Сегодня приложение — desktop-chrome (11 табов + сайдбар), на телефоне это
«desktop, ужатый в 380px». v1 этой спеки предлагала перенести существующие
поверхности как есть (читалка). Данные chat 90 (live MCP) показали, что
переносить читалкой нечего:

- 20 из 24 игроков привязаны к PC — аудитория есть и онбордилась;
- **0 из 31 PC-листа правились за 30 дней** — чарники живут не у нас;
- транзакций 2, летописей 2 — серия 009–019 игроками не используется;
- чарники реально живут в **Google-таблице Стасяна** (листы Маркус,
  Никандр, Бальтазар, Дамбиниус, …) — системе с автоформулами,
  задокументированной в `.specify/memory/character-sheet-excel-system.md`
  (модификаторы, БМ, навыки ×0/0.5/1/2, спасы, ресурсы; «реактивный граф
  зависимостей»; связка с IDEA-017 «конструктор слоями»).

Мосты уже существуют: миграции 112–113 импортировали PC/НПС из этой самой
таблицы (у ноды «Маркус Грейсон» есть `fields.stats`), node_type **`spell`
заведён пустым** под будущие заклинания, факультативы — уже 64 ноды.

**Пивот v2**: режим игрока строится не как читалка, а как **инструмент,
которым пользуются на каждой сессии** — живой чарлист по системе Стасяна
с бросками кубов из листа. Это даёт игроку причину открывать приложение
еженедельно и переносит source of truth чарников в продукт.

## Эпик и фазировка (зафиксированное решение скоупа)

Запрос v2 «сначала спарсить базу» сознательно **инвертирован**:

| Фаза | Что | Спека |
|---|---|---|
| **1** | Живой чарлист + кубы + импорт листов из таблицы | **эта (022)** |
| 2 | Компендиум: спеллы / классы+подклассы / фиты / бэкграунды / расы (парсер dnd.su по образцу spec-018; факультативы уже в БД; пустой тип `spell` — слот) | отдельная, слот при взятии |
| 3 | Билдер персонажа + level-up («конструктор слоями», IDEA-017, event sourcing per конституция V) | отдельная |

Почему так: (а) данные листов уже существуют — импорт даёт игрокам ценность
без месяцев парсинга справочников; (б) кубам нужен лист, а не база
заклинаний; (в) урок серии 009–019: ценность без поверхности потребления =
склад; (г) билдеру нужны и компендиум, и живой лист — он последний по
зависимостям. Листы полны хоумбрю («Ильза» как источник заклинаний,
ГМ-выданные умения, «Лечение ран 24») — значит, лист обязан жить и без
компендиума: свободный текст + источник, линковка на ноды появится в Фазе 2
без миграции данных.

**MVP-срез Фазы 1** (Clarify Q6): US1 + US2 + US3; US6 — первый fast-follow
сразу за срезом; US4/US5 — P2 после него.

## User Scenarios & Testing

### US1 — «Живой чарлист» (P1, ядро)

Игрок открывает PWA с домашнего экрана и попадает на **свой** лист
персонажа — структуру и автоматику системы Стасяна, вертикальный скролл,
тач-таргеты ≥ 44px, без desktop-сайдбара и DM-табов.

#### Acceptance Scenarios

1. **Открытие**: standalone PWA → один PC у игрока — сразу его лист;
   несколько — компактный список, лист за ≤ 2 тапа. (наследует v1)
2. **Блоки листа**: шапка (имя, класс, раса, уровень, предыстория);
   6 характеристик (значение → авто-модификатор → авто-спасбросок);
   18 навыков (мод + БМ × множитель 0/0.5/1/2); боевой блок (КД,
   инициатива, скорость, попадание, спас-DC); HP (макс/текущие/временные);
   кость хитов; ресурсы класса (ци / ячейки по кругам / вдохновение /
   канал — «всего/осталось»); умения с источником; заклинания по кругам
   с источником; снаряжение; владения.
3. **Ввод vs формула** (принцип из memory-doc): игрок вводит статы,
   уровень, множители мастерства, текущие HP, остатки ресурсов; всё
   производное считается мгновенно (БМ = ⌊(ур−1)/4⌋+2; мод = ⌊(стат−10)/2⌋;
   навык = мод + ⌊БМ×множ⌋; спас = мод [+БМ]). Классозависимые производные
   (КД, попадание, иниц, макс HP) до Фазы 3 — **формула по умолчанию +
   ручной override**.
4. **Текущие HP и ресурсы** правятся тапом (±N / −1 ячейка) без формы;
   изменение видно сразу и сохраняется.
5. **Права**: править лист может владелец PC и DM (текущая модель
   `canEditNode` / `isPcOwner`); остальные участники — read.
6. **Деньги на листе**: в Фазе 1 блок валюты — **плейсхолдер** без данных
   (Clarify Q4); связка с ledger подключается позже, вместе с US4.

#### Edge Cases
- 0 PC у игрока → empty-state «Напиши ДМ» (v1). Несколько кампаний →
  текущая по cookie, смена — явный жест (v1).
- PC существует, но структурного листа нет (легаси-нода) → лист в режиме
  «черновик»: пустые блоки + кнопка «Импортировать из таблицы» (US3) или
  ручное заполнение.
- Прямой переход на DM-страницу с телефона → не падаем в сломанный
  desktop-лейаут (деградация/редирект — вопрос Plan, унаследован из v1).

### US2 — «Кубы из листа» (P1)

1. **Проверка одним тапом**: тап по навыку / спасброску / характеристике /
   попаданию → бросок d20 + соответствующий модификатор; результат крупно,
   с разбивкой (🎲 14 + 7 = **21**).
2. **Преимущество/помеха**: переключатель adv/dis рядом с результатом —
   перебрасывает парой d20, показывает оба.
3. **Дайс-пад**: произвольные XdY±Z (d4–d100) для урона и прочего —
   доступен с листа в один тап.
4. **История**: последние ~20 бросков локально на устройстве (имя
   проверки, разбивка, время); переживает перезагрузку страницы.
5. **Офлайн**: бросок работает без сети (client-only вычисление).

Edge: шаринг броска другим (в сессию/ДМу/энкаунтер) — **вне скоупа**,
осознанный мост к реворку энкаунтеров (032) / US7.

### US3 — «Импорт листа из таблицы» (P1)

1. Оператор загружает xlsx (выгрузка Google-таблицы), выбирает вкладку-PC →
   парсер маппит канонические области из memory-doc → создаёт/обновляет
   структурный лист ноды PC.
2. **Отчёт распознавания** обязателен: ✅ поле взято / ⚠️ взято с
   неуверенностью / ❌ не распознано; нераспознанное падает в черновые
   свободные поля, ничего не теряется молча. (Формат живой и дрейфует:
   лист «Никандр» уже отличается колонками от канона.)
3. Повторный импорт существующего листа → **overwrite с подтверждением**;
   если лист правился после последнего импорта — явное предупреждение
   перед перезаписью (Clarify Q1b).
4. Импорт запускает любой, кто вправе править ноду PC — владелец и DM
   (существующий `canEditNode`, новых прав не вводим; Clarify Q1a).
5. Поле золота листа **не импортируется** (Clarify Q4); в отчёте
   помечается «пропущено по решению», а не ❌.

Edge: хоумбрю-источники («Ильза», «ГМ дал») сохраняются свободным текстом
с полем «источник» как есть; пометки версии правил («…24») — часть
названия, без отдельного поля (Clarify Q7).

### US4 — «Мои деньги» (P2; v1-US2 сжат)

Баланс текущей петли + последние транзакции + мобильная форма транзакции
с существующим approval flow (spec-014), shortfall-prompt — как в v1.

### US5 — «Предметы и общак» (P2; v1-US3 сжат)

Инвентарь PC, карточка предмета read-вью, общак put/take — как в v1.

### US6 — «Лента кампании» (P2, новый; первый fast-follow за MVP-срезом)

Вкладка home: последние сессии-рекапы (единственный регулярно пополняемый
контент: 72 ноды, импорт идёт), тап → мобильное read-вью ноды сессии.

### US7 — «Живой энкаунтер» (P3; v1-US4 без изменений)

Вертикальный список инициативы, своё HP/условия, действия своего хода по
правам; вопрос realtime — на Plan; кандидат на вынос в отдельную спеку.
Кубы US2 — будущий мост сюда.

## Functional Requirements

**Shell/PWA**
- FR-001 PWA: manifest, иконка, standalone, установка на домашний экран.
- FR-002 Режим игрока — отдельная mobile-first поверхность поверх тех же
  данных/auth/actions (конституция: «линза, а не дублирование»); desktop
  DM-режим не меняется. Вход — выделенный роут (`/m/...`), manifest ведёт
  туда; UA-автоопределения нет (Clarify Q5).
- FR-003 Игрок не видит DM-инструментов; видимость данных = текущая
  RLS-модель; правки гейтятся как сейчас (auth-гейт каждого action).

**Чарлист**
- FR-004 Структурный лист хранится на PC-ноде (модель полей — Plan) и
  правится через server actions с `canEditNode`/`isPcOwner`; при записи
  применяется правило `cleanFields` (неизвестные ключи не уничтожаются)
  и сайдбар-инвалидация.
- FR-005 Derived-граф формул из memory-doc: БМ, модификаторы, 18 навыков
  с множителями {0, 0.5, 1, 2}, спасброски, спас-DC; пересчёт мгновенный.
- FR-006 Классозависимые производные (КД, попадание, инициатива, макс HP)
  имеют формулу по умолчанию и ручной override с пометкой «вручную».
- FR-007 Ресурсы (HP тек/врем, ци, ячейки по кругам, вдохновение, канал):
  модель «всего/осталось», инкременты тапом. Кнопки отдыха — в Фазе 1
  (Clarify Q2): у ресурса флаг восстановления {short, long}; short rest
  восстанавливает short-ресурсы, long rest — все ресурсы, HP до максимума
  и половину костей хитов (мин. 1).
- FR-008 Умения и заклинания: название + источник (свободный текст в
  Фазе 1); заклинания группируются по кругам; снаряжение — список строк.
- FR-009 Все новые ключи полей получают русскую локализацию в
  `node-form-constants.ts` одновременно с введением (правило проекта).
- FR-010 Блок валюты в Фазе 1 — плейсхолдер без данных; связка с
  ledger-балансом подключается позже, вместе с US4 (Clarify Q4).

**Кубы**
- FR-011 Бросок d20+мод от любого кликабельного модификатора листа;
  adv/dis; разбивка результата.
- FR-012 Дайс-пад XdY±Z, d4–d100; криты (нат 20/1) визуально выделяются.
- FR-013 Локальная история ~20 бросков (device-local storage), офлайн-
  работоспособность бросков.

**Импорт**
- FR-014 Парсер xlsx канонических областей листа Стасяна; повторный прогон —
  overwrite с подтверждением и предупреждением, если лист правился после
  последнего импорта (Clarify Q1b).
- FR-015 Отчёт распознавания ✅/⚠️/❌ по каждому полю; ❌ → черновые
  свободные поля, без молчаливых потерь.
- FR-016 Импорт «Маркус», «Бальтазар», «Дамбиниус» (канон) — ≥90% полей
  автоматически; «Никандр» (дрейф) — без падения, с честным отчётом.

**Прочее**
- FR-017 Деградация DM-страниц на мобиле: ссылка не роняет игрока в
  непригодный лейаут — баннер-переход в мобильный режим (точная механика —
  Plan; Clarify Q5).
- FR-018 Перф: лист открывается и считает формулы без заметной задержки
  на среднем телефоне (бюджет — Plan).
- FR-019 Учёт открытий PWA: минимальный privacy-лёгкий счётчик —
  (пользователь, дата) уникально, без device-данных; кормит SC-5
  (Clarify Q3).

## Success Criteria

- SC-1 Лист любого из 4 эталонных PC открывается на телефоне и совпадает
  с таблицей по всем derived-числам (мод/навыки/спасы/БМ).
- SC-2 Бросок проверки — ≤ 2 тапа от открытого листа.
- SC-3 Импорт: ≥90% полей канон-листов автоматически, отчёт без ❌-потерь.
- SC-4 Через 30 дней после релиза `pcs_touched_30d` (метрика chat 90) > 0,
  цель ≥ 15 из 31 — листы живут в приложении.
- SC-5 ≥ 10 из 20 игроков открыли PWA в первые 2 недели (счётчик — FR-019).

## Out of scope (Фаза 1)

Компендиум-парсинг и линковка на ноды (Фаза 2); билдер/level-up (Фаза 3);
realtime; push-уведомления; портреты (spec-030); шаринг бросков; экспорт
обратно в xlsx; правка чужих листов игроком.

## Clarifications

### Session 2026-06-10 (chat 92)

- **Q1a — кто запускает импорт**: любой, кто вправе править ноду PC —
  владелец и DM (существующий `canEditNode`); новых прав не вводим.

- **Q3 — счётчик открытий PWA**: да, минимальный — (пользователь, дата)

- **Q4 — золото**: в Фазе 1 полностью мимо — импорт игнорирует поле,
  блок валюты на листе остаётся плейсхолдером; связку с ledger делаем

- **Q5 — точка входа**: отдельный роут (`/m/...`), manifest туда;

- **Q7 — пометки версий правил («…24»)**: часть названия в Фазе 1;
