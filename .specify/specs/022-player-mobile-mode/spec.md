# Feature Specification: Player Mobile Mode (mobile-first PWA для режима игрока)

**Feature Branch**: `022-player-mobile-mode`
**Created**: 2026-06-02
**Status**: Specify draft — awaiting Clarify
**Input**: User prompt — «пора делать полезность для игроков: переносить
фичи денег, предметов, а также чарники и трекер энкаунтера на мобилку.
По сути переносить то, что уже есть. Сделать отдельной веб-приложкой
(PWA), чтобы не возиться с App Store / Google Play.»

## Context

Конституция v3.0.0 («Два режима») с самого начала описывает **два
первоклассных пользователя поверх одних данных**:

- **ДМ** — рабочий стол: граф, каталог, трекер, подготовка. **Desktop-only.**
- **Игрок** — «книга-помощник», читалка, не дашборд: «Mobile-first.
  Скролл, текст, минимум кнопок. Информация потребляется, а не
  конфигурируется… Игроку достаточно открыть телефон и увидеть своего
  персонажа.»

На сегодня реализована только половина: всё приложение — это desktop-chrome.
`app/c/[slug]/layout.tsx` строит `h-screen` лейаут с горизонтальной
панелью из 11 nav-табов и постоянным левым сайдбаром каталога. На телефоне
это: переполненная панель табов, всегда занятый экран сайдбаром, таблицы
энкаунтера и ledger'а, рассчитанные на широкий экран. Игрок физически
может открыть это в мобильном браузере, но это «desktop, ужатый в 380px»,
а не режим игрока, который обещает конституция.

Спека закрывает эту дыру: **впервые материализует режим игрока** как
отдельную mobile-first поверхность, устанавливаемую на домашний экран
(PWA), без App Store / Google Play. Принцип IV (данные-первичны) делает
это **линзой, а не дублированием**: те же ноды, та же авторизация, тот же
Supabase, те же server actions — другой shell и другая компоновка.

### Что переносим (по словам пользователя, «как есть»)

Четыре существующих, уже работающих на десктопе поверхности:

1. **Деньги** — кошелёк PC + лента транзакций + создание транзакции
   (через существующий approval flow spec-014).
2. **Предметы** — инвентарь PC + общак (складчина): просмотр + put/take.
3. **Чарник** — страница PC (`/catalog/<pcId>`, `<NodeDetail>`): статы,
   поля, связи.
4. **Трекер энкаунтера** — инициатива / HP / условия / лог; для игрока
   read-mostly + его собственные действия (если ДМ разрешает).

Правило проекта (AGENTS.md, NEXT.md): **«сначала перенеси как есть, потом
улучшай»**. Эта спека — про паритет, а не про новые фичи. Новые
возможности (портреты персонажей, push-уведомления, оффлайн-правки)
явно вынесены в Out of Scope.

### Это эпик, 022 — первый срез

NEXT.md уже помечает мобилку как «**character-sheet/mobile epic** —
большая, серия спек». Поэтому 022 сознательно режется на user stories
с независимыми чекпойнтами:

- **US1–US3 + shell (PWA) = MVP** — режим игрока на чтение/запись для
  персонажа, денег и предметов. Это самодостаточная, демонстрируемая
  ценность.
- **US4 (живой энкаунтер) = P2** — тяжелее остальных, требует решения
  про live-обновления (сейчас энкаунтер **не realtime**). Может вырасти
  в собственную спеку (предположительно 023). Это явная, не «тихая»
  декомпозиция: если US4 окажется крупнее ожидаемого, выносим его, и
  это фиксируется здесь, а не молча выпиливается.

Слот 022 свободен (021 зарезервирован под Wiki/Markdown editor, 020 PC
Holdings остаётся в очереди). Очерёдность этих трёх — за пользователем;
эта спека не переставляет роадмап, только занимает свободный номер.

### Связь с инфраструктурой (вне scope этой спеки)

Пользователь параллельно поднял вопрос «Supabase vs свой сервер» и
хранение нескольких ГБ картинок. Это **ортогонально** мобильной фиче:
режим игрока работает на любом бэкенде. Решение по инфраструктуре и по
объектному хранилищу для изображений принимается отдельно (см. чат-аналитику
и будущую infra-задачу), не в этой спеке. Портреты персонажей зависят от
этого решения и поэтому здесь — в Out of Scope как естественный fast-follow.

## User Scenarios & Testing

### US1 — «Мой персонаж» (P1, ядро режима игрока)

Игрок устанавливает приложение на домашний экран, открывает одним тапом
и **сразу видит своего персонажа** — без выбора кампании, без сайдбара,
без табов ДМ.

#### Acceptance Scenarios

1. **Игрок открывает PWA с домашнего экрана** — приложение открывается
   в standalone-режиме (без адресной строки браузера), показывает home
   режима игрока: карточка(и) его персонажа(ей) с именем, ключевыми
   статами, текущим HP, балансом и кратким «что у меня есть». Никакого
   desktop-сайдбара, никакой панели из 11 табов.

2. **У игрока один персонаж** — home сразу открывает экран этого
   персонажа (чарник), без промежуточного выбора.

3. **У игрока несколько персонажей** — home показывает компактный список
   его PC; тап открывает конкретный чарник. Переключение между «своими»
   персонажами — за ≤ 2 тапа.

4. **Игрок листает чарник** — статы, поля ноды (как на десктопной
   `<NodeDetail>`), связи (рёбра) — всё читается вертикальным скроллом,
   крупным шрифтом, тач-таргеты ≥ 44px. Связанные ноды кликабельны и
   ведут на их мобильные read-вью.

5. **Игрок видит только то, что ему положено** — игрок не видит DM-only
   инструментов (каталог целиком, петли-редактор, настройки кампании).
   Видимость данных = текущая RLS-модель кампании (кооператив: в основном
   всё видно всем участникам), правки гейтятся как сейчас.

#### Edge Cases

- **У игрока 0 персонажей** (он в кампании, но PC ещё не назначен) —
  empty state «За тобой пока не закреплён персонаж. Напиши ДМ.» вместо
  пустого экрана или ошибки.
- **Игрок состоит в нескольких кампаниях** — режим игрока показывает
  персонажей текущей кампании; смена кампании — явный, но не первичный
  жест (cookie `current_campaign_slug` уже существует).
- **Игрок открыл прямую ссылку на DM-страницу на телефоне** — он не
  выпадает в сломанный desktop-лейаут; см. FR про деградацию / редирект
  (Open question for Plan).

### US2 — «Мои деньги» (P1)

Игрок проверяет баланс и историю и записывает транзакцию с телефона,
подчиняясь существующему approval flow.

#### Acceptance Scenarios

1. **Проверка баланса** — на чарнике (или отдельной вкладке «Деньги»)
   игрок видит баланс в текущей петле (монеты по номиналам, gp-эквивалент)
   и последние ~10 транзакций компактными строками.

2. **Запись расхода/дохода** — игрок открывает форму транзакции,
   оптимизированную под телефон (крупные поля, числовой ввод, минимум
   разворачивающихся блоков), отправляет. Если для роли игрока действует
   approval (spec-014) — транзакция уходит в pending, игрок видит её
   статус.

3. **Перевод другому PC / в общак** — доступно с телефона тем же путём,
   что на десктопе (`createTransfer` / `createItemTransfer`), с тем же
   гардом владения предметом (chat 43 fix).

4. **Статусы заявок** — pending / approved / rejected рендерятся ясно
   (как на десктопе), включая комментарий отклонения.

#### Edge Cases

- **Нет активной петли** — empty state (балансы wipe-аются по петлям,
  FR-015 spec-010), как на десктопе.
- **Shortfall при расходе** — мобильный аналог shortfall-prompt
  (rich/poor/empty), а не тихая ошибка.

### US3 — «Мои предметы и общак» (P1)

Игрок просматривает инвентарь, открывает карточку предмета, кладёт/берёт
из общака.

#### Acceptance Scenarios

1. **Инвентарь персонажа** — сетка/список предметов с количеством,
   читаемый на узком экране; тап по предмету — детали.
2. **Общак (складчина)** — игрок видит баланс и инвентарь общака,
   кладёт/берёт одним-двумя тапами (`<StashButtons>`-эквивалент).
3. **Каталог предметов кампании read-вью** — если игрок переходит к
   ноде предмета по ссылке, она открывается в мобильном read-вью (не
   в desktop edit-cells таблице).

#### Edge Cases

- **Предмет без `item_node_id`** (free-text) отображается корректно
  (текущая модель допускает это).
- **Попытка положить в общак предмет, которого нет** — тот же понятный
  отказ, что на десктопе.

### US4 — «Живой энкаунтер» (P2, кандидат на вынос в отдельную спеку)

Во время боя игрок открывает энкаунтер на телефоне и следит за порядком
инициативы, своим HP и условиями; совершает действия своего хода, если
ДМ разрешает.

#### Acceptance Scenarios

1. **Игрок открывает текущий энкаунтер** — видит список участников в
   порядке инициативы, чей сейчас ход, своё HP и условия. Компоновка —
   вертикальный список карточек участников, а не широкая таблица-грид.

2. **Игрок следит за боем** — когда ДМ меняет HP/условия/ход, экран
   игрока **отражает изменения** (live или по явному обновлению — см.
   Open question про realtime). На сегодня десктоп-трекер не realtime;
   для мобильного «следить за боем» это значимое решение Plan-фазы.

3. **Игрок совершает своё действие** (если роль это позволяет) —
   доступны его действия/атаки; write гейтится теми же правилами, что
   на десктопе (BUG-018: игрок не пишет в чужие строки, ранний exit с
   одним alert'ом).

4. **Read-only наблюдатель** — игрок, не участвующий в этом бою, может
   смотреть, но не править.

#### Edge Cases

- **Нет активного энкаунтера** — empty state «Сейчас боя нет.»
- **Энкаунтер с большим числом участников** — список скроллится; «чей
  ход» закреплён/выделен, чтобы не теряться.
- **Игрок офлайн посреди боя** — поведение определяется решением про
  live-обновления; как минимум — не показывать заведомо устаревшее
  состояние как актуальное без индикации.

### US5 — Устанавливаемый PWA-shell (P1, сквозной)

Игрок добавляет приложение на домашний экран и пользуется им как нативным,
без магазина приложений.

#### Acceptance Scenarios

1. **Android (Chrome)** — браузер предлагает / позволяет установить
   приложение; иконка появляется на домашнем экране; запуск открывает
   standalone-режим с иконкой и сплэшем.

2. **iOS (Safari)** — игрок добавляет через Share → «На экран Домой»;
   приложение получает иконку и открывается в standalone. Поскольку iOS
   не показывает авто-промпт установки, приложение даёт **ненавязчивую
   подсказку** «как установить» (одноразовую, скрываемую).

3. **Уже установлено** — повторный запуск с домашнего экрана не
   показывает подсказку об установке.

4. **Обновления доставляются сами** — новый деплой подхватывается без
   действий пользователя и без прохождения ревью магазина (это и есть
   причина выбора PWA вместо нативной обёртки).

#### Edge Cases

- **Десктоп-браузер открывает приложение** — режим игрока не ломает
  существующий desktop-UX ДМ; см. Open question про маршрутизацию и
  определение мобильного контекста.
- **iOS в ЕС** — на 2026 home-screen web apps и web-push в ЕС
  восстановлены (Apple отменила ограничение iOS 17.4 после реакции
  сообщества); установка на домашний экран работает.

## Requirements

> Примечание: ниже — WHAT/WHY. Конкретные технические решения (структура
> маршрутов, realtime vs polling, манифест/SW, объектное хранилище) —
> в Plan-фазе. Ссылки на **существующие** компоненты/queries даны как
> карта переиспользования, не как новые архитектурные решения.

### Player shell & navigation

- **FR-001**: Существует отдельная mobile-first поверхность режима
  игрока — не desktop-chrome, ужатый по ширине. У неё **нет** левого
  сайдбара каталога и **нет** горизонтальной панели из 11 nav-табов.
- **FR-002**: Навигация режима игрока ограничена релевантными игроку
  разделами: «Мой персонаж», «Деньги», «Предметы/Общак», «Энкаунтер»
  (когда есть активный). DM-only разделы (полный каталог, петли-редактор,
  сессии-редактор, факультативы-админка, участники-админка, настройки)
  в навигацию игрока **не входят**.
- **FR-003**: Точка входа режима игрока ведёт игрока **прямо к его
  персонажу** (US1), без обязательного выбора кампании при единственной
  активной кампании.
- **FR-004**: Все интерактивные элементы режима игрока пригодны для
  тача (целевые зоны ≥ ~44px), читаемы без зума, работают вертикальным
  скроллом. Никакой горизонтальной прокрутки таблиц для базовых сценариев.
- **FR-005**: Видимость данных и право на правку = действующая RLS-модель
  кампании и существующие auth-гарды (`getMembership`, `isPcOwner`,
  `canEditNode`). Режим игрока **не вводит новых правил доступа** —
  только новую компоновку поверх тех же прав. Любой server action,
  вызываемый из мобильной поверхности, проходит тот же auth-контракт
  (AGENTS.md).

### My Character (US1)

- **FR-006**: Игрок видит персонажа(ей), закреплённого(ых) за ним через
  владение PC (`node_pc_owners` / `isPcOwner`). Поддержан случай 0, 1 и
  N персонажей (FR-003, edge cases US1).
- **FR-007**: Чарник на мобильном показывает то же содержимое, что
  desktop `<NodeDetail>` для PC-ноды: статы/поля и связи (рёбра),
  кликабельные на read-вью связанных нод. Паритет содержимого, мобильная
  компоновка.
- **FR-008**: С чарника доступны блоки «Деньги» (US2) и «Предметы» (US3)
  — как секции экрана и/или вкладки (UI-decision на Plan).

### My Money (US2)

- **FR-009**: Игрок видит баланс своего PC в текущей петле (монеты по
  номиналам + gp-эквивалент) и последние ~10 транзакций компактными
  строками-рядами (паритет с `<TransactionRow>`).
- **FR-010**: Игрок может создать транзакцию (доход/расход/перевод) с
  телефона через существующие server actions
  (`createTransaction`/`createTransfer`/`createItemTransfer`),
  подчиняясь approval flow (spec-014): pending → виден статус → approved/
  rejected с комментарием.
- **FR-011**: Форма транзакции на мобильном оптимизирована под телефон
  (крупные числовые поля, минимум разворачивающихся блоков), но создаёт
  **те же** записи и проходит **те же** валидации/гарды, что десктопная.
- **FR-012**: Мобильный аналог shortfall-prompt при недостатке средств;
  empty state при отсутствии активной петли.

### My Items & Stash (US3)

- **FR-013**: Игрок видит инвентарь своего PC (паритет с
  `<InventoryGrid>`/`<InventoryTab>`), читаемый на узком экране; тап —
  детали предмета.
- **FR-014**: Игрок видит баланс и инвентарь общака (складчина) и может
  положить/взять (`<StashButtons>`-эквивалент) с теми же гардами
  владения предметом.
- **FR-015**: Ноды предметов открываются на мобильном в read-вью (а не
  в desktop edit-grid).

### Live Encounter (US4 — P2)

- **FR-016**: Игрок видит активный энкаунтер вертикальным списком
  карточек участников в порядке инициативы; видны: чей ход, своё HP,
  свои условия. Не широкий грид.
- **FR-017**: Экран игрока отражает изменения состояния энкаунтера,
  внесённые ДМ. **Механизм** (live-подписка vs polling vs pull-to-refresh)
  — Open question Plan-фазы; сегодня трекер не realtime.
- **FR-018**: Игрок может совершать действия своего хода, если роль
  это позволяет; write гейтится как на десктопе (BUG-018: никакой
  записи в чужие строки, ранний exit с одним alert'ом).
- **FR-019**: Игрок-наблюдатель (не участник боя) может смотреть, но
  не править. Empty state при отсутствии активного энкаунтера.

### Installable PWA (US5)

- **FR-020**: Приложение устанавливается на домашний экран на Android и
  iOS и запускается в standalone-режиме (своя иконка, без браузерной
  адресной строки). Без App Store / Google Play.
- **FR-021**: На iOS (нет авто-промпта) приложение показывает
  ненавязчивую, скрываемую, одноразовую подсказку про установку через
  Share → «На экран Домой».
- **FR-022**: Новые деплои доставляются пользователю без ручных действий
  и без ревью магазина.
- **FR-023**: Открытие приложения на десктопе **не ломает** существующий
  desktop-UX ДМ (см. Open question про маршрутизацию/детект контекста).

### Non-functional / parity

- **FR-024**: Режим игрока — **линза, не форк**: те же данные, та же
  авторизация, тот же Supabase, те же server actions. Никакого
  дублирования бизнес-логики между десктопом и мобильным.
- **FR-025**: Перенос идёт «как есть»: фича-паритет с существующими
  десктопными поверхностями по содержанию. Новые возможности (портреты,
  push, оффлайн-правки) — вне scope (см. Out of Scope).

## Key Entities

Новых таблиц **не ожидается** (MVP). Используем существующее:

- `nodes` (PC-ноды, `type` персонажа) — чарник.
- `node_pc_owners` — связь игрок↔PC (чьи персонажи показывать).
- `transactions` — баланс, инвентарь, история, статусы approval
  (поля spec-014), `is_starter_setup` (spec-012).

## Success Criteria

1. **SC-001**: Игрок устанавливает приложение на домашний экран (iOS и
   Android) и запускает его в standalone-режиме одним тапом, без магазина.
2. **SC-002**: При запуске игрок с одним PC видит своего персонажа без
   промежуточного выбора кампании; с несколькими — попадает к нужному
   PC за ≤ 2 тапа.
3. **SC-003**: Чарник, деньги и предметы полностью пригодны к
   использованию на телефоне без горизонтального скролла и без зума;
   тач-таргеты ≥ ~44px.
4. **SC-004**: Игрок создаёт транзакцию с телефона; она проходит тот же
   approval flow и появляется с корректным статусом — так же, как при
   создании на десктопе.
5. **SC-005**: Игрок кладёт/берёт предмет из общака с телефона с тем же
   результатом и теми же гардами, что на десктопе.
6. **SC-006**: Никакой бизнес-логики/валидации не продублировано — мобильные
   действия вызывают те же server actions и проходят тот же auth-контракт
   (проверяется чтением кода / отсутствием дублей).
7. **SC-007**: Игрок не имеет доступа к DM-only инструментам из режима
   игрока; видимость данных и право правки не изменились относительно
   десктопа.
8. **SC-008**: Открытие приложения на десктопе не ломает существующий
   desktop-UX ДМ.
9. **SC-009** (если US4 в MVP): игрок открывает активный энкаунтер на
   телефоне, видит порядок инициативы / своё HP / условия вертикальным
   списком и видит изменения, внесённые ДМ (механизм — по решению Plan).

## Out of Scope

- **Миграция инфраструктуры** (Supabase vs self-host) и **объектное
  хранилище для изображений** — ортогонально; отдельное решение/задача.
- **Портреты/аватары персонажей и изображения предметов** — новых
  изображений в коде сейчас нет; добавление зависит от решения по
  объектному хранилищу → fast-follow, не 022 MVP.
- **Push-уведомления** (например «ДМ одобрил транзакцию», «начался бой») —
  технически возможны в PWA (iOS 16.4+ для установленных), но вне MVP.
- **Оффлайн-правки / синхронизация при потере сети** — по умолчанию режим
  работает онлайн; полноценный оффлайн — отдельная фича.
- **Нативная обёртка / публикация в App Store / Google Play** — сознательно
  отвергнуто в пользу PWA (это и есть цель пользователя).
- **DM-инструменты на мобильном** (граф, каталог-редактор, подготовка
  сессий, трекер с правами ДМ) — остаются desktop-only (конституция).
- **Спеллы и слоты** (IDEA-029) — отдельная большая фича, ждёт своей очереди.
- **Полный мобильный редизайн всех страниц кампании** — 022 покрывает
  четыре названных игроком поверхности, не весь сайт.
- **Мобильная версия публичного `/docs`** — отдельно, по запросу.
