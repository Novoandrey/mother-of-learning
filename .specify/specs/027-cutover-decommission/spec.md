## Context

### Зачем этот срез

К началу 027 на боксе Hetzner живут одновременно: (а) приложение в Dokploy
(`staging.theloopers.org`, фронт+бэк, standalone), которое **сейчас смотрит на
managed Supabase**; (б) self-hosted Supabase (обрезанный стек db/auth/rest/kong/
studio/meta, PG17), в котором **уже лежит верная копия прод-данных** (026:
counts == managed, `auth.users` с хешами, реальный логин проверен). Боевой
трафик идёт через **Vercel** (`mother-of-learning.vercel.app`) → managed
Supabase — это сегодняшний прод, которым пользуются реальные игроки.

Цель эпика — съехать с managed-инфры на собственный сервер ради DevOps-навыка.
024–026 подготовили всё **параллельно**, ничего не переключая. 027 — единственный
срез, который трогает **живой трафик**: переводит приложение и игроков на
собственный бокс + self-hosted Supabase, выводит Vercel и (после грейса) managed.

### Текущая топология и что меняется

| | Сейчас (до 027) | После 027 |
|---|---|---|
| **Хостинг приложения** | Vercel (`mother-of-learning.vercel.app`) — боевой; Dokploy (`staging.theloopers.org`) — стейджинг | **Dokploy на боксе** — боевой; Vercel выведен |
| **База** | managed Supabase (боевая) | **self-hosted Supabase** на том же боксе (боевая) |
| **Боевой домен** | `mother-of-learning.vercel.app` | `theloopers.org` (Cloudflare → Traefik на боксе) |
| **managed Supabase** | боевая | держится грейс ~1–2 нед как откат/эталон → гасится |
| **Версия** | 0.9.0 | **1.0.0** (веха «своя инфра») |

Полный переезд фронта+бэка на бокс зафиксирован ещё на Clarify 023 («A2: полный
переезд; Vercel выводится из эксплуатации на cutover»). 027 это исполняет.

### Ключевая вводная: браузер ходит в Supabase напрямую → API нужно публиковать

Аудит кода (chat 87): клиентский слой использует
`createBrowserClient(NEXT_PUBLIC_SUPABASE_URL, NEXT_PUBLIC_SUPABASE_ANON_KEY)` —
**браузер обращается к Supabase напрямую** (логин через GoTrue, часть запросов
под anon/authenticated с RLS). Значит вариант «self-hosted доступен только по
внутренней docker-сети, наружу ничего» — **не годится**: до self-hosted API
должен дотягиваться браузер игрока. Это разрешает развилку из `NEXT.md`
(внутренняя сеть vs публикация kong) в пользу **публичного TLS-эндпоинта API
за Traefik**.

Из этого — требование-ограничение к экспозиции (см. US1): наружу публикуется
**только** API-гейтвей (kong) одним TLS-эндпоинтом; Postgres (5432) остаётся
закрытым; Studio — по-прежнему только по SSH-туннелю; в браузер уходит **только
anon-ключ**, `service_role` живёт server-side; RLS работает как на проде.
Конкретный поддомен, маршрут Traefik и разделение «server-side по внутренней
сети vs browser по публичному URL» — решение Plan; **сам факт необходимости
публичного API-эндпоинта — установлен здесь**.

### Дельта данных с 026 → нужен финальный свежий синк

026 скопировал данные **параллельно проду** (snapshot). С тех пор боевой трафик
(Vercel → managed) продолжает накапливать изменения: игроки правят персонажей,
пишут транзакции, и т.д. На момент cutover между managed (живой, растёт) и
self-hosted (snapshot 026) будет **дельта**. Если переключиться на snapshot,
свежие правки игроков потеряются.

Поэтому cutover обязан включать **финальный свежий перенос** managed →
self-hosted под коротким **фризом записи** (объявленное окно обслуживания), чтобы
self-hosted держал актуальное прод-состояние **на момент щелчка**. Данные малы
(~1200 нод) → полный повторный перенос тривиален и быстр; механика фриза и
повторного прогона (переиспользует скрипт 026) — деталь Plan, исход
(«ноль потерянных правок») — требование здесь.

### Откат: managed как страховочный эталон, не горячий standby

Managed Supabase держится **untouched** короткий грейс-период (~1–2 нед, совет
Леши) как путь отката и эталон сравнения. Важный нюанс: как только игроки начнут
писать **в self-hosted** после cutover, откат на managed (который с момента
фриза не растёт) потеряет эти новые записи. Поэтому откат — **немедленная
страховка** (минуты/часы сразу после щелчка, пока новых записей мало), а не
ленивый тумблер «через неделю вернёмся». Через грейс managed служит источником
для восстановления/сравнения, не горячим зеркалом. Это фиксируется в правиле
отката (US4).

### Что 027 выводит из эксплуатации (decommission)

- **Vercel** — выводится **на cutover** (deployment остановлен/удалён, домен на
  него больше не указывает, доступ снят). Зафиксировано на Clarify 023.
- **managed Supabase** — гасится **после грейс-периода** (~1–2 нед), когда
  self-hosted стабильно отработал как боевой. Фактическое удаление — финальный
  пункт чек-листа, исполняется оператором по истечении окна; до этого managed
  жив и не тронут.
- **`staging.theloopers.org`** — выводится; на боксе остаётся **один env**
  (боевой на apex `theloopers.org`). Отдельного стейджинга после переезда нет
  (Clarify chat 87). Поддомен/деплой staging в Dokploy снимается на cutover.

## Operator Scenarios & Testing

### US1 — Безопасный доступ к self-hosted + end-to-end репетиция параллельно проду (P1)

Оператор подключает приложение в Dokploy к self-hosted Supabase **на нерабочем
URL** (текущий `staging.theloopers.org`), публикует self-hosted API минимально и
безопасно, и проверяет, что **всё приложение работает end-to-end против
self-hosted** — это генеральная репетиция, ещё **без** переключения боевого
трафика (Vercel/managed-прод не тронут).

#### Acceptance Scenarios

1. **API опубликован минимально** — self-hosted Supabase доступен браузеру по
   одному TLS-эндпоинту через Traefik; наружу опубликован **только** kong;
   Postgres (5432) закрыт; Studio — только по SSH-туннелю.
2. **Ключи разведены правильно** — в браузер уходит **только anon-ключ**;
   `service_role` нигде не утекает в клиент; server-side actions работают
   (через внутреннюю сеть или тот же эндпоинт — решение Plan).
3. **Логин работает** — существующий игрок логинится против self-hosted Auth
   **текущим паролем** (хеши из 026), сессия держится между переходами (cookie-
   refresh в proxy работает).
4. **Чтение под RLS корректно** — каталог/ноды/транзакции читаются; запрос под
   `authenticated` уважает RLS так же, как на проде; `anon` не видит
   защищённого (выборочно).
5. **Запись проходит весь путь** — игрок подаёт транзакцию (pending), DM
   аппрувит (approval flow), запись фиксируется; server action + RPC отрабатывают
   против self-hosted.
6. **Прод не затронут** — Vercel/managed-прод в ходе репетиции доступны и
   неизменны; реальные игроки продолжают пользоваться продом на Vercel.

#### Edge Cases

- **`NEXT_PUBLIC_*` в Dokploy** — известный грабли (023): эти значения надо
  дублировать в **Build-time Arguments**, иначе сборка отдаёт 500 «URL and Key
  are required». Новый `NEXT_PUBLIC_SUPABASE_URL` (self-hosted) пересобирается.
- **Cloudflare ↔ Let's Encrypt** — TLS на боксе через Traefik/LE; при оранжевом
  облаке Cloudflare нужен SSL-режим «Full (strict)» или DNS-only, иначе redirect-
  петля/ошибка серта. Паттерн уже отлажен для `staging.theloopers.org` (023) —
  переиспользовать.
- **CORS / Site URL GoTrue** — self-hosted Auth должен принимать редиректы/
  origin нового домена; проверить `SITE_URL`/`ADDITIONAL_REDIRECT_URLS`.

### US2 — Финальный свежий синк managed → self-hosted под фризом (ноль потерь) (P1)

В момент cutover оператор выполняет **свежий полный перенос** актуальных
прод-данных с managed в self-hosted под коротким объявленным окном обслуживания
(запись в managed заморожена), так что self-hosted держит последнее прод-состояние
на момент щелчка — свежие правки игроков с момента 026 не теряются.

#### Acceptance Scenarios

1. **Окно объявлено** — игрокам сообщено короткое окно обслуживания; на время
   синка запись в managed не идёт (механика фриза — Plan).
2. **Свежий перенос отработал** — повторный прогон скрипта 026 (managed →
   self-hosted) завершается без `FATAL`; конфликтов находки 025 (ownership /
   duplicate `schema_migrations`) нет.
3. **Counts совпали на момент щелчка** — для ключевых таблиц `public` и
   `auth.users` число строк в self-hosted **== managed** на момент фриза
   (подсчёт мимо клэмпа `db_max_rows=1000`, прямой SQL `count(*)`).
4. **Целостность подтверждена** — выборочно: транзакции → существующие ноды,
   рёбра → endpoints, `item_node_id` разрешается, `auth.users` ↔ public.
5. **Последовательности синхронизированы** — `setval` по `max(id)` выполнен;
   тестовая вставка не падает duplicate-key (как в 026).
6. **Свежие правки на месте** — выборочно: изменение, сделанное игроком на
   проде **после** 026, присутствует в self-hosted после финального синка.

#### Edge Cases

- **Длина окна** — объём мал (~1200 нод), окно должно быть короткое (минуты);
  зафиксировать фактическую длительность (метрика для отчёта).
- **Повторный прогон при сбое** — если синк упал на полпути, его можно
  безопасно перезапустить («снёс self-hosted data → залил заново» либо
  idempotent — выбор Plan), не накапливая дубли.
- **Расхождение counts** — если не сходятся, cutover **не продолжается**:
  откат фриза (managed снова принимает запись), диагностика.

### US3 — Cutover: боевой трафик на self-hosted; Vercel выведен; реальный игрок проверил (P1)

Оператор переключает боевой трафик: приложение на боксе (Dokploy) против
self-hosted Supabase обслуживается на **боевом домене** (`theloopers.org`),
Vercel выводится из эксплуатации, реальный игрок проходит end-to-end проверку
на новом проде, окно обслуживания закрывается.

#### Acceptance Scenarios

1. **Домен указывает на бокс** — боевой домен `theloopers.org` резолвится в
   приложение на боксе (через Cloudflare → Traefik), HTTPS валиден.
2. **Приложение на self-hosted** — приложение (env: Supabase URL + anon-ключ)
   указывает на self-hosted; `service_role` server-side — на self-hosted.
3. **Vercel выведен** — Vercel-деплой остановлен/удалён, боевой домен на него
   не указывает; старый `mother-of-learning.vercel.app` больше не обслуживает
   боевой трафик.
4. **Реальный игрок проверил end-to-end** — существующий игрок на боевом домене:
   логин текущим паролем → читает свои данные → делает запись (полный путь,
   при необходимости с аппрувом DM) — успех.
5. **Окно закрыто** — фриз снят, запись идёт **в self-hosted** (новый прод);
   игрокам сообщено, что обслуживание завершено.
6. **Нет двойной записи** — после щелчка прод-записи идут только в self-hosted;
   managed больше не получает боевых записей (он заморожен/выведен из тракта).

#### Edge Cases

- **DNS-распространение** — переключение домена может распространяться не
   мгновенно; учесть TTL (на Cloudflare обычно быстрый) и проверить с
   нескольких точек.
- **Залипшие сессии** — у игроков, залогиненных на старом домене, могут быть
   cookie старого origin; задокументировать «перелогиньтесь», если потребуется.
- **Smoke до анонса** — end-to-end проверку реальным игроком делать **до**
   объявления «всё готово», а не после.

### US4 — Откат отрепетирован и задокументирован; managed — страховочный эталон (P1)

Оператор умеет быстро откатиться, если cutover повёл себя плохо: вернуть env на
managed (и/или вернуть домен на Vercel), потому что managed жив и не тронут весь
грейс-период. Шаги и **правило** отката задокументированы и хотя бы прогнаны
вхолостую **до** реального cutover.

#### Acceptance Scenarios

1. **Правило отката задокументировано** — что считается «cutover пошёл не так»
   (какие сбои триггерят откат), и **окно**, в котором откат чистый (сразу после
   щелчка, пока новых записей в self-hosted мало).
2. **Шаги отката copy-paste** — вернуть Supabase env приложения на managed +
   (при необходимости) вернуть боевой домен на Vercel + снять фриз/принять
   запись обратно на managed — конкретные команды/действия.
3. **Откат отрепетирован** — путь назад прогнан вхолостую (dry-run) до
   реального cutover; время отката замерено.
4. **managed untouched весь грейс** — managed Supabase жив, неизменён, доступен
   как источник восстановления/сравнения на ~1–2 нед.
5. **Осознанная потеря при позднем откате** — задокументировано: откат **после**
   накопления новых записей в self-hosted теряет их (managed не рос) → поздний
   откат требует обратного синка, это не штатный путь.

#### Edge Cases

- **Частичный сбой** — если упал только домен/DNS, но приложение и self-hosted
  здоровы — откатывать домен, а не всё; правило различает уровни.
- **Откат без потерь невозможен** — если новые записи уже значимы, фиксируем
  «вперёд + чинить», а не откат; критерий — в правиле.

### US5 — Бамп версии + бэкапы покрывают теперь живой прод (P1)

На cutover версия бампается (инфра-эпик откладывал бамп сюда), и off-box
R2-бэкапы (025/026), отрепетированные на параллельном self-hosted, подтверждённо
продолжают идти теперь, когда self-hosted стал боевым — защищая **реальные**
прод-данные.

#### Acceptance Scenarios

1. **Версия бампнута** — `package.json` версия поднята **0.9.0 → 1.0.0** (веха
   «своя инфра»), изменение закоммичено в составе cutover'а.
2. **Ночной cron активен на боевом self-hosted** — бэкап-cron работает,
   снимает физический бэкап теперь-боевого self-hosted, заливает в R2, exit 0,
   без `ERROR`/`FATAL`.
3. **Бэкап содержит боевые данные** — выборочно подтверждено, что свежий бэкап
   захватывает актуальные прод-данные + `auth.users` с хешами (на restore, не
   на глаз).
4. **Ротация цела** — ротация (30/28) и off-box хранение не сломаны
   переключением; меняется только то, что бэкапится теперь боевая БД.

#### Edge Cases

- **Бэкап во время живого трафика** — физический cold-copy подразумевает краткую
  остановку `db`; на боевом self-hosted это короткий ночной downtime — учесть
  окно/тайминг (или подтвердить `pg_basebackup` без downtime — наследие 026).
- **Версия в UI/футере** — если версия где-то показывается, проверить, что
  бамп отразился после деплоя.

### US6 — Decommission: managed (после грейса) + Vercel, по чек-листу; runbook полон (P1)

У оператора есть проверенный copy-paste runbook cutover'а и чек-лист
decommission. Vercel выведен на cutover; managed Supabase гасится по истечении
грейса по задокументированным шагам — финальный пункт эпика.

#### Acceptance Scenarios

1. **Runbook cutover задокументирован** — без догадок: публикация API →
   dress-rehearsal → объявление окна → фриз → финальный синк → щелчок env/домена
   → вывод Vercel → end-to-end проверка → снятие фриза → бамп версии. Переносимая
   часть (публикация API, бэкап live) — в `infra/`; app-specific (env, домен,
   синк) — в `.specify/specs/027-*`.
2. **Vercel + staging выведены и зафиксированы** — шаги вывода Vercel
   (остановка/удаление деплоя, снятие домена/доступа) **и снятия
   `staging.theloopers.org`** (на боксе остаётся один env на apex)
   задокументированы и исполнены.
3. **Чек-лист decommission managed** — задокументированы шаги гашения managed
   **после грейса** (что проверить перед удалением: self-hosted стабилен,
   свежий бэкап есть; затем удаление проекта managed). Дата старта грейса
   зафиксирована.
4. **Грейс соблюдён** — managed не гасится до истечения окна; гашение —
   осознанный финальный тик после подтверждённой стабильности.
5. **Эпик закрыт** — после вывода Vercel и (по истечении грейса) гашения managed
   приложение полностью на собственной инфре; `NEXT.md` отражает новое прод-
   состояние; цель эпика (DevOps-навык на своём боксе) достигнута.

#### Edge Cases

- **Гашение managed — отложенное действие** — фактическое удаление исполняется
  оператором через ~1–2 нед, не в этой сессии; спека поставляет чек-лист и
  фиксирует грейс, тик закрывается позже.
- **Откат Vercel после вывода** — если Vercel удалён, а не приостановлен,
  быстрый откат на него невозможен; правило отката (US4) учитывает, что после
  вывода Vercel страховка — это self-hosted-сторона (env назад на managed при
  живом приложении на боксе), а не возврат на Vercel.

## Out of Scope

- **Изменения функциональности приложения** — 027 чисто инфра/переключение, код
  приложения не меняется (кроме бампа версии).
- **Публикация наружу чего-либо кроме kong** — Studio остаётся по SSH-туннелю;
  Postgres не публикуется; analytics/realtime/storage не поднимаются (их нет с
  024).
- **PITR / WAL-archiving** (`wal-g`/`pgBackRest`) — будущее улучшение.
- **Шифрование бэкапов** — опциональная фаза, не здесь (как в 025/026).
- **Мониторинг/алертинг прода и бэкапов** — кандидат в backlog (после переезда
  желателен, но вне этого среза).
- **CI/CD-полировка деплоя на боксе** — авто-деплой по push, превью-окружения и
  т.п. — отдельно; здесь достаточно ручного `git pull`/redeploy в Dokploy.
- **Spec-022 (Player Mobile Mode)** и **портреты/R2** — следующие после эпика.
- **Двусторонняя/постоянная репликация managed ↔ self-hosted** — cutover это
  one-shot переключение, не репликация; после грейса managed уходит.

## Где живут артефакты

Переносимая инфраструктура (публикация API за Traefik как паттерн, бэкап
теперь-боевого self-hosted, обновлённый runbook) — в repo-root `infra/` (как
весь эпик: переносимое отделено от app-specific). App-specific (env приложения,
переключение домена, финальный синк-прогон, чек-лист decommission, бамп версии)
и спека/план — в `.specify/specs/027-*`.
