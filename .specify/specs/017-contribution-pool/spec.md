# Feature Specification: Складчина (real-money chip-in)

**Feature Branch**: `017-contribution-pool`
**Created**: 2026-04-24 · **Rewritten**: 2026-04-26 (chat 72)
**Status**: Clarified (2026-04-26, chat 72)
**Input**: User prompt — "новая страница для real-money скидок:
любой игрок создаёт карточку с заголовком, реквизитами,
участниками и суммами; автор закрывает чекбоксами кто скинулся;
когда все чеки проставлены — автоматом в Архив; две вкладки —
Текущие и Архив. Не городить сверхсложного, использовать
бухгалтерию как основу, закрыть быстро".

## Context

За реальным столом всегда есть боковой канал реальных денег:
кто-то заказал пиццу на сессию, кто-то снял комнату, кто-то
скинулся на миниатюры. Сейчас это живёт в Telegram-треде или в
голове DM'а, и человек, который выложил деньги, бегает за всеми
и помнит наизусть «кто отдал, кто нет». Фича — одно место в
приложении, где автор скида ведёт чеклист «кто скинулся», а
остальные участники видят сколько они должны.

**Naming.** В UI фича называется **Складчина** — точное русское
слово для «скинулись на общее». Не путается с «Сборами» (слишком
формально, плюс ассоциация с налогами), «Кассой» (cash register)
и «Общаком» (уже занят spec-011 — in-game stash).

**Architectural placement.** Складчина — sidecar к in-game ledger
(spec-010/011). Она **не** трогает таблицу `transactions`, ноды,
петли, сессии. Тот же Supabase, та же auth, отдельные таблицы.
Visually переиспользует patterns страницы `/accounting`: tabs,
list-based UI, simple row components.

## Goals

- **P1**: автор скида в одном месте видит «кто отдал, кто нет»
  по каждой Складчине, и ставит галочки одним тапом.
- **P2**: участник кампании за 1 навигацию (sidebar → Складчина)
  отвечает на вопрос «должен ли я кому-то».
- **P3**: закрытые Складчины сами уходят в архив; история
  читается как лента «на что мы скидывались».

## Non-goals

- Реальные платежи / payment processing. Деньги ходят out-of-band
  (Тинькофф, Сбер, наличка). Приложение трекает intent +
  confirmation.
- Currency picker. Одна implicit валюта на кампанию (для
  mat-ucheniya — рубли). На UI — просто число, без валютного
  суффикса в MVP. Если в будущем нужна другая валюта — это
  отдельная спека.
- Уведомления, push, email, telegram-боты. Чейзинг — ручной.
- FX, конвертация, «эквивалент в рублях».
- Cross-campaign Складчины. Если две кампании скинулись на одну
  комнату — две отдельные Складчины.
- Любая интеграция с in-game `transactions` / wallet / общаком.
- Per-pool комментарии / дискуссии. Болталка в Telegram.
- CSV / XLSX export.
- Шаблоны / повторяющиеся Складчины.
- Hard delete. Soft delete only.
- Role gating создания. MVP: любой member кампании создаёт.

## User Scenarios

### US1 — Андрей заказал пиццу на сессию (P1)

**Setup**: Андрей — DM кампании mat-ucheniya. На сессии заказали
пиццу на 4500 ₽, расплатился он картой. Хочет получить деньги с
участников.

**Flow**:

1. Открывает sidebar → **Складчина** → таб **Текущие** → кнопка
   `+ Складчина`.
2. Заполняет:
   - **Название**: `Пицца 24 апреля`
   - **Реквизиты** (free text): `Тинькофф +7 999 555 12 34` (или
     просто `мой`, или `Сбер 4276…` — что захочет).
   - **Участники**: multi-select из списка member'ов кампании +
     поле «добавить вручную» (free-text имя), куда вбивает «Петя»
     (друг, который пришёл на сессию, в кампанию не добавлен).
     Получает 5 member'ов + 1 ad-hoc = 6 участников.
   - **Общая сумма**: `4500`.
   - **Разделить поровну** (кнопка) → таблица заполняется: 6 строк
     × 750 ₽. Можно отредактировать любую строку вручную.
3. Если sum(строк) ≠ total — красный баннер «Сумма не бьётся:
   4500 ≠ 4490», кнопка `Создать` disabled.
4. Сохраняет → Складчина появляется в табе **Текущие** для всех
   member'ов кампании.

**В течение недели**: люди скидывают Андрею деньги по Тинькоффу.
Каждый раз он открывает страницу, тапает чекбокс рядом с именем
заплатившего → строка получает `paid_at = now()`.

**Когда все 6 чекбоксов проставлены** — Складчина автоматом
исчезает из **Текущие** и появляется в **Архив**.

**Acceptance**:

1. **Given** member кампании на странице Складчина, **When**
   тапает `+ Складчина` и заполняет всё валидно, **Then**
   Складчина создаётся, видна в табе «Текущие» всем member'ам.
2. **Given** Складчина открыта, **When** автор тапает чекбокс
   участника, **Then** строка получает `paid_at`, summary
   обновляется (`750 / 4500`), no confirm.
3. **Given** все участники отмечены paid, **When** автор тапает
   последний чекбокс, **Then** Складчина уходит из «Текущие» и
   появляется в «Архив» с пометкой «закрыта DD.MM.YYYY».
4. **Given** archived Складчина, **When** автор расжимает paid
   чекбокс на любой строке, **Then** Складчина возвращается в
   «Текущие». Status — computed, не хранится.
5. **Given** member кампании, который **не** в участниках,
   **When** открывает страницу, **Then** видит Складчину в
   «Текущие» как read-only с пометкой «не участвую».
6. **Given** member, который **в** участниках и не отмечен paid,
   **When** открывает страницу, **Then** видит «ты должен 750 ₽»
   на карточке.

### US2 — Андрей правит Складчину (P2)

После создания автор может:

- Поменять название.
- Поменять реквизиты.
- Добавить/убрать участников (только тех, кто ещё не paid).
- Поменять total и/или per-person суммы.

**Правило**: `paid` строки заморожены в своей сумме. Если автор
меняет total — diff расходится между `unpaid` строками. Если
`total < sum(paid rows)` — блок «нельзя сделать total меньше уже
собранного, отметьте кого-то unpaid и попробуйте снова».

### US3 — Удаление Складчины (P2)

Автор (или DM) тапает «удалить» в меню карточки → confirm →
soft-delete. Складчина исчезает отовсюду. История остаётся в БД
для случая «ой, не то удалил» (вручную через SQL — UI undelete
не делаем в MVP).

## Functional Requirements

### Data model

Две таблицы (точные DDL — на этапе `/plan`, схема ниже —
концептуальная):

**`contribution_pools`** (header)

- `id` uuid PK
- `campaign_id` uuid → `campaigns(id)`
- `created_by` uuid → `auth.users(id)`
- `title` text NOT NULL (≤ 100 chars)
- `payment_hint` text (≤ 200 chars, nullable, free-form)
- `total` numeric(12, 2) > 0
- `created_at` / `updated_at` / `deleted_at` (soft delete)

**`contribution_participants`** (rows)

- `id` uuid PK
- `pool_id` uuid → `contribution_pools(id)` ON DELETE CASCADE
- `user_id` uuid → `auth.users(id)`, **nullable** (NULL = ad-hoc
  участник, не member кампании)
- `display_name` text NOT NULL — snapshot для linked rows (чтобы
  отображение не ломалось если member выйдет из кампании); raw
  для ad-hoc
- `share` numeric(12, 2) ≥ 0
- `paid_at` timestamptz, nullable (NULL = unpaid)
- `created_at`

**No status enum.** Архивность Складчины — derived:

```
archived = (deleted_at IS NOT NULL)
        OR (every participant has paid_at IS NOT NULL)
```

### Behaviour

- **FR-001**: Любой member кампании создаёт Складчину.
- **FR-002**: Title required, ≤ 100 chars.
- **FR-003**: Реквизиты (`payment_hint`) optional, ≤ 200 chars,
  free-form.
- **FR-004**: Total > 0, 2-decimal precision.
- **FR-005**: Минимум 1 участник.
- **FR-006**: Кнопка **«Разделить поровну»** = `total / N`
  округление вниз до копеек, остаток (`total - sum(rounded)`)
  добавляется к **первой** строке (детерминизм важнее equity —
  автор может перетыкать вручную если хочет).
- **FR-007**: На submit: `sum(shares) == total`. Если нет —
  inline баннер «Сумма не бьётся: {sum} ≠ {total}», save
  заблокирован.
- **FR-008**: Mark-paid = 1 тап на чекбокс участника. Без
  confirm dialog. Только автор + DM.
- **FR-009**: Когда **все** participants имеют `paid_at IS NOT
  NULL` → Складчина derives как archived → автоматом скрывается
  из таба «Текущие» и появляется в табе «Архив». Никаких schema
  flag'ов / триггеров — view-level filter.
- **FR-010**: Если автор расжимает paid чекбокс на archived
  строке → Складчина возвращается в «Текущие» автоматически
  (опять же — derived).
- **FR-011**: Edit Складчины: автор + DM.
- **FR-012**: Mark-paid: автор + DM.
- **FR-013**: View Складчины: любой member кампании.
- **FR-014**: Soft-delete Складчины: автор + DM. Удалённые
  попадают в «Архив» как archived с пометкой «удалено».
- **FR-015**: Top-level страница `/c/[slug]/skladchina`. Два
  таба: **Текущие** (active, default) / **Архив** (closed +
  deleted).
- **FR-016**: Sidebar nav entry «Складчина» рядом с
  «Бухгалтерия».
- **FR-017**: На карточке Складчины в списке: title, автор,
  total, прогресс-сабтайтл `Y / Total ₽` (`750 / 4500 ₽`),
  per-user hint:
  - Автор → chip «Автор».
  - Участник unpaid → «ты должен X ₽».
  - Участник paid → «ты сдал ✓».
  - Не участник → «не участвую» (серым).
- **FR-018**: На детальной странице Складчины — таблица всех
  участников с чекбоксами; видно всем member'ам кампании,
  чекбоксы кликабельны только автору + DM.

### RLS (concept, finalize в `/plan`)

- `contribution_pools`: SELECT всем member'ам кампании; INSERT
  любому member'у; UPDATE / DELETE — created_by или DM.
- `contribution_participants`: SELECT всем member'ам кампании
  через `pool_id`; INSERT / UPDATE / DELETE — created_by пула
  или DM.

## Success Criteria

- **SC-001**: Создание Складчины с 6 участниками + equal split
  ≤ 15 секунд на мобиле от cold нав.
- **SC-002**: Mark paid = **1 тап** (no confirm, no nav).
- **SC-003**: Pizza-test: следующую IRL пиццу/комнату/миньки
  Андрей трекает в Складчине, не в Telegram. Если уходит в
  Telegram — фича провалена.
- **SC-004**: Member кампании отвечает на «должен ли я» в **1
  навигацию** (sidebar → Складчина).

## Clarifications

_Resolved 2026-04-26 in chat 72._

### Q1: Имя фичи

**Answer: Складчина.** Точное русское слово для «скинуться на
общее»; в современной речи живо, не путается с «Сборы»
(формально-канцелярское), «Касса» (cash register / fund),
«Общак» (уже занят spec-011 in-game stash). Альтернативы
рассмотрены и отклонены: «Раскладка» — слишком широкое,
«Скид» — slang, «Кружка» — узко.

Если пользователь захочет другое имя — переименование = sed
по 4–5 файлам (компонент + page + sidebar entry + spec
references). Не блокер ни для plan, ни для implement.

### Q2: Currency display

**Answer: В MVP — просто число.** Без валютного суффикса в UI
(`750`, не `750 ₽`). mat-ucheniya — единственная активная
кампания, валюта подразумевается RUB. Любой суффикс — это
неявный лок на одну валюту, а это уже отдельная спека с
per-pool / per-campaign currency setting. Один pull-rope,
который тянет за два узла, не трогаем.

В schema колонки `currency` нет. Если в будущем появится — это
ALTER TABLE add column with default 'RUB', не миграция данных.

### Q3: Edit участников после создания

**Answer: Paid строки нельзя удалить или редактировать share.**
Чтобы убрать paid участника — сначала unmark (расжать чекбокс).
Edit total → diff расходится между unpaid строками; если
`new_total < sum(paid)` → блок «нельзя сделать total меньше
уже собранного». Unpaid строки — добавляются / убираются /
правятся свободно (автор + DM).

Rationale: paid снимок — это «зафиксированный факт получения
денег». Менять share постфактум = потерять историю «сколько
конкретно мне отдал Вася». Если автор реально хочет переделать
— unmark, перераспредели, mark обратно.

### Q4: Ad-hoc → linked post-hoc

**Answer: Skip в MVP.** Выбор «из member'ов / свободный текст»
делается **только** при создании / редактировании participant
row. Если автор написал «Петя» текстом, а потом Петю добавили
в кампанию — автор удаляет ad-hoc строку и добавляет Петю как
linked. Один шаг, один раз — не стоит отдельной фичи.

Если возникнет реальный pain (Петя уже сдал деньги, а его
надо «связать с user_id» — паника из-за потери истории) — это
P3, не MVP.

### Q5: Реквизиты — free text vs structured

**Answer: Free text, ≤ 200 chars, никакой валидации.**
Поле `payment_hint` text. Может быть пустым. Может быть
телефон («+7 999 555 12 34»), может быть «Тинькофф ХХХХ»,
может быть «мой» (если автор — единственный с очевидной
платёжкой), может быть «наличкой при встрече». Структурный
формат («номер карты», «номер телефона», «SBP») — отдельная
итерация (P3+ после реального usage).

«Скопировать в буфер» в MVP **есть** на UI как кнопка
рядом с реквизитами (cost = 1 строчка `navigator.clipboard`,
QoL win) — но без парсинга / структуры.

### Q6: Visibility удалённых Складчин

**Answer: Видны в табе Архив с пометкой «удалено».** Soft-
delete не означает «исчезла из UI». В Архив попадают:
1. Складчины где все participants paid (closed).
2. Складчины с `deleted_at IS NOT NULL` (deleted).

Визуально различаются: closed — серый чек, дата закрытия;
deleted — overlay «удалено DD.MM.YYYY», заголовок
зачёркнут, без интеракций. Undelete в UI **нет** — если
автор передумал, восстанавливать вручную через SQL (rare
edge case, не оправдывает UX).

### Q7: Sort + pagination в Архив

**Answer: Sort by `archived_at desc` (computed), без
pagination в MVP.** Closed → `MAX(participants.paid_at)`,
deleted → `deleted_at`. Coalesce двух → ORDER BY desc.
Лимит — 100 строк (~год активности кампании на 30 pool/год).
Если когда-то упрётся — добавим infinite scroll (как в
ledger, см. spec-010).

### Q8: Phone / payment validation

**Answer: Никакой валидации.** Free text — значит free text.
Не парсим, не нормализуем, не подсвечиваем. Если автор написал
«мой» — show «мой». Если написал «+7 999 555 12 34» — show
ровно так. Кнопка «копировать» рядом — копирует whatever as
is.

## Constitution check

- ✅ **I. Петля как ядро** — N/A. Складчина намеренно вне петли.
- ✅ **II. Атомарность** — paid flip = 1 update. Edit = 1 form
  save.
- ✅ **III-b. Плоская навигация** — top-level страница, не
  вложена.
- ✅ **IV. Данные-первичны** — состояние («кто сдал») в таблице,
  derived archived в SELECT.
- ✅ **V. Event sourcing** — флипаем `paid_at` in place.
  Сознательное упрощение для side-фичи. Если понадобится «когда
  именно я снял галочку» — добавим audit-log потом.
- ✅ **VI. Читалка** — mobile-first; форма ≤ 5 полей; main page
  = список карточек.
- ✅ **VII. Каждый релиз играбелен** — MVP альтернатив не
  требует.
- ✅ **VIII. Простота стека** — Supabase + Next.js, никакой новой
  инфры.
- ✅ **IX. Универсальность** — currency / campaign defaults
  будут читаться из `campaign_settings`, не хардкод mat-ucheniya.
- ✅ **X. Конституция кампании** — N/A.
