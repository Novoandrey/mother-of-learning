## Context

West-marches формат кампании (см. `docs/roadmap/west-marches.md`): ad-hoc пачки
PC ходят на вылазки, DM ведёт результат. Сейчас, чтобы отправить пачку и оформить
итог, DM руками: перечисляет участников в `session.participants`, вбивает награду
в общак, списывает расходники по кошелькам, пишет рекап. Это набор
детерминированных строк, которые каждый раз пишутся вручную — та же боль, что
закрывали spec-012 (loop-start) и spec-013 (encounter loot).

**Ключевой инсайт (архитектура):** вылазка структурно = spec-013 encounter loot —
**батч леджер-транзакций, материализованный из определения** (нода-вылазка) и
пересобираемый через autogen-примитив spec-012. Награда→общак и потраченные
расходники — это autogen-строки `transactions` с `autogen_wizard_key='expedition'`
и `autogen_source_node_id=<нода вылазки>`. Пачка = `session.participants`
(отдельная сущность `party` не нужна — конституция §3, DEBT-002). Нарратив — новый
тип события в ленте spec-053.

Вылазка НЕ тянет полную tick-миграцию времени (spec-057): её время — **локальные
метки на вылазке** (длительность в тиках кратно 6с), не сквозные часы кампании.
057 позже может формализовать это как частный случай.

## Decisions from briefing (✅ Andrey)

Дословно из брифинга: «пачка PC + время (тики кратны 6с) + награда→общак +
расходники (авто-покупка при отправке) + сохранить/повторить одной кнопкой
(дефолт — прошлый состав); в чат "X пошли туда, получили Z, потратили W"».

- ✅ **Пачка PC** — набор участников вылазки.
- ✅ **Время** — длительность в тиках, кратных 6с; метки локальны для вылазки.
- ✅ **Награда → общак** — итоговая добыча падает в общак.
- ✅ **Расходники** — авто-покупка при отправке.
- ✅ **Сохранить/повторить одной кнопкой** — дефолт = прошлый состав.
- ✅ **Нарратив в ленту** — «X пошли туда, получили Z, потратили W».

## User Scenarios & Testing

### US1 — DM собирает пачку и отправляет на вылазку (P1)

DM открывает «Вылазки», жмёт «Новая». Выбирает участников (тапами по PC
кампании — как быстрый party-picker поверх `session.participants`), задаёт
длительность (в понятных единицах → тики), указывает набор расходников
(предметы из каталога + количества) и награду. Жмёт «Отправить». Без лишних
шагов: расходники авто-покупаются (списываются), участники помечены «в вылазке»,
в ленту уходит событие «пачка вышла».

**Independent test:** пустая вылазка + 3 PC + 1 расходник → отправка → создаётся
нода вылазки, autogen-строки расхода, событие в ленте.

### US2 — Повторить прошлый состав одной кнопкой (P1)

Через сессию DM отправляет ту же пачку туда же. Жмёт «Повторить» — форма
предзаполнена прошлым составом + расходниками (дефолт). Правит при нужде,
отправляет. **Why P1:** это множитель ценности — рутинная пачка не должна
собираться заново каждый раз.

### US3 — Итог вылазки: награда в общак, расход зафиксирован (P2)

Вылазка завершается. DM вписывает добычу (деньги и/или предметы). Жмёт
«Свести». Награда падает autogen-строками в общак; расходники уже списаны при
отправке. В ленту уходит «X вернулись, получили Z, потратили W». Пересобираемо
(autogen 012): DM может поправить добычу и пересвести без дублей.

### US4 — Новый предмет прямо из вылазки (P2)

Добыт предмет, которого нет в каталоге. DM добавляет его как нода каталога
(`node_type='item'`) с названием + описанием, тут же кладёт в награду. (Развилка:
насколько «лёгкое» это добавление — см. открытые вопросы.)

## Requirements (draft FR)

- **FR-001** Вылазка — нода (`node_type='expedition'`, mirror-паттерн spec-013),
  несущая: участников (→ `session.participants` или ребро), длительность в тиках,
  набор расходников, награду, статус (draft / sent / done).
- **FR-002** Отправка вылазки MUST авто-покупать расходники → autogen-строки
  `transactions` (`autogen_wizard_key='expedition'`, source=нода вылазки),
  reuse buy-path spec-052 для цены.
- **FR-003** Свод итога MUST материализовать награду в общак autogen-строками;
  идемпотентно по `(expedition, source_node)` (реконсиляция spec-012).
- **FR-004** Форма MUST предзаполняться прошлым составом (дефолт «повторить»);
  где хранится «прошлый состав» — открытый вопрос (per-DM / per-campaign).
- **FR-005** Отправка и свод MUST слать событие в ленту 053 (новый
  `LedgerEvent` type `'expedition'`): «кто · куда · получили · потратили».
- **FR-006** DM-only (визард — действие ведущего). Игроки видят результат в
  ленте/общаке/на PC по обычным правилам видимости.
- **FR-007** Длительность MUST храниться как тики (кратно
  `tick_unit_seconds`=6); представление в UI — человекочитаемо. Метки локальны
  для вылазки, не пишут в сквозной campaign-clock (до spec-057).

## Открытые вопросы (нужно решение Andrey до Plan)

1. **Расходники — откуда списываются?** С общака (командный запас) или с
   кошельков участников (каждый платит свою долю)? Брифинг флагнул это развилкой.
   *Предложение Claude:* с общака по умолчанию (командная вылазка), опция «с
   кошельков поровну» later.
2. **Новый предмет = нода каталога с описанием?** ✅ похоже да. Насколько лёгкое
   добавление (только имя+описание, цена/атрибуты потом)? *Предложение:* имя +
   описание сразу, остальное — опционально (E3-подобно).
3. **Авто-слив батча.** Когда вылазка «сводится» — вручную кнопкой или авто
   (cron-подметание завершённых)? *Мой дефолт (из брифинга): cron-подметание.*
   Плюс — ручная кнопка «Свести сейчас».
4. **Модель времени вылазки.** Локальные метки — как именно хранить (просто
   `duration_ticks` на ноде? старт-метка? фронтир участников)? Связь с spec-057.
5. **Награда — фикс или бросок?** DM вписывает вручную или движок катает по
   таблице добычи? *Предложение:* v1 — вручную (как encounter loot); таблицы — позже.
6. **Только общак или можно на PC?** Награда всегда в общак, или DM может часть
   раздать участникам (как encounter loot US1)? *Предложение:* в общак по
   умолчанию, раздача — later (пересекается с 013).
7. **Место в UI:** десктоп DM-инструмент (`/c/[slug]/...`) и/или /tg?
   (AGENTS.md: desktop-first; /tg — если DM ведёт с телефона.)

## Dependencies / Reuse

- **spec-012 autogen-примитив** + **spec-013 encounter-loot** — переиспользуем
  wizard/reconcile/mirror-node/two-phase-confirm целиком, новый `wizard_key`.
- **spec-053 ledger feed** — новый `LedgerEvent` тип `'expedition'`.
- **spec-052 buy path** — авто-покупка расходников по цене каталога.
- **`session.participants`** — пачка (не новая `party` сущность).
- **spec-057 Время** (мягкая) — вылазка-локальные тики; 057 формализует сквозную модель.

---

## Out of Scope

- Полная tick-миграция кампании (→ spec-057).
- Таблицы случайной добычи / генерация энкаунтеров вылазки (→ позже; ср. `docs/roadmap/npc-movement-and-encounters.md`).
- `party` как постоянная сущность (конституция §3 — пачка это UI-агрегация).
- Календарь/availability-board west-marches (→ отдельная тема).
- Раздача награды по PC с even-split (пересекается с spec-013; v1 — в общак).

---

## Success Criteria

- **SC-001** DM собирает и отправляет пачку из N PC с расходниками за ≤ 30с;
  расходники списаны, событие в ленте.
- **SC-002** «Повторить» предзаполняет прошлый состав; повтор — ≤ 3 тапа.
- **SC-003** Свод кладёт награду в общак; пересвод неизменной вылазки = 0 новых строк (идемпотентность 012).
- **SC-004** Лента показывает «кто · получили · потратили» одним событием.
