## Context

Сейчас время в проде — `(loop_number:int, day_in_loop:int)` в `transactions`
(миграция 034). День — атомарная единица; «час» и «раунд боя» не различаются.
Этого хватает бухгалтерии, но ломается там, где нужен масштаб мельче/крупнее дня:
раунд боя (6с), длинное действие («3 месяца тренировок»), сравнение «кто раньше
дойдёт». Вылазки (055) и Крафт (056) уже упираются в это — им нужны локальные тики
и длинные действия.

**Целевая модель** (`docs/roadmap/tick-time-model.md`): `at_tick: bigint` от точки
отсчёта кампании вместо `(loop_number, day_in_loop)`. Один масштабируемый счётчик:
`tick_unit_seconds` (D&D 5e раунд = 6с) конфигурируется per-кампания; читаемое
«День 5, 14:00» — presentation layer, не схема.

Это **Группа B** инженерного пивота (`docs/roadmap/engine-pivot.md`) и в идеале
опирается на **Группу A** (универсальная таблица `events` с `at_tick`,
`generic-events-table.md`). Отсюда главное напряжение спеки: Andrey поставил
«Время» в очередь **до** RPG-движка, но полная миграция времени — тяжёлая и
трогает каждый запрос по `loop_number`. Нужно решить объём (ниже).

---

## Целевая модель (из роадмапа — не спорно)

- **`at_tick: bigint`** — тики от эпохи кампании. `bigint` ≈ бесконечность.
- **Per-кампания конфиг:** `tick_unit_seconds` (деф. 6), `ticks_per_day`
  (деф. 4800 = 8ч), `epoch_at` (опц. реальная дата).
- **Per-actor clock:** нода PC получает `current_tick` (null для не-PC). Два PC
  могут быть на разных `at_tick` одновременно (west-marches async).
- **Длинные действия — один event:** `{start_tick, duration_ticks}` вместо тысяч
  строк. Resolver вычисляет завершение. (Первые потребители — крафт 056, даунтайм.)
- **Travel modes:** `Normal ×1.0`, `Cautious ×1.5` на старте; стек модификаторов
  → `time-and-modifiers.md`.
- **Presentation:** `tickToHuman(tick, campaign)` → `{day, hour, minute}`; в схему
  БД не лезет.

## User Scenarios (draft)

### US1 — DM продвигает PC/пачку во времени (P1)
DM завершает вылазку/действие → указывает длительность → clock участников
двигается на `duration_ticks`. UI показывает «День 5, 14:00» или «+90 дней».

### US2 — Длинное действие без итерации (P1)
Крафт/тренировка на 90 дней = один event `duration_ticks=432000`. Никаких
промежуточных строк; resolver знает дату завершения.

### US3 — Разные «сейчас» у разных PC (P2)
Один PC в дне 5, другой в дне 12 — одновременно (async clock). UI «где кто во
времени» (temporal state viewer) — возможно отдельно (spec-035/036).

## Открытые вопросы (объём и порядок — тебе, Andrey)

1. **Масштаб — главное.** Три уровня:
   - **(A) Полная миграция:** `(loop_number, day_in_loop)` → `(loop_id, at_tick)`
     во всей `transactions` + все запросы. Правильный конец, но тяжело + freeze-окно.
   - **(B) Аддитивно:** добавить `at_tick` рядом, dual-write, старое не трогать;
     мигрировать запросы постепенно. Меньше риска, техдолг «две модели времени».
   - **(C) Лайтово:** только per-actor `current_tick` + длинные действия для
     вылазок/крафта, `transactions` не мигрировать вовсе (день остаётся атомом
     бухгалтерии). Минимум, разблокирует 055/056.
   *Предложение Claude:* **(C) сейчас** (разблокирует вылазки/крафт дёшево),
   **(A) — отдельной большой спекой** в рамках пивота, когда дойдёт Группа A/B.
2. **Зависимость от Группы A** (`events`-таблица). Полная модель хочет `events`
   сперва. Лайтовый (C) — не хочет. Порядок с пивотом?
3. **Freeze-окно** миграции `transactions` (если A/B) — приемлемо (~секунды на
   объёме кампании)?
4. **Что реально нужно сейчас?** Похоже — только длинные действия + локальные
   тики для 055/056. Это (C). Полное время — когда пойдёт NPC-movement/энкаунтеры (032/035/036).
5. **`loop` как нода vs `loop_number`.** Роадмап меняет `loop_number:int` →
   `loop_id:uuid`. Делать в этой спеке или отложить?

## Out of Scope (зависит от выбора объёма)

- Локации и traversal (spec-031) — время их использует, но не определяет.
- NPC movement / encounter resolver по тикам (spec-032, Группа C).
- Temporal state viewer «где кто» (spec-035/036).
- Погода/календарь-эффекты (`time-and-modifiers.md`, отдельно).

## Success Criteria (для варианта C — лайтового)

- **SC-001** У PC есть `current_tick`; DM двигает его на длительность действия.
- **SC-002** Длинное действие хранится как один event (`start+duration`), без
  промежуточных строк; resolver даёт дату завершения.
- **SC-003** `tickToHuman` показывает человекочитаемое время; конфиг
  `tick_unit_seconds`/`ticks_per_day` — per-кампания.
- **SC-004** Вылазки (055) и Крафт (056) используют тики без миграции `transactions`.
