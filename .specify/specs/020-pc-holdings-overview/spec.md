# Feature Specification: PC Holdings Overview (live wallet/inventory dashboard для всей кампании)

**Feature Branch**: `020-pc-holdings-overview`
**Created**: 2026-04-28
**Status**: Clarified (2026-04-28)
**Input**: User prompt — «обзор PC холдингов: что у каждого
персонажа в кошельке и инвентаре прямо сейчас в текущей петле,
+ свёрнутый ledger для понимания свежей активности». Параллель
к spec-019 (starter setup overview): тот же DM-дашборд паттерн,
но смотрим **текущее состояние**, а не стартовую конфигурацию.

## Context

Spec-019 закрыла дыру стартового сетапа: одностраничный обзор
+ apply-кнопка на одной странице. Аналогичная дыра существует
для **текущего** состояния: чтобы понять «у кого сейчас сколько
монет и что в инвентаре», DM должен открыть 29 страниц PC одну
за другой, или перейти в `/accounting?pc=<id>` для каждого
по отдельности и сравнивать вручную.

Существующие surfaces:
- `/c/<slug>/catalog/<pcId>` — страница одного PC: `<WalletBlock>`,
  `<InventoryGrid>`, `<TransactionRow>` лента. Отлично работает
  для одного PC, но **не сравнима**: 29 вкладок не вариант.
- `/c/<slug>/accounting?pc=<pcId>` — глобальная лента + actor
  bar; полезно когда DM ищет конкретную транзакцию, но это
  хронологический view, не структурный («что у каждого PC сейчас»).
- `/c/<slug>/accounting/stash` — общак как один актор, со
  своим хедером + ledger. Хорошо работает, **является шаблоном**
  для нашей карточки.

**Дыра**: нет страницы где DM видит «снимок» всех PC одновременно —
кошелёк + инвентарь + последние операции. Это нужно для (а)
быстрого ответа «кто сейчас разорился, кому надо подкинуть»,
(б) подготовки сессии («посмотрел, кто в этой петле активен,
кому что выпадало»), (в) аудита («все ли получили честную
долю с прошлого энкаунтера»).

**Решение**: новая страница `/c/<slug>/accounting/holdings`,
доступная всем участникам кампании (не только DM —
кооперативная кампания, секретов между PC обычно нет; общак
по определению публичный). Стек карточек: stash сверху отдельно,
затем все PC, отсортированные по балансу. PC без активности в
текущей петле — скрыты под disclosure «N персонажей без активности
в этой петле ▾», чтобы не засорять scroll.

## User Scenarios & Testing

### Acceptance Scenarios

1. **DM собирается провести сессию** — открывает
   `/accounting/holdings`. Видит карточку общака сверху, затем
   ~6 карточек активных в текущей петле PC, отсортированных по
   балансу desc. Богатейший — Алексий с 247 gp. Видно цифру.
   Под disclosure скрыты ~23 PC без активности в этой петле.
   Прокрутил карточки активных, увидел у каждого баланс +
   количество предметов. На каждой подпись «42 транзакции в
   этой петле» — прикинул, кто сейчас в гуще движняка.

2. **DM хочет сравнить инвентари двух PC** — раскрывает
   карточку одного, прокручивает, раскрывает второго. Каждая
   карточка содержит inline `<InventoryGrid>` — те же
   предметы которые видны на странице PC, но в одном месте.

3. **DM ищет свежую транзакцию у конкретного PC** —
   тапает «Показать историю» в карточке PC. Lazy-loaded
   list из 10 последних транзакций. Если нужно глубже —
   ссылка «Все транзакции →» ведёт на
   `/accounting?pc=<pcId>` (full ledger UI с пагинацией +
   фильтрами уже есть).

4. **Игрок зашёл проверить — кто сколько у него отнимет** —
   видит ту же страницу что и DM (read-only). Не может
   редактировать чужие транзакции, но видит чужие балансы.
   Поправок к существующим RLS не нужно — все видят всё в
   рамках своей кампании, как и сейчас на других страницах.

5. **PC без активности в текущей петле** — Marina не
   участвовала в последней петле (сессии прошли без неё).
   Её карточка скрыта под disclosure «23 персонажа без
   активности в этой петле ▾» внизу страницы. Если DM
   тапает disclosure — раскрывается список этих карточек
   с тем же шаблоном (баланс, инвентарь, история lazy).

6. **Стартовый сетап применён, но игрок не появился** —
   у Marina есть стартовая монетка через
   `applyPcStarterConfig`, но это не считается активностью
   (по определению стартового сетапа). Marina всё равно в
   disclosure'е — `is_starter_setup=true` транзакции
   игнорируются для флага активности.

7. **Активная петля отсутствует** — DM удалил все loops
   или они все archived. Страница показывает empty state
   «Активной петли нет — создайте петлю на /loops».
   Без петли понятие «текущий стейт» теряет смысл (балансы
   wipe per-loop, FR-015 из spec-010); рендерить нулевые
   цифры было бы вводящим в заблуждение.

### Edge Cases

- **PC удалён в текущей петле**: если PC node был soft-
  deleted (или hard-deleted), не появляется в списке.
  Транзакции с него остаются в ledger'е (исторический
  факт), но карточки нет. Это поведение совпадает с
  тем как `<LedgerActorBar>` уже фильтрует actors.

- **Stash пустой**: если кампания только создана и в общак
  никто ничего не клал — карточка stash всё равно есть
  сверху, но с балансом 0 и empty inventory. Не скрывать —
  это валидный визуальный якорь.

- **Активность есть, но баланс 0**: PC сходил на сессию,
  все деньги потратил. Карточка раскрытая (активный),
  баланс 0 gp. Сортировка по балансу desc положит его
  в конец активных. ОК.

- **Все PC без активности**: можно представить кампанию
  где DM создал 30 PC, прокатил starter setup и петля
  ещё не стартовала (нет сессий). Все 30 карточек —
  под disclosure. Активных — ноль. Над disclosure
  должна быть надпись «В этой петле пока никто не
  активничал». Всё ещё валидное состояние, не error.

- **Performance — 29 lazy histories**: если все 29 DM
  раскроет одновременно, это 29 ledger queries. На
  prod это OK (LedgerPage делает один SQL с pagination,
  ~50 ms × 29 = 1.5 s, не блочит UI). История загружается
  параллельно, спиннеры в карточках. Если кто-то пытается
  это автоматизировать — out of scope, optimize позже.

## Requirements

### Functional Requirements

#### Page structure
- **FR-001**: Новая страница на route
  `/c/<slug>/accounting/holdings`. Sibling к существующим
  `/accounting`, `/accounting/stash`, `/accounting/starter-setup`.
  Как и они, требует `getMembership` (любая роль) и
  редиректит на `/` если нет membership.

- **FR-002**: Заголовок страницы — «Холдинги», подзаголовок
  «Текущее состояние всех персонажей и общака в активной
  петле». Над списком — небольшой блок с loop-context
  (номер петли, день активности): «Петля 4 · день 12»,
  чтобы юзер понимал относительно какого момента считается
  снимок. Day = последний день, по которому есть транзакции
  в этой петле; если нет ни одной — day=1.

- **FR-003**: Под списком — empty state когда активная
  петля отсутствует: «Активной петли нет. Создайте петлю
  на странице /loops, чтобы видеть текущие холдинги.»
  Ссылка `<Link href="/loops">` на ту страницу.

#### Stash card
- **FR-004**: Карточка общака (`stash` node) рендерится
  **сверху** списка. Один и тот же визуальный шаблон что
  у PC карточек, отличается только подписью «Общак»
  (вместо имени PC) и иконой 🏦 (или иной — UI-decision
  на implement phase).

- **FR-005**: Stash-карточка не подчиняется правилу «без
  активности — под кат». Всегда раскрыта, всегда сверху.

#### PC cards — base shape
- **FR-006**: Каждый PC из текущей петли — одна карточка.
  Состав:
  - **Header**: имя PC, аватар (если есть), баланс справа
    (gp + currency mix).
  - **Body** (раскрытая часть): inline `<InventoryGrid>` со
    snapshot инвентаря на текущий день текущей петли.
    Reuse существующего компонента (один-в-один как на
    `/catalog/<pcId>` странице).
  - **History collapsed bar**: подпись «N транзакций
    в этой петле» + кнопка «Показать ▾».
  - **History expanded** (lazy): первые 10 транзакций
    sorted DESC by (loop_number, day_in_loop, created_at);
    каждая — компактная строка ~как в `<LedgerListClient>`
    но без бордеров и без edit/delete actions; ссылка
    «Все транзакции →» ведёт на
    `/c/<slug>/accounting?pc=<pcId>` (full ledger).

#### Active vs hidden cards
- **FR-007**: «Активным в текущей петле» считается PC у
  которого в текущей петле есть **хотя бы одна** транзакция
  (любого kind: money / item / transfer), которая **не
  принадлежит стартовому сетапу**. Стартовые транзакции
  отличаются по полю `is_starter_setup=true` (мигрировано
  spec-012; авто-сет на `applyPcStarterConfig` writes).
  Это включает транзакции совершённые в сессии (sessionId
  IS NOT NULL) и транзакции вне сессии (sessionId IS NULL),
  кроме starter-setup.

- **FR-008**: Активные PC — раскрытыми, в основном списке
  страницы (после stash, перед disclosure'ом неактивных).

- **FR-009**: Неактивные PC — под disclosure'ом
  «N персонажей без активности в этой петле ▾» в самом
  низу страницы. Каждая карточка использует тот же шаблон
  что активные, но default state — collapsed body
  (показывается только header с балансом). Кликом по
  заголовку карточки можно её развернуть (full inventory
  + lazy history).

- **FR-010**: Disclosure «без активности» начально свёрнут.
  Раскрытие — клик по самому disclosure'у — лениво
  фетчит карточки внутри (не on page load) — иначе
  ломаем performance в кампании из 30 PC где активны 5.

#### Sorting
- **FR-011**: Активные PC сортируются по балансу desc
  (богатейшие сверху). Баланс агрегируется по `loop_number =
  current_loop` per `actor_pc_id`, sum across all coin
  denominations to gp-equivalent.

- **FR-012**: Неактивные PC внутри disclosure'а сортируются
  по имени PC asc (тот же ordering что и в spec-019 starter
  overview — устоявшаяся convention).

#### History — lazy + counter
- **FR-013**: Каждая карточка PC заранее знает «N транзакций
  в этой петле» — counter, фетчится на page load одним
  агрегатным запросом для всех PC сразу (`SELECT actor_pc_id,
  COUNT(*) FROM transactions WHERE campaign_id=$1 AND
  loop_number=$2 GROUP BY actor_pc_id`). Это **не считая**
  starter-setup транзакций (для счётчика мы их вычитаем,
  чтобы цифра отражала «реальную» активность).

- **FR-014**: При клике «Показать историю ▾» — fetch первых
  10 транзакций для этого PC через существующий
  `getLedgerPage(campaignId, { actorPcId }, undefined, 10)`.
  Loading state — спиннер внутри карточки. Не блочит другие
  карточки.

- **FR-015**: Раскрытая история не имеет своей пагинации.
  Если транзакций больше 10 — внизу expanded блока
  ссылка «Все транзакции (N) →» на
  `/c/<slug>/accounting?pc=<pcId>`. Глубокая работа с
  историей делается на той странице, где это полноценный
  ledger UI.

#### Permissions
- **FR-016**: Доступ — все members кампании (как FR-001
  гласит), read-only для всех ролей. Инвентари и балансы
  чужих PC видны игроку (это уже так на других страницах
  кампании — кооперативная модель).

- **FR-017**: На странице холдингов **нет inline-edit
  controls** (никаких ✏️ / 🗑️ на транзакциях, никаких
  «+ Доход» кнопок на чужих карточках). Это чисто
  read-only дашборд. Действия делаются на дочерних страницах
  (`/accounting`, `/catalog/<pcId>`), куда холдинги ведут
  ссылками.

#### Non-functional / interaction
- **FR-018**: На мобиле (< 640px) карточки занимают
  всю ширину viewport'а, друг под другом. Header
  карточки прилипает (sticky) сверху во время скролла
  body, чтобы было ясно «в чьей карточке я сейчас».
  Реализация — `position: sticky; top: 0` на header
  внутри своей карточки, а не глобально.

- **FR-019**: Между stash и активными PC, между активными
  и disclosure — простая разделительная горизонтальная
  линия / spacing. Никаких больших title-разделителей,
  чтобы не дробить визуально.

- **FR-020**: Loop-context («Петля 4 · день 12») кликабельный
  — ведёт на `/loops` чтобы юзер мог переключить активную
  петлю. Это глобальная UX convention уже устоявшаяся
  (см. spec-010, balance-hero).

### Key Entities

Никаких новых таблиц. Используем:
- `nodes` (filter by `type='character'`) — список PC.
- `pcs` или эквивалент-материализация — для имени, аватара.
- `transactions` — балансы, инвентарь, история, counter.
  Поле `is_starter_setup` (spec-012) для фильтрации активности.
- `loops` — текущая активная петля (`is_active=true`).
- `getStashNode(campaignId)` — для stash card.

### Reuse — никаких новых компонентов
- `<WalletBalance>` — для рендера баланса в header карточки.
- `<InventoryGrid>` — body карточки.
- `<TransactionRow>` (read-only mode без onEdit/onDelete) —
  history rows. Если read-only mode не поддерживается, маленький
  shim-компонент `<TransactionRowReadonly>` (~30 LOC).
- `getLedgerPage` — fetch истории.

### Read-side — server queries
1. `getActiveLoop(campaignId)` — already exists.
2. `getCharactersForCampaign(campaignId)` — already exists.
3. `getStashNode(campaignId)` — already exists.
4. **NEW:** `getHoldingsSummary(campaignId, loopNumber)` — один SQL,
   возвращает `Array<{ actor_pc_id: string; balance_gp_equivalent:
   number; tx_count_excl_starter: number; last_activity_day:
   number | null }>`. Это позволяет отрендерить header'ы всех
   карточек одним запросом, без N+1.
5. `getInventoryFor(actorPcId, loopNumber, day)` — already
   exists. Вызывается на render каждой PC карточки (29 lookup'ов,
   но они быстрые — индекс по `(actor_pc_id, loop_number,
   item_node_id)`).
6. `getLedgerPage(campaignId, { actorPcId }, undefined, 10)` —
   only when «Показать историю» tapped. Already exists.

#### Open question for Plan phase
- Может ли `getHoldingsSummary` объединить (4) + (5) per-PC
  в один tabular SQL? Или лучше отдельный inventory лукап
  per-PC оставить? Решение в Plan phase.

## Clarifications

### Session 1 (2026-04-28, Q&A perpared in chat 78)

**Q1**: Какой scope инвентаря — current loop / lifetime /
day-picker?
→ **A1**: current loop, current day. Bonus (не MVP): «архив»
с последним стейтом каждой прошлой петли — отложено.

**Q2**: История — eager / lazy / lazy + counter?
→ **A2**: lazy + counter. На свёрнутом виде показываем
«N транзакций», по тапу — fetch first 10.

**Q3**: Stash — отдельной карточкой / в общем стеке /
не показываем?
→ **A3**: отдельной карточкой сверху, всегда раскрытой.

**Q4**: Pagination истории — last 10 / full / none?
→ **A4**: last 10 + ссылка «Все транзакции →» на существующий
`/accounting?pc=<id>`. Двойная пагинация на странице была бы
плохой mental model.

**Q5**: Доступ — DM-only / DM full + player self / all read-only?
→ **A5**: все members read-only, DM full. На странице нет
edit controls, edit делается на дочерних страницах.

**Q6**: Сортировка — по имени / по балансу / toggle?
→ **A6**: по балансу desc. PC без активности в текущей петле —
скрыты под disclosure внизу.

**Q7**: Что есть «активность в текущей петле»?
→ **A7**: PC участвовал в сессии или совершал транзакции вне
сессий, **кроме** starter-setup. Технически: транзакция с
`actor_pc_id=<pc>` AND `loop_number=<current>` AND
`is_starter_setup=false`.

## Success Criteria

1. **SC-001**: DM открывает страницу, видит снимок всех
   активных PC + общак за < 3 секунды на проде.

2. **SC-002**: На странице нет ни одной кнопки которая
   что-то редактирует. Чисто read-only.

3. **SC-003**: PC без активности в текущей петле скрыты под
   disclosure. Disclosure-кнопка показывает корректное
   количество.

4. **SC-004**: При раскрытии истории одного PC, остальные
   карточки не блочатся (parallel fetch / нет global
   loading state).

5. **SC-005**: Карточка общака всегда сверху, всегда раскрыта,
   с балансом и инвентарём общака.

6. **SC-006**: Loop-context «Петля N · день M» отображает
   корректную активную петлю.

7. **SC-007**: Empty state когда нет активной петли — ссылка
   на `/loops` для создания.

8. **SC-008**: На мобиле карточки занимают всю ширину viewport'а,
   header карточки sticky внутри своей карточки.

9. **SC-009**: Сортировка активных по балансу desc корректная
   (тестим: PC с 1000 gp выше PC с 100 gp).

10. **SC-010**: Counter транзакций per-PC исключает
    starter-setup записи.

## Out of Scope

- **Архив прошлых петель** (последний стейт П1, П2, П3
  для ностальгии) — отдельная фича, спек по запросу.
- **Сравнительный mode** (выбрать 2 PC, показать diff
  инвентарей бок о бок) — useful, но scope creep.
- **Графы/тренды** баланса по дням / петлям — это для
  другой страницы (статистика).
- **Inline edit транзакций / inventory bulk operations** —
  делаются на дочерних страницах, не дублируем.
- **Filter / search по PC** — 29 PC помещаются на экран,
  поиск избыточен. Если кампания вырастет до 100+ PC,
  тогда спек.
- **Экспорт в CSV / PDF** — out of scope MVP.
