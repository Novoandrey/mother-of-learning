# Feature Specification: Прозрачные cutout-варианты портретов

**Feature Branch**: `codex/portrait-cutout-variants`
**Created**: 2026-07-21
**Status**: Implement — production delivery authorised
**Epic**: MEDIA-04 follow-up; does not start MEDIA-07 editor UX

## Зачем

Круглый токен на карте сейчас вырезает круг из прямоугольного preview-портрета:
фон персонажа остаётся внутри токена. Прозрачный cutout отделяет фигуру от фона
на сервере один раз и даёт карте готовый слой для отображения. Это улучшает
читаемость карты, не меняя иллюстрацию, не добавляя пользователю редактор и не
создавая новую сущность использования.

## User Story — PC-токен получает прозрачную фигуру (P1)

Участник открывает карту после подготовки cutout для основного портрета своего
персонажа. Токен показывает фигуру с прозрачным фоном; если cutout ещё не
подготовлен или не прошёл проверку, карта продолжает показывать существующий
безопасный preview, а не оригинал и не пустой токен.

**Why this priority**: PC-токен — единственный уже существующий потребитель,
который сразу получает пользу без нового UX или доменной модели.

**Independent Test**: Для связанного с PC asset подготовлен `cutout`; после
reload desktop-карты и Telegram Mini App токен получает PNG cutout. Для asset
без cutout обе поверхности продолжают отображать preview.

## Requirements

- **FR-001**: Cutout MUST быть отдельным immutable rendition `cutout` в
  `media_asset_variants`, формата `image/png`; существующие `thumb`, `preview`
  и `scene` остаются только `image/webp`.
- **FR-002**: Генератор MUST выбирать только портреты с derived `portrait_tag`
  `pc` или `npc`; `--tag pc|npc|all` управляет выбором. `campaign_maps` и их
  R2-объекты никогда не являются входом или целью.
- **FR-003**: Генератор MUST по умолчанию быть dry-run: перечислять план без
  чтения/записи R2 и без изменения БД. Только `--commit` разрешает обработку.
- **FR-004**: Cutout MUST сохранять размер canvas и RGB каждого исходного
  пикселя; модель сегментации вычисляет только alpha. Невалидный результат не
  попадает в R2 или БД и сохраняет QA/debug-артефакты для диагностики.
- **FR-005**: Для одного запуска генератора MUST создаваться ровно одна
  inference session, используемая всем batch.
- **FR-006**: Для успешного результата ключ MUST быть детерминированным и
  ASCII-safe: `media/cutout/<asset-id>-v<version>.png`. Upsert сохраняет
  существующую уникальность `(asset_id, rendition, version)`.
- **FR-007**: Карта MUST предпочитать готовый `cutout` для токена PC и
  корректно fallback-ить к существующему preview. Card/carousel портреты,
  карты-фоны и NPC-токены не меняются.
- **FR-008**: Оригинал, его ключ и существующие preview/thumb/scene не
  перезаписываются; реализация не добавляет crop editor, token prototype,
  поиск/метаданные или отдельную usage-сущность.

## Edge Cases

- У портрета нет `media_asset_id`, asset не готов или отсутствует: он
  пропускается и отражается в отчёте; legacy-оригинал не скачивается.
- Несколько portrait usages ссылаются на один asset: asset планируется один раз.
- У asset уже есть текущий cutout: он пропускается идемпотентно.
- Сегментация, проверка PNG или загрузка R2 завершается ошибкой: БД-вариант не
  создаётся; локальные QA/debug файлы остаются, путь сообщается оператору.
- Cutout ещё не сгенерирован в production: карта продолжает использовать
  существующий preview без изменения поведения.

## Success Criteria

- Подготовленный PC cutout виден на desktop-карте и в Telegram Mini App после
  reload.
- Один запуск без `--commit` не делает сетевых или data-write действий.
- Проверка подтверждает одинаковые source/cutout dimensions и RGB, а также
  alpha с прозрачными и непрозрачными пикселями.
- План/тесты показывают, что map assets не попали в выборку.
