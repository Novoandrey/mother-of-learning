# Feature Specification: Server operations — team access & auto-deploy (spec-028)

**Feature Branch**: `028-access-and-autodeploy`
**Created**: 2026-06-07 (folded auto-deploy in + full Specify, chat 88)
**Status**: US2 auto-deploy **shipped ✓** · US1 access **awaiting SSH keys** ·
US3 Telegram **Specify** (awaiting its own Clarify/Plan/Tasks)
**Input**: Operationalize the production box for the team. Two slices: (1) give
trusted collaborators their own operational access to the box; (2) make pushes to
`main` deploy themselves (Vercel-parity), instead of manual redeploys.

## Context

После 027 приложение целиком на собственном боксе (Dokploy + self-hosted Supabase,
прод = https://theloopers.org). Сегодня коробкой руками управляет только Андрей, и
каждый деплой — ручной `git pull` + redeploy в Dokploy. Эта спека закрывает два
операционных пробела разом: (1) коллабораторы не могут помогать, не гоняя команды
через Андрея; (2) нет push→deploy. Auto-deploy раньше был отдельным стабом (043) —
**сворачиваем обратно сюда (chat 88)** как вторую пользовательскую историю.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Collaborator operational access (Priority: P1)

Лёша (DevOps-советчик), Никита и Сергей (ведущий DM) могут работать с боксом
напрямую под своей личностью — смотреть запущенное приложение, деплоить и смотреть
данные кампании — не прося Андрея ретранслировать команды.

**Why this priority**: разблокирует команду немедленно; самый ценный и самый
дешёвый срез; шипается сам по себе (выдал доступ → готово), независимо от
auto-deploy.

**Independent Test**: коллаборатор с выданным доступом заходит под собственными
кредами, получает шелл на сервере, дашборд деплоя и консоль БД; Андрей может
отозвать одного человека, не затронув остальных.

**Acceptance Scenarios**:

1. **Given** коллаборатор с доступом, **When** он подключается своим ключом,
   **Then** получает шелл под non-root пользователем; общий/root-логин по-прежнему
   отклонён.
2. **Given** коллаборатор с доступом, **When** он открывает дашборд деплоя своим
   аккаунтом (2FA), **Then** может запустить релиз и видит логи + rollback.
3. **Given** коллаборатор с доступом, **When** он открывает консоль БД, **Then**
   видит прод-данные (с задокументированными правилами «обращайся с прод-данными
   аккуратно»).
4. **Given** уходящий коллаборатор, **When** Андрей его отзывает, **Then** доступ
   пропадает <5 мин, остальные коллабораторы и сам Андрей не затронуты, действие
   атрибутируемо.

---

### User Story 2 — Auto-deploy on push to `main` (Priority: P2)

Пуш в `main` автоматически собирается и выкатывается в прод без ручного шага —
возвращает vercel-петлю «смержил → оно в проде».

**Why this priority**: quality-of-life + CI-парити; ценно, но Андрей пока может
редеплоить руками, поэтому идёт после доступа. Шипается независимо.

**Independent Test**: пушим тривиальное изменение в `main`; через несколько минут
прод отражает его без ручного вмешательства; намеренно сломанная сборка прод не
заменяет.

**Acceptance Scenarios**:

1. **Given** настроенный auto-deploy, **When** коммит попадает в `main`, **Then**
   сборка стартует автоматически и при успехе прод отдаёт новую версию в ~5 мин.
2. **Given** пуш, чья сборка падает, **When** сборка ошибается, **Then** прод
   продолжает отдавать последнюю исправную версию (сломанного релиза нет), а сбой
   виден.
3. **Given** любой деплой, **When** он завершился, **Then** он атрибутируем к
   коммиту и предыдущую версию можно вернуть (rollback).
4. **Given** пуш в ветку не-`main`, **When** он попадает, **Then** прод НЕ
   передеплоивается.

---

### User Story 3 — Telegram-уведомления о ветках и мержах (Priority: P3)

> **Статус**: Specify (chat 88, бывш. IDEA-065). Ждёт **своих** Clarify/Plan/Tasks/
> Implement — US1/US2 уже прошли свой цикл, US3 — отдельный срез, берётся позже.

Команда видит в Telegram, когда создана новая ветка и когда что-то влилось в `main`
(= поехал автодеплой), не заходя в GitHub.

**Why this priority**: чистое удобство/видимость; не блокирует ни доступ, ни деплой.
Шипается независимо поверх Actions-инфры из US2.

**Independent Test**: создать ветку и отдельно влить коммит в `main` — в заданном
Telegram-чате появляются два сообщения с автором, веткой/коммитом и ссылкой.

**Acceptance Scenarios**:

1. **Given** настроенные уведомления, **When** создана новая ветка, **Then** в
   Telegram-чат приходит сообщение (ветка, автор, ссылка).
2. **Given** то же, **When** коммит попадает в `main` (мерж/пуш), **Then** в чат
   приходит сообщение (коммит, автор, ссылка) — рядом с автодеплоем US2.
3. **Given** сбой самого нотификатора, **When** Telegram недоступен/ошибка,
   **Then** это НЕ ломает гейт и НЕ блокирует деплой (уведомление — best-effort).

---

### Edge Cases

- Два пуша в `main` подряд — в проде оказывается более поздний коммит; нет
  «полудеплоя».
- Сборка зависла / таймаут — не заклинивает прод; сбой виден, последняя исправная
  версия остаётся.
- Коллаборатор потерял устройство/ключ — отзыв + перевыпуск нового ключа.
- Коллаборатор с урезанным scope (если решим) не может выйти за его пределы.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: У каждого коллаборатора MUST быть индивидуально атрибутируемая
  личность на боксе (не общие аккаунты) — ради аудита и независимого отзыва.
- **FR-002**: Коллабораторы в scope: **Лёша, Никита, Сергей**.
- **FR-003**: Система MUST позволять отозвать любого одного коллаборатора, не
  затронув остальных, за <5 минут.
- **FR-004**: Root- и парольный вход MUST оставаться отключённым (по ключу,
  non-root).
- **FR-005**: Все трое коллабораторов (Лёша, Никита, Сергей) получают
  **одинаковый full-ops доступ**: (a) шелл на сервере (non-root sudo), (b) запуск
  и наблюдение деплоев вкл. rollback, (c) консоль БД. (Clarify chat 88.)
- **FR-006**: Доступ к консоли БД MUST быть задокументирован как полный доступ к
  прод-данным (service-role, обходит RLS); деструктивные/массовые правки — через
  ревью-миграции, не ad-hoc.
- **FR-007**: MUST существовать короткий онбординг-док для коллабораторов (карта
  коробки, как зайти, как деплоить, как открыть консоль БД, правила безопасности)
  + операторские шаги grant/revoke.
- **FR-008**: Пуш в `main` MUST автоматически триггерить сборку и при успехе
  выкатывать в прод без ручного шага.
- **FR-009**: Упавшая сборка MUST NOT заменять запущенный прод; прод остаётся на
  последней исправной версии.
- **FR-010**: Каждый деплой MUST быть атрибутируем к коммиту, предыдущую версию
  MUST можно вернуть (rollback).
- **FR-011**: Пуши в ветки не-`main` MUST NOT передеплоивать прод.
- **FR-012**: Система MUST NOT выкатывать в прод коммит, чей quality gate
  (**lint + tsc + vitest**) красный — в прод доезжает только зелёное.
  (Clarify chat 88.)
- **FR-013**: Прямой внешний доступ к порту БД MUST оставаться закрыт (вне scope
  здесь; read-only путь для Claude — spec-029).
- **FR-014** (US3): При создании новой ветки система SHOULD слать сообщение в
  заданный Telegram-чат (ветка, автор, ссылка).
- **FR-015** (US3): При попадании коммита в `main` (мерж/пуш) система SHOULD слать
  сообщение в Telegram-чат (коммит, автор, ссылка).
- **FR-016** (US3): Сбой/недоступность Telegram-нотификатора MUST NOT блокировать
  гейт или деплой (уведомления — best-effort).

### Key Entities

- **Collaborator**: доверенный человек с атрибутируемой личностью и набором грантов
  доступа (шелл / деплой / консоль БД); независимо отзываем.
- **Deployment**: сборка+релиз приложения, триггеренная коммитом в `main`; имеет
  исходный коммит, исход success/failure, обратима (rollback).

## Success Criteria *(mandatory)*

- **SC-001**: Нового коллаборатора можно завести с полным рабочим доступом за <15
  минут по онбординг-доку.
- **SC-002**: Отзыв любого коллаборатора — <5 минут, остальные (и Андрей) полностью
  рабочие.
- **SC-003**: После пуша в `main` прод отражает изменение за ~5 минут без ручных
  шагов.
- **SC-004**: Упавшая сборка никогда не роняет прод — доступность прода не зависит
  от падений сборки (метрика: 0 сломанных релизов доезжают до пользователей).
- **SC-005**: 100% деплоев атрибутируемы к коммиту и имеют rollback в одно действие.
- **SC-006**: Коммит с красными lint/tsc/vitest никогда не выкатывается в прод
  (метрика: 0 таких релизов).
- **SC-007** (US3): Создание ветки и мерж в `main` дают сообщение в Telegram-чат в
  пределах ~1 минуты; падение нотификатора не влияет на гейт/деплой.

## Assumptions

- Бокс — это боевой прод (spec-027 закрыт); Dokploy — поверхность деплоя; Supabase
  Studio — консоль БД; `main` — релизная ветка.
- Лёша, Никита, Сергей доверенные; персональные учётки приемлемы.
- Доступ всех троих (вкл. Сергея) — одинаковый full-ops; auto-deploy гейтится на
  lint+tsc+vitest. (Решено в Clarify chat 88 — см. `## Clarifications`.)
- Репо — единственный источник правды (коммит/пуш прямо в `main`).

## Out of Scope

- Preview-окружения на каждый PR (vercel-style per-PR URL).
- Автоматизированный multi-env (staging убран на cutover; один env).
- Нотификации в Slack/Discord и другие каналы (Telegram про ветки/мержи — теперь
  **US3**); blue-green / zero-downtime стратегии.
- Пер-человека гранулярные роли в БД (Studio под service-role; пересмотреть, если
  команда вырастет).
- Read-only Postgres-доступ для Claude — это **spec-029**.

## Clarifications

### Session 2026-06-07 (chat 88)

- **Q: Сергей (ведущий DM, не ops) — какой уровень доступа к боксу?**
  **A: Полный ops, как Лёша/Никита** — шелл (non-root sudo) + деплой + консоль БД.

  vitest**; красные проверки блокируют релиз. Отражено в FR-012 (SHOULD→MUST),
  SC-006, US2 AC#2. Механика гейта (вероятно через GitHub Actions, т.к. голый
  Dokploy-webhook места под гейт не даёт) — в Plan (`notes-plan-input.md`).
