# Feature Specification: Медиатека — выбор существующего ассета для портрета

**Feature Branch**: `codex/media-asset-picker`
**Created**: 2026-07-20
**Status**: Implement — production delivery authorised
**Input**: Продолжить эпик медиатеки: MEDIA-04 — ДМ выбирает ранее загруженный
ассет для портрета, а игрок получает только безопасный preview в Telegram App.
**Epic**: `.specify/epics/media-library/constitution.md` (`MEDIA-04`)

## Context

MEDIA-01…03 уже дали кампании общую библиотеку, варианты и перенесли legacy
портреты. Но `character_portraits` всё ещё владеет строкой `r2_key`, а Mini App
строит URL оригинала напрямую. Поэтому библиотечный ассет нельзя назначить
портретом без повторной загрузки, а игрокская поверхность обходит safe-delivery
контракт MEDIA-02.

MEDIA-04 делает первое настоящее **usage** общей медиатеки: портрет ссылается
на существующий ассет. Одновременно она фиксирует шов для следующих
потребителей: круглый портрет и боевой токен используют один исходный asset,
но имеют независимые настройки использования. Она не добавляет cropper или
редактор токена и не вводит видимость по ролям.

## Target usage model

Модель намеренно разделяет неизменяемый файл и его появления в игре:

1. **Media asset** — исходник и immutable derived variants, самостоятельная
   сущность библиотеки.
2. **Portrait usage** — строка портретной карусели ноды; ссылается на asset,
   хранит primary, порядок и будущий круглый crop.
3. **Token prototype** *(будущий MEDIA-07/эпик сцен)* — дефолт внешнего вида
   токена для ноды; может сослаться на тот же asset, но хранит независимый
   token crop/configuration.
4. **Placed map token** *(будущий эпик сцен)* — конкретный экземпляр token
   prototype на карте. После размещения он не обязан изменяться вслед за
   прототипом; массовое обновление требует отдельного явного действия.

MEDIA-04 реализует только второй уровень и контракт его безопасного чтения.
Она не создаёт token prototype или placed-token UI, но не может закладывать
зависимость, где token навсегда читает портретный `r2_key` или его crop.

## User Scenarios & Testing

### User Story 1 — Участник назначает портрет из медиатеки (Priority: P1)

Участник с уже существующим правом редактировать ноду открывает карточку
персонажа, НПС или существа, запускает выбор из уже
загруженных ассетов кампании, выбирает готовое изображение и сохраняет его как
новый портрет в существующей карусели. После reload в карточке остаётся тот же
портрет, а исходный файл не дублируется.

**Why this priority**: Это впервые превращает библиотеку в общий источник для
существующего потребителя вместо ещё одного загрузчика файлов.

**Independent Test**: Участник выбирает один из 140 production assets для НПС,
перезагружает страницу ноды и видит этот же портрет; число `media_assets` и
объектов R2 не меняется.

**Acceptance Scenarios**:

1. **Given** участник состоит в кампании, может редактировать ноду и у неё уже
   есть портреты, **When** он
   выбирает готовый ассет и подтверждает, **Then** в карусели появляется новый
   usage с этим asset, а ассет остаётся единственной библиотечной записью.
2. **Given** ассет ещё обрабатывается или обработка завершилась ошибкой,
   **When** участник открывает выбор, **Then** состояние понятно показано и такой
   ассет нельзя назначить как готовый портрет.
3. **Given** пользователь не может редактировать ноду, **When** он пытается
   сохранить выбор вручную, **Then** сервер не создаёт usage.
4. **Given** два запроса пытаются назначить тот же asset той же ноде,
   **When** они доходят до сохранения одновременно, **Then** остаётся ровно
   один portrait usage.

---

### User Story 2 — Игрок читает безопасный портрет в Telegram (Priority: P1)

Игрок открывает Mini App после назначения портрета и видит его в списке
персонажей и/или в карточке каталога. Клиент получает только rendition
`preview` (или меньший `thumb` там, где он достаточен), никогда не получает
`media_assets.storage_key`, legacy `r2_key` или URL оригинала.

**Why this priority**: Player/TG путь — обязательный критерий готовности
эпиков A–F; без него picker остаётся только вебовой заготовкой.

**Independent Test**: Участник кампании открывает `/tg` на телефоне после
reload, видит назначенный портрет, а network/read model содержит ключ preview
варианта, но не ключ оригинала.

**Acceptance Scenarios**:

1. **Given** назначенный ассет имеет готовый preview, **When** игрок открывает
   карточку соответствующей ноды в `/tg`, **Then** отображается изображение
   preview-качества и не загружается оригинал.
2. **Given** asset usage принадлежит другой кампании или участник не состоит в
   кампании, **When** он выполняет TG-запрос, **Then** usage и rendition не
   раскрываются.
3. **Given** у legacy-портрета ещё нет связанного library asset, **When**
   игрок открывает TG, **Then** интерфейс показывает безопасный placeholder,
   а не legacy-original; оператор получает этот случай в отчёте backfill.

## Edge Cases

- У ноды нет портретов: выбор создаёт первый и делает его основным по текущему
  правилу карусели.
- Один asset можно назначить нескольким нодам, но повторный выбор того же asset
  для той же ноды не создаёт дублирующий portrait usage.
- Отменённый picker и сетевой сбой не создают usage.
- Удаление usage не удаляет asset и R2-объекты (MEDIA-05 решит их жизненный
  цикл отдельно).
- Временно недоступный preview и legacy-портрет без сопоставления показывают
  placeholder/status в TG, а не оригинал.
- Смена будущего token prototype не меняет уже поставленные на карту токены
  без отдельного явного действия.

## Requirements

### Functional Requirements

- **FR-001**: Портрет MUST хранить явную ссылку на `media_assets` той же
  кампании как источник изображения; `r2_key` не является источником для
  новых назначений.
- **FR-002**: Система MUST сохранить совместимость с уже существующими
  portrait rows и выполнить безопасный, проверяемый backfill по совпадению
  кампании и original storage key для legacy-портретов, которые уже
  сопоставлены с библиотечными ассетами. Backfill MUST выдать отдельный отчёт
  о каждой несопоставленной строке и не создавать R2-объекты.
- **FR-003**: Редактор portrait-потребителя MUST позволять выбрать готовый
  существующий asset кампании и сохранить его без загрузки либо копирования
  файла.
- **FR-004**: Сохранение usage MUST повторять существующую авторизацию
  `canEditNode` (без нового ролевого барьера «только ДМ»), проверять
  принадлежность ноды и ассета одной кампании, не принимать произвольный R2
  key от клиента и не создавать второй usage того же asset для той же ноды.
  Последний инвариант MUST быть обеспечен ограничением данных, не только UI.
- **FR-005**: Telegram read model MUST получать изображение назначенного
  library usage через единый consumer-safe delivery contract: `thumb` для
  компактного списка и `preview` для карточки/карусели. Original storage key
  и его URL MUST NOT попасть в DTO или сетевой ответ.
- **FR-006**: Вебовая карточка и `/tg` MUST корректно отображать статусы
  pending/failed variant без fallback на оригинал.
- **FR-007**: Успешное назначение MUST переживать reload на desktop и в Mini
  App и оставлять текущие порядок, primary-статус и crop-поля портретов
  нетронутыми.
- **FR-008**: MEDIA-04 MUST NOT добавлять редактор кадрирования, token
  prototype/placed-token UI, удаление media asset, поиск/теги, категории или
  visibility-политику.
- **FR-009**: Текущая загрузка портрета MUST сохраниться как быстрый ingestion
  путь: она создаёт library asset, а его последующее назначение использует тот
  же portrait-usage механизм, что и picker.
- **FR-010**: После backfill `media_asset_id` — единственный источник
  изображения для новых portrait usages. Legacy `r2_key` может существовать
  только как временный desktop-read fallback для несопоставленной исторической
  строки и MUST NOT использоваться в `/tg`.
- **FR-011**: Plan MUST определить projection/route, которым browser-клиент
  Mini App получает готовый rendition URL, не джойня самостоятельно
  `media_assets.storage_key` или `media_asset_variants.storage_key`.
- **FR-012**: В будущих usage-путях token prototype и placed map token MUST
  ссылаться на asset/его собственный configuration layer, а не на legacy key
  либо на portrait crop. Изменение prototype MUST NOT неявно менять уже
  размещённые map tokens.
- **FR-013**: Новая связь portrait usage → asset MUST быть пригодна для
  будущей проверки использования в MEDIA-05; удаление asset не входит в эту
  спеку и не должно каскадно удалять portrait usage скрыто.

### Key Entities

- **Media asset usage (portrait)**: запись карусели конкретной ноды, которая
  ссылается на один library asset и хранит только настройки использования
  (primary, порядок, будущий crop).
- **Safe portrait rendition**: текущий `thumb` или `preview` производный
  вариант asset, разрешённый участнику кампании и достаточный конкретной
  поверхности.
- **Legacy portrait**: существующая portrait row с историческим `r2_key`,
  сохраняемая до успешного сопоставления с library asset; в `/tg` без
  сопоставления отображается placeholder.
- **Token prototype / placed token**: будущие раздельные usage-слои. Они
  ссылаются на один asset или собственный override и не являются копией
  portrait usage.

## Success Criteria

- **SC-001**: ДМ назначает готовый library asset портретом без новой R2
  загрузки и снова видит его после reload.
- **SC-002**: Игрок-участник видит назначенный портрет в `/tg` после reload;
  его read model и загрузка изображений не содержат original key/URL.
- **SC-003**: Backfill не меняет ID, порядок, primary или crop ни одного
  существующего portrait usage, выдаёт отчёт о несопоставленных строках и не
  создаёт новые объекты в R2.
- **SC-004**: Попытка назначить asset другой кампании или без права на ноду
  завершается отказом без новой строки usage.
- **SC-005**: Два параллельных назначения одного asset одной ноде оставляют
  ровно один usage; этот факт проверен интеграционным или database-level
  тестом.

## Assumptions

- `media_assets` и все 140 production variant jobs остаются источником
  доступных для выбора изображений; отдельного телеграмного хранилища нет.
- Текущая membership-модель применяется всем участникам кампании до
  отдельного visibility-эпика.
- Потребитель MEDIA-04 — только существующий portrait carousel; карта, сцена
  и токен станут отдельными usage-путями.

## Clarifications

- **OQ-001 — RESOLVED (2026-07-20)**: назначить asset может любой участник,
  которому `canEditNode` уже разрешает изменять эту ноду; отдельная роль ДМа
  не вводится. Выбор добавляет портрет в карусель, но не дублирует уже
  назначенный той же ноде asset.
- **OQ-002 — RESOLVED (2026-07-20)**: кнопка «Загрузить» остаётся рядом с
  picker как быстрый ingestion-путь.
- **OQ-003 — RESOLVED (2026-07-20)**: `r2_key` остаётся временным read-fallback
  для legacy-строк, которые backfill не сумел сопоставить; новые usages всегда
  получают `media_asset_id`. В `/tg` legacy fallback запрещён: несопоставленный
  portrait показывает placeholder до операторского исправления.
- **OQ-004 — RESOLVED (2026-07-20)**: portrait, token prototype и placed map
  token — разные usage-уровни одного asset. MEDIA-04 реализует portrait; token
  prototype будет иметь независимый crop/configuration; уже размещённые токены
  не обновляются неявно вслед за prototype.
