# Feature Specification: Self-hosted Supabase (своя инфра, срез 2/5)

**Feature Branch**: `024-self-hosted-supabase`
**Created**: 2026-06-03
**Status**: ✅ Clarify complete (2026-06-03, Session 1: Q1–Q4 resolved) — awaiting Plan
**Input**: Продолжение эпика «Переезд на свою инфру». 023 (фундамент:
бокс + Dokploy + SSL + staging-деплой) **готов и в проде**. Следующий
шаг — поднять на том же боксе **self-hosted Supabase**, чтобы потом (026)
перевезти на него данные без переписывания кода приложения.
Это **срез 2/5** эпика (023 фундамент → **024 self-hosted Supabase** →
025 бэкапы+drill → 026 миграция данных+auth → 027 cutover).

## Context

### Зачем этот срез

Аудит (см. 023) показал: приложение опирается только на **Postgres +
Auth (GoTrue) + PostgREST** плюс RLS (76 политик, `auth.uid()` ×22) и
RPC/триггеры. По аудиту 023 **не используются**: Storage, Realtime, Edge
Functions — но прежде чем вырезать Realtime/Edge, в 024 это
**перепроверяется** на боевом коде (аудит мог чего-то не увидеть); если
использование под вопросом — продукт оставляем, не режем вслепую.
Дроп-ин путь без переписывания кода — поднять у себя **обрезанный**
self-hosted Supabase-стек, идентичный по продуктам тому, что реально
нужно. 023 дал фундамент (защищённый бокс, Dokploy, авто-SSL, доказанный
git-деплой). 024 ставит на этот фундамент сам Supabase — **пустой,
здоровый, параллельный проду** инстанс.

### Роль этого среза (2/5)

024 — встать рядом с продом, не трогая его. Риск всё ещё низкий: **ни
реальных данных, ни прод-трафика, ни гашения managed-инстанса**. Цель —
получить рабочий self-hosted Supabase и **доказать паритет** (расширения,
схемы, версия Postgres) с прод-инстансом, чтобы будущая миграция данных
(026, `pg_restore`) не упала на отсутствующем расширении. Это совет
Леши: проверять совместимость на пустом стеке заранее, до того как
поедут реальные данные.

Бокс уже готов: тот же Hetzner **CPX32** (4 vCPU AMD / 8 ГБ / 40 ГБ SSD,
Helsinki) из 023, отресайженный с CX23 под Supabase, +2 ГБ swap. Новый
сервер не провижится.

### География

Без изменений к 023: мать учения хостится **за рубежом** (EU/Hetzner),
вне юрисдикции РФ. Проекты с ПД граждан РФ — на отдельном российском боксе
(152-ФЗ), вне этого эпика. Self-hosted инстанс в 024 живёт на зарубежном
боксе.

### Делёж труда (важно)

Как и в 023, **исполнение — на боксе оператора, не в песочнице Claude**:

- **Claude поставляет**: эту спеку, `plan.md`/runbook (compose-правки,
  команды, конфиги, чек-листы), скрипты сверки.
- **Оператор (Andrey + Леша) исполняет**: разворачивает стек в Dokploy,
  правит DNS, гоняет команды, присылает логи. Claude разбирает ошибки и
  правит артефакты.

## Operator Scenarios & Testing

### US1 — Обрезанный Supabase-стек работает (P1)

Оператор поднимает self-hosted Supabase, содержащий **только** продукты,
которые приложение реально использует, без лишних сервисов.

#### Acceptance Scenarios

1. **Ядро здорово** — Postgres, Auth (GoTrue), PostgREST, API-gateway
   (Kong), Studio и postgres-meta (зависимость Studio) запущены и в
   состоянии healthy.
2. **Лишнее отсутствует — после подтверждения** — Storage, Realtime,
   Edge Functions, imgproxy, analytics/Vector не запущены; исключение
   каждого подтверждено отсутствием использования в приложении. Realtime
   и Edge Functions перепроверяются отдельно — если использование под
   вопросом, продукт **оставляется** в стеке до выяснения.
3. **Пусто и параллельно** — БД содержит только дефолтные служебные схемы
   Supabase, без данных приложения; прод managed-Supabase и приложение не
   затронуты.
4. **Переживает reboot** — после `reboot` бокса стек поднимается сам.

#### Edge Cases

- **Бюджет RAM** — обрезанный стек + Dokploy + уже задеплоенное staging-
  приложение (023) должны влезать в 8 ГБ + 2 ГБ swap; если нет — заранее
  решено, что приоритезируем / гасим.
- **Мажор Postgres** — версия PG в self-hosted стеке должна быть
  совместима с прод-инстансом (понадобится для `pg_restore` в 026); если
  прод на другом мажоре — учтено до 026.
- **Использование Realtime/Edge неоднозначно** — если перепроверка не
  даёт чёткого «не используется», продукт остаётся в стеке, решение об
  исключении откладывается (лучше лишний контейнер, чем урезанный
  функционал).

### US2 — Studio за HTTPS и проверенный API, безопасно (P1)

Оператор управляет новым инстансом через Studio по HTTPS и проверяет
работоспособность API-gateway — оба защищены, порт Postgres наружу закрыт.

#### Acceptance Scenarios

1. **Studio по HTTPS** на (под)домене за авторизацией (кандидат —
   `db.theloopers.org`), не по голому IP, не анонимно.
2. **API-gateway проверен изнутри** — Auth/REST отвечают из Docker-сети /
   с бокса; наружу gateway в 024 **не публикуется** (публичный HTTPS — на
   cutover 027). Прод-трафик в 024 не переключаем.
3. **Postgres наружу закрыт** — доступ к БД только с бокса / внутри
   Docker-сети; firewall из 023 держит 5432 закрытым извне.
4. **Свежие секреты** — JWT secret, anon/service-ключи, пароль Postgres,
   логин Studio сгенерированы заново, отличны от прод-инстанса, и не
   хранятся в git.

#### Edge Cases

- **CVE дашборда/gateway** — Studio/Kong держать пропатченными (как
  Dokploy в 023).
- **Конфликт доменов/портов** с уже задеплоенным `staging.theloopers.org`
  (023) на том же боксе — учтён при маршрутизации.

### US3 — Паритет расширений/схем с продом (P1)

До того как хоть какие-то данные поедут (026), оператор доказывает, что
пустой инстанс имеет **все** расширения и схемы Postgres, что есть в
прод-инстансе — чтобы будущий `pg_restore` не упал на недостающем.

#### Acceptance Scenarios

1. **Сверка расширений** — `\dx` на self-hosted сверён с прод; список
   совпадает либо недостающие явно зафиксированы (pgcrypto, uuid-ossp,
   pgjwt, pg_graphql, pgsodium, …).
2. **Сверка схем** — `\dn` на self-hosted сверён с прод (auth, storage,
   extensions, graphql, realtime, …).
3. **Доустановка недостающего** — всё, чего нет в self-hosted,
   доустановлено/включено **до** 026; результат повторно сверён.
4. **Зафиксированный отчёт** — diff расширений/схем сохранён (в
   runbook/чате) как пруф готовности к 026.

#### Edge Cases

- **Расширение есть в прод, но недоступно в образе** self-hosted — найден
  способ (другой образ / `CREATE EXTENSION` / сборка) либо зафиксировано
  как блокер для 026.

## Requirements

> WHAT/acceptance. Конкретные compose-правки, команды, выбор образов и
> доменов — в `plan.md`/runbook (фаза Plan). Названные ниже инструменты
> (`db.theloopers.org`, Kong, Studio) — кандидаты для решения в

### Стек (обрезанный)

- **FR-001**: На боксе из 023 через Dokploy поднят self-hosted Supabase-
  стек, содержащий только используемые приложением продукты: Postgres,
  Auth (GoTrue), PostgREST, API-gateway (Kong), Studio — плюс postgres-meta
  (обязательная зависимость Studio для SQL/table-редактора).
- **FR-002**: Продукты, которые приложение не использует — Storage,
  Realtime, Edge Functions, imgproxy, analytics/Vector — исключаются из
  стека **только после подтверждения**, что приложение их не задействует.
  Realtime и Edge Functions подтверждаются отдельно (повторная проверка
  использования на боевом коде); при неоднозначности продукт остаётся.
- **FR-003**: Инстанс пустой и параллельный проду: реальные данные не
  переносятся; прод managed-Supabase и приложение (`staging.theloopers.org`)
  не затрагиваются.
- **FR-004**: Стек переживает `reboot` (контейнеры стартуют сами).

### Доступ и безопасность

- **FR-005**: Studio доступен только по **аутентифицированному
  шифрованному каналу** — либо публичный HTTPS на (под)домене за
  авторизацией (кандидат — `db.theloopers.org`), либо SSH-туннель (как
  панель Dokploy в 023). Не по голому IP, не анонимно. Какой из двух —
  решает оператор при реализации.
- **FR-006**: Работоспособность API-gateway (Kong → GoTrue/PostgREST)
  подтверждена **изнутри** — проверками из Docker-сети / с бокса. Наружу
  gateway в 024 **не публикуется**; публичный HTTPS-доступ к API
  настраивается на cutover (027). Прод-трафик в 024 не переключается.
- **FR-007**: Порт Postgres наружу закрыт; доступ к БД — только с бокса /
  внутри Docker-сети. ⚠️ Не полагаемся только на ufw из 023: там Docker
  уже обошёл ufw (порт 3000, погашен `--publish-rm`) — поэтому в 024
  проверяем, не публикует ли compose 5432 (и прочие внутренние порты)
  наружу, и при необходимости гасим тем же приёмом.
- **FR-008**: Секреты инстанса (JWT secret, anon/service-ключи, пароль
  Postgres, логин Studio) сгенерированы свежими, отличны от прод-инстанса,
  и не хранятся в git.

### Паритет с продом (дериск 026)

- **FR-009**: Набор расширений Postgres (`\dx`) self-hosted инстанса
  сверен с прод-инстансом; недостающие доустановлены до 026.
- **FR-010**: Набор схем (`\dn`) self-hosted инстанса сверен с прод;
  расхождения устранены или явно зафиксированы.
- **FR-011**: Версия Postgres self-hosted стека совместима с прод-
  инстансом для будущего `pg_restore` (026); несовместимость зафиксирована
  как блокер.

### Границы

- **FR-012**: Этот срез **не** переносит схему/данные/`auth.users` (026),
  **не** настраивает авто-бэкапы/restore-drill (025), **не** переключает
  прод-трафик и **не** гасит managed-Supabase (027). Только пустой
  здоровый параллельный инстанс + доказанный паритет.

## Clarifications

### Session 1 (2026-06-03)

**Q2: API-gateway — публиковать наружу (A) или проверять изнутри (B)?**
→ **B.** В 024 приложение всё ещё смотрит на managed-Supabase, внешнего
потребителя у нового API нет — публиковать пустой инстанс на общем боксе
незачем (лишняя поверхность атаки). Gateway (Kong → GoTrue/PostgREST)
проверяется изнутри Docker-сети / с бокса. Публичный HTTPS-доступ к API
настраивается на cutover (027). Studio остаётся наружу по HTTPS+auth
(US2). FR-006 / US2-AS2 / SC-003 / SC-005 приведены к варианту B.

**Q3: Какой мажор Postgres у прода и под что пинимся?**
→ **PG 17**, прод-сборка **`17.6.1.104`** (по Dashboard → Infrastructure →
Service Versions). Self-hosted `supabase/postgres` пиним в ту же сборку
(`17.6.1.104`), при недоступности тега — ближайшую `17.6.1.x` той же
линии. Совпадение один-в-один минимизирует дрейф расширений/каталогов для
`pg_restore` в 026. **Прод не апгрейдим** до доступной `17.6.1.127` —
self-hosted догоняет прод, не наоборот; новее поднимем обе стороны в
локстепе позже. Сопутствующие прод-версии (для пиннинга образов в Plan):
GoTrue/Auth `2.189.0`, PostgREST `14.5`.

**Q1: Стек — официальный обрезанный compose или ручной?**
→ **A: официальный `supabase/docker` compose, обрезанный** (неиспользуемые
сервисы удалены — с подтверждением, FR-002), заведён в Dokploy как
Docker-Compose-приложение. Официальный compose уже корректно сшивает
Kong-роуты, GoTrue↔Postgres, роли PostgREST, проброс JWT-секрета,
Studio↔meta — ручная пересборка этого слоя ровно то, что ломает auth/RLS.
Бонус — совместимость с апстримом для будущих апдейтов.

**Q4: Где хранить секреты?**
→ **A: env-переменные приложения в Dokploy** (шифруются в его БД). Имена
секретов и команды генерации (`openssl rand` и т.п.) — в runbook; значения
в git не попадают. Отдельный secret-store (Vault/Doppler) для соло-
однобоксового сетапа — оверкилл. Секреты генерируются свежими, не
копируются с прода (FR-008).

**Clarify-раунд закрыт (Q1–Q4 решены).** Дальше — Plan по запросу.

## Success Criteria

1. **SC-001**: `docker ps` / Dokploy показывает Postgres, GoTrue,
   PostgREST, Kong, Studio, postgres-meta как healthy; Storage/Realtime/
   Edge/imgproxy/Vector отсутствуют.
2. **SC-002**: Studio доступен только аутентифицированно и шифрованно по
   выбранному пути — **(публичный)** HTTPS на (под)домене с валидным
   сертификатом + авторизация, **либо** через SSH-туннель (наружу не
   опубликован).
3. **SC-003**: API-gateway подтверждён изнутри — Auth- и REST-эндпоинты
   отвечают из Docker-сети / с бокса; наружу не опубликованы (публичный
   HTTPS-доступ к API — на cutover 027).
4. **SC-004**: Скан портов снаружи: 5432 закрыт; открыты только SSH +
   80/443 (как в 023).
5. **SC-005**: После `reboot` бокса стек поднимается сам, и подтверждено:
   - все целевые контейнеры (Postgres, GoTrue, PostgREST, Kong, Studio) —
     healthy;
   - Studio доступен;
   - Auth endpoint отвечает;
   - REST endpoint отвечает
     (Auth/REST — изнутри Docker-сети / с бокса; наружу в 024 не
     публикуются);
   - Postgres отвечает изнутри Docker-сети / с бокса;
   - порт 5432 снаружи по-прежнему закрыт.
6. **SC-006**: `\dx` self-hosted ⊇ `\dx` прод (diff пустой либо
   зафиксирован); `\dn` сверены **по scope будущего restore** — in-scope
   схемы присутствуют, out-of-scope расхождения (схемы неиспользуемых
   продуктов) классифицированы и зафиксированы в `parity-report.md` (не
   строгое равенство).
7. **SC-007**: Версия Postgres self-hosted совместима с прод; зафиксирована
   в runbook.
8. **SC-008**: Прод managed-Supabase и `staging.theloopers.org` (023)
   продолжают работать — новый инстанс на них не повлиял.

## Out of Scope

- **Перенос схемы/данных/`auth.users`** — срез 026.
- **Авто-бэкапы и restore-drill** — срез 025.
- **Cutover (переключение env приложения) и гашение managed Supabase** —
  срез 027.
- **Storage / Realtime / Edge Functions** — по аудиту приложением не
  используются и в стек не входят **при подтверждении** (см. US1/FR-002);
  если перепроверка выявит использование — продукт остаётся в стеке.
- **Cloudflare R2 / картинки** — вне эпика.
- **Observability / мониторинг** (метрики, дашборды, алертинг, агрегация
  логов) — **не в 024**. Здесь только базовая живость: healthy-контейнеры
  (SC-001) + переживание `reboot` (SC-005). Полноценная observability —
  срез 025 (ops) или позже, когда на инстансе появятся реальные данные.
- **Тюнинг Postgres под нагрузку, репликация, HA** — по запросу позже;
  здесь достаточно рабочего одиночного инстанса.
