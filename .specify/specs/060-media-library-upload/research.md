# Research: MEDIA-01 — загрузка и библиотека

## R1. Центральная запись ассета

**Decision**: каждое новое изображение получает запись `media_assets`, а ключ
объекта остаётся техническим атрибутом этой записи.

**Rationale**: R2 умеет хранить байты, но не является моделью поиска,
происхождения, связей и жизненного цикла. Стабильный UUID ассета позволит
последующим спекам ссылаться на изображение из портретов, карт и фонов.

**Alternatives considered**:

- Продолжить хранить ключ непосредственно в доменных таблицах — сохраняет
  дублирование и не даёт общей библиотеки.
- Сделать каждый ассет нодой — перегружает граф техническими объектами и
  смешивает игровые сущности с файлами.

## R2. Серверная прокси-загрузка

**Decision**: MEDIA-01 принимает файл через защищённый Route Handler и пишет его
в текущее S3-совместимое хранилище серверными credentials.

**Rationale**: этот путь уже работает для портретов и карт, лимит 12 МБ мал для
прокси-загрузки, а credentials не попадают в браузер. Presigned upload имеет
смысл при больших пакетах и относится к MEDIA-06.

**Alternatives considered**:

- Presigned URL прямо в браузер — сложнее завершать/откатывать запись и не нужен
  для одиночной загрузки до 12 МБ.
- Supabase Storage — намеренно исключён из self-hosted стека; включать новый
  сервис только ради этой спеки противоречит простоте стека.

## R3. Согласованность двух хранилищ

**Decision**: сначала загрузить объект, затем создать запись. Если запись не
создалась, выполнить best-effort компенсирующее удаление объекта.

**Rationale**: карточка не должна появиться до успешной записи данных. Обратный
порядок создаёт видимый ассет без файла; выбранный порядок с компенсацией
локализует отказ.

**Alternative considered**: фоновая уборка сирот. Она всё равно пригодится
позже, но не заменяет немедленную компенсацию в коротком синхронном пути.

## R4. Права и видимость

**Decision**: owner/DM загружают; все участники кампании читают; RLS отсекает
другие кампании. Индивидуального visibility-поля в MEDIA-01 нет.

**Rationale**: это подтверждённый текущий режим продукта. Будущая видимость
принадлежит использованию ассета или его владельцу, а не исходному файлу.

**Alternative considered**: добавить `visibility` заранее. Поле без работающей
серверной политики создаёт ложное обещание безопасности.

## R5. Выдача изображения

**Decision**: строить URL через общий публичный base URL с сохранением
совместимости с текущей переменной портретов; карточка использует обычный
лениво загружаемый `<img>`.

**Rationale**: текущий R2 bucket уже public-read, а его домен задаётся во время
сборки. Для `next/image` потребовалась бы отдельная динамическая конфигурация
remote patterns и известные размеры, чего MEDIA-01 не собирает.

**Deferred**: эпик Visibility должен отдельно решить private delivery/signed
URLs, прежде чем скрытые изображения станут продуктовым обещанием.

## R6. Страница и обновление

**Decision**: Server Component читает данные напрямую через Supabase/RLS;
Client Component отвечает только за форму, сообщения и `router.refresh()`.

**Rationale**: соответствует Next.js 16 и текущим страницам проекта, не создаёт
лишний GET API и не передаёт service-role клиенту.

## R7. Заимствования из Odysseus

**Decision**: взять централизацию валидации, лимитов и атомарного завершения;
не переносить gallery editor, draft model и локальную файловую архитектуру.

**Rationale**: Odysseus подтверждает правильность разделения upload/library/
editor, но MEDIA-01 реализует только первые два шага пользовательского пути.
