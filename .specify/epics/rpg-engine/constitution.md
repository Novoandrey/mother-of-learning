## Зачем эпик

Данные chat 90: у продукта есть поверхности данных, но нет поверхности
*игры* — 0 из 31 листа правились за месяц, чарники живут в таблице Стасяна.
Эпик переносит ежесессионный артефакт — чарлист — в продукт и строит под ним
универсальный механизм: персонаж = пирамида модулей, модуль декларирует, что
он делает. Одна система вместо хардкода категорий (спеллы/фиты/предметы/…).

## Принципы

### E1. UX диктует данные

Каждая спека эпика проектируется от поверхности использования: сценарии
игрока → модель данных → код. Ни одна спека не шипится «только бэкендом» —
у движка тоже есть лицо (карточка модуля, добавление, разложение чисел).
Новые поверхности проходят design-проход до Plan (флоу spec-022/design.md).

### E2. Одна система

Раса, классовая фича, фит, предмет, спелл, бафф, факультатив, заметка —
один механизм «модуль». Никакого пер-категорийного хардкода; новая категория
контента = данные, не код (Конституция IX).

### E3. Текст — первоклассный гражданин

Имя + свободный текст + источник — полноценный модуль навсегда. Структура
(эффекты, ресурс) добавляется по желанию, никогда не требуется. Никаких стен
валидации; ничего не теряется молча. Сложное может остаться текстом навсегда.

### E4. Прозрачность по умолчанию

Любой участник кампании видит любой чарлист **целиком, всегда** и добывает
его легко (список всех PC — главная поверхность навигации, лист — в пару
тапов от входа). Тайн в чарлистах нет; тайны мира — отдельная история
(spec-033) и к модулям PC не относится. Правка — по-прежнему по правам
(owner/DM, `canEditNode`); прозрачность — про чтение.

### E5. Инспектируемость

Любой модуль — свой, чужой, канонный, форкнутый — открывается в карточку,
которая человеческим языком говорит, что он делает: текст, эффекты
(«+1 к КД», «преимущество на Скрытность»), ресурс с остатком, происхождение
(канон / форк-от / автор). Любое вычисленное число раскладывается на
слагаемые; каждое слагаемое ведёт в карточку модуля-источника.
«Почему КД 18?» — ответ в один тап из листа.

### E6. Добавление дешевле бумаги

Записать новое свойство/предмет/спелл — быстрее, чем карандашом на бумаге:
текстом, за секунды, с телефона, без обязательных полей кроме имени.
Прикрепить существующее (канон-предмет, спелл) — поиск + тап. Слежение там
же: остатки ресурсов видны и правятся в один тап везде, где модуль виден.

### E7. Одна правда, реалтайм у всех

Состояние листа живёт в одном месте, и правка видна всем **в реальном
времени**: открытые листы и карточки обновляются пушем, без действий
пользователя — игрок тратит Ци, ДМ видит это на лету. Ревалидация при
фокусе/реконнекте — слой устойчивости (мобильные сессии засыпают), а не
замена пуша. Конфликты — last-write-wins по полю + честный откат с тостом,
не тихий мерж. Таблица Стасяна — стартовая точка через импорт (one-way);
после переезда PC источник правды — приложение, параллельного канала нет.

*Инфра-пререквизит*: Realtime-сервис был вырезан из самохост-стека как
неиспользуемый (spec-024/T009); возвращение (контейнер + websocket-маршрут
+ авторизация каналов) — часть Plan spec-044, проверяется на staging.

### E8. Канон прибит, хоумбрю — через форк

Канон-ноды read-only для всех, кроме owner (правка owner'ом — in-place, без
версий, v1). Изменить канон под себя = форк: полная копия с `forked_from`,
апстрим не тянется. Хоумбрю — норма, не исключение («Ильза», «ГМ дал»,
обрезанные фиты факультативов).

### E9. Слой-0 — формулы Стасяна

Базовая деривация = канон таблицы (`memory/character-sheet-excel-system.md`):
мод, БМ, навыки, спасы, DC. Эффекты складываются поверх в детерминированном,
задокументированном порядке; ручное ✎ — абсолютно и обратимо. Паритет с
таблицей — пол, не потолок (Конституция VII, правило переноса).

### E10. Mobile first; шип игрокам — мерило

Каждая спека эпика обязана быть полезной на ближайшей сессии (Конституция
VII). Порядок работ минимизирует время до листа в руках игроков; движок
режется ярусами ровно под это (P1 — то, без чего лист не живёт).

### E11. Ratchet против усложнения

Словарь операций и целей эффектов растёт только от реальных свойств:
новый op — при ≥ 3 строках в coverage-checklist. Урок DiceCloud: настройка
не должна стать дороже бумаги. Сомнение → текст (E3), не схема.

## Процесс

- Specify каждой спеки эпика обязан содержать UX-сценарии своих поверхностей
  (E1); «UI потом» — не статус.
- Новые поверхности → компактный design-проход (как design.md 022) между
  Clarify и Plan; существующие паттерны (карточка, пипы, разложение)
  переиспользуются, не изобретаются заново.
- Каждая контентная спека (045/047/049) обновляет coverage-checklist и
  отмечает закрытые строки.
- Изменение этой конституции — явное решение Андрея; бамп версии + строка в
  changelog ниже; копии (NEXT, spec headers) синхронизируются в том же
  коммите.
