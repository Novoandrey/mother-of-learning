## 1. Что план эпика делает правильно (подтверждения)

- **V1. Форк = копия (R3).** Nystrom для Type Object рекомендует
  «copy-down delegation» — скопировать атрибуты один раз при создании вместо
  хождения по цепочке наследования в рантайме. Его же предупреждение: какой
  бы тип-систему мы ни построили, её должны легко понимать не-программисты —
  это наш E11.
- **V2. Неразрушающие эффекты поверх базы.** Foundry AE хранит base data
  отдельно, эффекты применяются к derived-копии при подготовке данных —
  снять предмет = эффект исчез, база цела. Ровно наша модель
  «слой-0 → эффекты → ✎» и compute-on-read.
- **V3. Гранты по уровням — проверенная траектория.** dnd5e начинал с
  хардкода classFeatures.js и перешёл на data-driven Advancement
  (ItemGrant / ItemChoice / ScaleValue / Trait на классах, расах,
  бэкграундах, фитах). Наши 049/050 повторяют этот путь сознательно;
  ItemChoice = «точки выбора», ScaleValue = скейлинг от уровня класса.
- **V4. Loud validation (FR-011).** DiceCloud чинил баг «неверно
  таргетированные эффекты → рассинхрон клиента и сервера». Молчаливый дроп
  невалидной цели — источник worst-case багов; наш отказ с ошибкой верен.
- **V5. Порядок set → add.** MTG 613, слой 7: сначала эффекты, *задающие*
  значение (7b), потом добавляющие (7c). Совпадает с FR-012.
- **V6. Broadcast > postgres_changes.** Supabase прямо рекомендует
  Broadcast как масштабируемый и безопасный путь; postgres_changes
  однопоточен и проверяет RLS на каждого подписчика на каждое изменение.

## 2. Подводные камни (P) и предложения (D)

### P-01 [ВЫСОКИЙ] Конфликт одинаковых op на одной цели не определён
Два `set` на КД — Доспех мага (13+Лвк) и Защита без доспехов (10+Лвк+Тел):
по 5e формулы КД не складываются, игрок **выбирает одну**. FR-012 порядок
*между* op задаёт, *внутри* op — нет. MTG решает таймстампами (поздний
побеждает), Foundry — числовым priority. Без правила — недетерминизм.
**D-1**: внутри op порядок = время прикрепления (стабильный таймстамп);
для `set` дополнительно — «активная формула»: при >1 `set` на цель игрок
явно выбирает (флаг на attachment, дефолт = max), карточка/лист показывают
выбор; при >1 `override` — поздний побеждает + warning-бейдж в разложении.
→ новый Clarify **C-07**.

### P-02 [ВЫСОКИЙ] add ↔ mult: индустрия расходится, выбрать осознанно
Foundry применяет MULTIPLY **до** ADD (mode 1 < 2). Наш FR-012 — add до
mult. Для 5e наша версия вернее: «скорость удваивается» (haste) удваивает
текущую скорость с бонусами, не базу. Но это надо решить с открытыми
глазами, а не унаследовать молча. **D-2**: подтвердить add→mult в C-01 с
этим контекстом; округление floor — один раз, после mult.

### P-03 [СРЕДНИЙ] Нет ops «не ниже / не выше» (upgrade/downgrade)
Канон полон «становится X, если ниже»: Обруч интеллекта (Инт = 19, если
ниже), пояса гигантской силы и т.п. — среди наших 844 предметов такие есть.
Foundry держит UPGRADE/DOWNGRADE отдельными modes. `set`-выбором это не
выражается. **D-3**: зарезервировать `upgrade`/`downgrade` в enum (P2,
ratchet-гейт активации остаётся); место в свёртке — после add/mult, до
override (как у Foundry).

### P-04 [ВЫСОКИЙ] Эффекты на броски — не числа; резерв P2 сейчас не той формы
`advantage` не вмещается в `{value: number}`, и цель у него — **бросок**
(Скрытность-чек), а не значение листа. PF2e решает селекторами по
roll options + предикатами. Если «schema reserved» оставить как есть,
зарезервируем не то. **D-4**: на Clarify зафиксировать форму roll-эффектов:
`{target: "roll.skill.stealth", op: "advantage"}` (без value; value
опционален для будущих «+1d4 к броску»); реализация P2, но словарь целей
делится на value-targets и roll-targets уже в FR-011.

### P-05 [СРЕДНИЙ] Эффекты на чужие ресурсы (кросс-модульное таргетирование)
«+2 к максимуму Ки от предмета» — эффект модуля A на ресурс модуля B.
Адресация чужого ресурса — зона сложности (slug-и, коллизии, порядок).
**D-5**: v1 — текстом (E3); зарезервировать синтаксис цели
`resource.<slug>.max` в словаре, активация по ratchet (≥3 строки чеклиста).

### P-06 [ВЫСОКИЙ] LWW съедает конкурентные дельты play-state
ДМ и игрок одновременно правят HP/остаток ресурса → при last-write-wins
один апдейт теряется — ровно «потерянная работа» из DEBT-011. FR-016 уже
определяет spend(n)/restore(n) как операции. **D-6**: закрепить в E7 и
FR-024: **play-state меняется атомарными дельтами** (SQL increment/
decrement), LWW остаётся только для замен (текст, ✎, выборы). Тогда
конкурентные траты не теряются вовсе, а реалтайм лишь доносит результат.

### P-07 [СРЕДНИЙ] Realtime: транспорт и самохост-грабли
postgres_changes: однопоточная обработка WAL + RLS-проверка на подписчика;
Supabase рекомендует Broadcast, для БД-изменений — `realtime.
broadcast_changes()` из триггера в приватный канал с RLS-политиками на
`realtime.messages`. На нашем масштабе (≤30 зрителей) выдержит и
postgres_changes, но broadcast-from-DB сразу = точный контроль payload
(можно слать только нужные поля) и отсутствие миграции потом. Самохост:
`wal_level=logical`, слот репликации; **упавший realtime держит слот → WAL
растёт → диск CPX32**. **D-7**: Plan-note 045 конкретизировать: канал на
PC-лист, broadcast_changes из триггеров, мониторинг лага слота в бэкап-крон.

### P-08 [СРЕДНИЙ] Провенанс грантов: одного указателя мало
dnd5e пришлось завести ДВА указателя — advancementOrigin (прямой источник)
и ultimateOrigin (корневой): класс → фит → фича, и лист группирует фичу по
классу, а карточка показывает «от фита». Наш attachment с одним source
сломается на двухзвенной цепочке. **D-8**: в модели attachment'а —
`granted_via` (прямой грантёр) отдельно от корневого источника показа;
расширить C-02.

### P-09 [СРЕДНИЙ] Глубина форка при grants-рёбрах не задана
R3 говорит про копию полей. Факультатив = обрезанный форк фита; а если фит
сам грантует модули? **D-9**: дефолт — исходящие `grants`-рёбра копируются
(копия ссылается на те же гранты, не клонирует их), переопределяемо при
форке; в Clarify 047.

### P-10 [НИЗКИЙ] Удаление модуля с живыми attachment'ами
Поведение не задано. **D-10**: запрет удаления при наличии attachment
(подсказка «отвяжи или архивируй»); канон не удаляется вовсе (R4). FR в 045.

### P-11 [НИЗКИЙ] Два «override» в одной системе
Op `override` и ручное ✎ «override» — одинаковое слово, разные вещи; доки и
UI будут путать. **D-11**: ✎ всюду в текстах и UI называть «ручная правка»,
op-лексику оставить Foundry-совместимой. Чисто словарная правка.

### P-12 [НИЗКИЙ] «Модуль» в UI игрока
Foundry зовёт всё «Item», и это известно путает пользователей. «Модуль» —
ок для кода и спек; в UI игроку — «свойство»/«способность» по контексту.
**D-12**: словарь UI решить в design-проходе 045 (отметка в design brief).

### P-13 [СРЕДНИЙ] Два раннера деривации = риск рассинхрона
DiceCloud v2 ушёл в server-side computation после desync-багов. Наш план —
одна TS-библиотека на клиент и сервер — верный, но контракт нужен зубастый.
**D-13**: золотые фикстуры (xlsx-эталоны + синтетика по каждому op) гоняются
в vitest против библиотеки; версия словаря targets/ops включается в payload
деривации; расхождение версии = принудительный server-recompute.

### P-14 [инфо] НЕ строить инкрементальный граф пересчёта
DiceCloud держит реактивный граф зависимостей — для их масштаба (тысячи
свойств, формулы) оправдано. У нас полный пересчёт пирамиды из сотен
модулей — микросекунды; Dirty Flag/мемоизация — преждевременная
оптимизация. Подтверждение простоты FR-019, не камень.

### P-15 [ПРОДУКТОВЫЙ] Полная прозрачность ломает «неопознанные предметы»
E4/R6 «все видят все модули» против стандартной 5e-механики
identification: ДМ выдал предмет, игрок видит вещь, но не знает эффектов.
Прозрачность сейчас сознательно убивает эту интригу. Варианты: (а) принять
как решение кампании («у нас всё опознано», West Marches с метазнанием);
(б) P2-флаг `identified` на attachment — карточка показывает текст, прячет
эффекты до опознания (исключение из R6, нужна формулировка). **Решение за
Андреем** — это вкус кампании, не техника.

## 3. Геймдизайн-книги: что берём

Коротко — глубокие уроки для этого эпика лежат в системной/программной
литературе и продакшен-системах выше; дизайн-книги в основном подтверждают
принципы. Schell: плейтест рано и с настоящими игроками — пилоты (Каэл,
Миряна, Британия) и есть плейтест-когорта, гонять на них каждую фазу.
Sylvester: элегантность = одно правило решает много задач — «модуль» един
для рас/фитов/предметов/заметок; и предостережение про глубину, до которой
игроки не доберутся — конструктор (051) последним стоит правильно. Koster:
освоение паттернов = удовольствие — карточка с человекочитаемыми эффектами
учит системе через игру. Sellers: ресурсы как stocks & flows — наша
rest-операция и есть управляемый flow, держать источники/стоки явными.

## 4. Что сознательно НЕ перенимаем

- **Полные 7 слоёв MTG + dependency system** — мощность для
  взаимопревращающихся перманентов; нам хватает set→add→mult(+up/down)→
  override + таймстампы. Dependency-разрешение — overkill навсегда.
- **Формулы в value эффекта** (DiceCloud: value = calculation). Это
  встроенный язык выражений = ловушка Bytecode у Nystrom («не изобретай
  язык раньше времени»). value = число; скейлинг от уровня — отдельным
  механизмом 050 (аналог ScaleValue), не выражениями.
- **Предикатный DSL PF2e** для `when` — у них predicate-алгебра по сотням
  roll options и это прямо помечено «для продвинутых». Наш `when` — узкий
  enum-словарь, выращиваемый ratchet'ом.
- **Инкрементальный реактивный пересчёт** — см. P-14.
