# chatlog/

История разработки по чатам. Один файл на чат.

## Зачем

`NEXT.md` раньше рос линейно — 20 секций «что сделано в чате», 119 KB.
ИИ-агенту, чтобы понять актуальное состояние, приходилось читать всё.
Нумерация чатов ломалась (две «chat 24»), даты повторялись.

Решение: `NEXT.md` — только актуальное состояние (~2-5 KB).
История — отдельные файлы здесь.

## Соглашение об именовании

```
YYYY-MM-DD-chatNN-короткое-название.md
```

Примеры:
- `2026-04-20-chat27-ultrareview-and-shared-world.md`
- `2026-04-19-chat23-players-pc-binding.md`

Если в один день несколько чатов — номер чата их разводит.

## Что класть в файл чата

```markdown
# Chat NN — <название>, YYYY-MM-DD

## Контекст (откуда пришли)
Одна-две строки: что было на входе, что просил пользователь.

## Что сделано
- фича/бага/рефакторинг 1
- фича/бага/рефакторинг 2

## Миграции
- `NNN_name.sql` — одна строка описания

## Коммиты
- `<sha>` `<title>` — одна строка про что

## Действия пользователю (после чата)
- [ ] применить миграцию
- [x] задеплоить (авто через main)
- [ ] протестировать X с реальным игроком

## Что помнить следующему чату
- открытый вопрос 1
- отложенный кусок 2
```

## Что НЕ класть

- Длинные куски кода — они в git, ищи по sha коммита.
- Подробные дампы аудитов — записывай в `backlog.md` как TECH-NNN или BUG-NNN,
  а здесь ссылайся: «полный аудит → backlog `🔒 TECH DEBT`».
- Персональные переписки и благодарности — не нужно.

## Как использовать при старте чата

Большинству чатов достаточно `NEXT.md`. Читай chatlog только если:
- Нужен контекст конкретного решения («почему мы пошли через `can_edit_node` v2?»).
- Предыдущий чат оставил что-то в работе — читаем его файл.

Не нужно читать весь chatlog линейно. Это архив, а не конспект.
