# Chat 30 — DEBT-003 SRD seed extraction, 2026-04-22

## Контекст (откуда пришли)
Свободное окно после chat 29. Из четырёх P2-кандидатов в NEXT.md
пользователь выбрал DEBT-003 — open source blocker: миграции 003
(conditions), 005 (effect type), 022 (exhaustion levels) инсертили
SRD-данные через `WHERE c.slug = 'mat-ucheniya'`. Любая новая кампания
получала пустой тип `condition`, ноль состояний, ноль эффектов →
трекер энкаунтера ломался из коробки.

## Что сделано
- **`lib/seeds/dnd5e-srd.ts`** — единый источник правды для SRD-сида.
  Экспортирует `seedCampaignSrd(supabase, campaignId)`. Идемпотентно:
  читает существующие `node_types` и `nodes`, инсертит только
  отсутствующие. Существующие НЕ перезаписывает (DM мог отредактировать
  описания — уважаем). Содержит:
  - 2 node_types: `condition` (🔴), `effect` (✨)
  - 14 SRD conditions (без Истощённого — он разбит на уровни)
  - 6 уровней Истощения (`Истощение 1`..`Истощение 6`)
  - 0 SRD effects (homebrew — DM наполняет сам)
- **`lib/campaign-actions.ts`** — server action
  `initializeCampaignFromTemplate(campaignId)`. Проверяет роль
  (только owner/dm), вызывает `seedCampaignSrd`, возвращает
  `{ ok, seed | error }`. Для будущей формы «Создать кампанию» —
  пока в проекте нет UI создания, поэтому action существует, но
  ещё нигде не вызывается.
- **`scripts/seed-srd.ts`** + npm script `seed-srd` — CLI для
  бэкфилла существующих/созданных вручную через SQL кампаний.
  Использование: `npm run seed-srd -- --campaign <slug>`.
  Использует service role key как `seed-owner`.
- **Архитектурное решение**: без SQL-миграции 032. Чистая TS-логика
  — один источник правды, проще поддерживать. Идемпотентность
  через query-then-insert, не через `ON CONFLICT` (для `nodes`
  нет уникального индекса по title, а добавлять его рискованно
  для DM-данных).
