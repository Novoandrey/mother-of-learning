### Phase 6 — Apply action + RPC (T021-T023)
`applyLoopStartSetup(loopNodeId, opts)` — двухфазовый, diff-based,
идемпотентный:
- 12 шагов из plan.md реализованы по порядку
- FR-014 orphan filter: ряды с удалённым actor'ом вытаскиваются из
  `toDelete` до RPC-вызова и до identifyAffectedRows
- Guard: `stashSeedNonEmpty && !stash` → чистая ошибка
- Actor titles для диалога одним `nodes.id IN (…)` запросом
- `ApplyResult` расширен третьей веткой `{ ok: false, error }`;
  тип в `lib/starter-setup.ts` обновлён
- Два `revalidatePath`: `/c/${slug}/loops` и `/c/${slug}/accounting`

Миграция `038_apply_starter_setup_rpc.sql` — функция
`apply_loop_start_setup(uuid, jsonb, jsonb, uuid[])`:
- `security definer` + явный `search_path = public, pg_temp`
- `set_config('spec012.applying','on',true)` глушит оба autogen-
  триггера на всю транзакцию
- INSERT через `jsonb_array_elements`, UPDATE в цикле (явно ставит
  `autogen_hand_touched = false` + `updated_at = now()`), DELETE по
  массиву с `array_length` guard'ом
- Tombstone cleanup ограничен source_node_id + spec-012 wizard keys
- Возвращает `table(inserted, updated, deleted, tombstones_cleared)`
- `grant execute ... to authenticated` (namespace hygiene)
