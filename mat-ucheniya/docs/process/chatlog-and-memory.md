# Chatlog и память между сессиями

> Как проект помнит, что происходило. Четыре уровня памяти: `NEXT.md` —
> текущее состояние (boot-файл каждой сессии), `chatlog/` — журнал, что было
> сделано в каждом чате, `.specify/memory/` — долгоживущие документы,
> `backlog.md` — открытые идеи и баги.

---

## Четыре уровня памяти

### NEXT.md — boot-файл сессии

`NEXT.md` в корне репо — единственный файл, который гарантированно читается в начале каждого нового чата. Содержит только **текущее состояние**:

- Прод-URL, деплой, staging, бэкапы, доступы к боксу.
- Дедлайны (если есть).
- «Активная работа» — что именно сейчас в работе, с вперёд-ссылкой на `tasks.md`.
- «Роадмап 030+» и «В проде» — краткие таблицы.
- «Правила» — ссылки на `AGENTS.md` и `meta/claude-project-instructions.md`.

Жёсткий лимит: **150 строк / 10 KB**. За этим следит `bash scripts/dev/status.sh` и кричит ❌, если файл разросся. История в `NEXT.md` не хранится — она идёт в `chatlog/` и `CHANGELOG.md`.

Обновляется **в конце каждой сессии** — только секции состояния (`Активная работа`, `Дедлайны`, `Last updated`). Никаких «что было сделано».

### chatlog/ — журнал сессий

`chatlog/YYYY-MM-DD-chatNN-короткое-название.md` — один файл на чат.
Соглашение об именовании: `2026-06-23-chat98-spec044-046-prod.md`.

Структура каждого файла (шаблон в `chatlog/README.md`):

```
# Chat NN — <название>, YYYY-MM-DD

## Контекст (откуда пришли)
## Что сделано
## Миграции
## Коммиты
## Действия пользователю (после чата)
## Что помнить следующему чату
```

Что **не кладут** в chatlog: длинные куски кода (они в git по sha), подробные дампы аудитов (они в `backlog.md` как `TECH-NNN`/`BUG-NNN`), персональные переписки.

`close-session.sh <slug>` создаёт файл по шаблону автоматически, вычисляя следующий номер чата.

**Когда читать chatlog:** большинству задач хватает `NEXT.md`. Chatlog открывается только если нужен контекст конкретного решения или предыдущий чат оставил незакрытую работу. Весь архив читать не надо — `chatlog/_legacy-NEXT-archive.md` держит старые состояния `NEXT.md`.

### backlog.md — открытые идеи и баги

`backlog.md` — мастер-бэклог: баги (`BUG-NNN`), идеи (`IDEA-NNN`), технический долг (`TECH-NNN`), будущие спеки. Не читается по умолчанию; открывается осознанно — «ищу идеи», «проверяю, не задублировал ли баг».

Закрытые тикеты и исторический контекст — в `backlog-archive.md`.

### .specify/memory/ — долгоживущая память

`.specify/memory/` хранит документы, которые не устаревают за одну сессию:

- `constitution.md` — конституция проекта v3: миссия, принципы продукта и процесса, для кого строится, два режима (DM/игрок).
- `bookkeeping-roadmap.md` — хронология и связи спек бухгалтерской серии.
- `encounter-tracker-v1-retrospective.md` — ретроспектива v1 трекера.
- Roadmap'ы и стратегические документы, которые переживают несколько спек.

`.specify/epics/rpg-engine/constitution.md` — отдельная конституция для эпика RPG-движка (E1–E11, R1–R12).

## Где что искать

| Вопрос | Куда смотреть |
|---|---|
| Что сейчас в работе? | `NEXT.md` |
| Что было сделано в чате 87? | `chatlog/` → файл с `chat87` в имени |
| Почему выбрали `can_edit_node v2`? | `chatlog/` → поиск по конкретному чату |
| Какие открытые баги? | `backlog.md` |
| Что за принцип «tool-first»? | `.specify/memory/constitution.md` → [`concepts/tool-first.md`](../concepts/tool-first.md) |
| Статус спеки? | `bash scripts/dev/status.sh` |
